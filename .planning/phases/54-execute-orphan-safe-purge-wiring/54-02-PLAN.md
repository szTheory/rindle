<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Oban job args -> `PurgeStorage.perform/1` | Async job payload can become stale relative to current attachment truth |
| `PurgeStorage.perform/1` -> storage adapter delete calls | Destructive byte deletion happens outside the DB transaction and must be gated by live attachment truth |
| existing `attach/4` / `detach/3` -> purge worker | Slot-scoped APIs enqueue purge work that may target shared assets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-54-05 | Tampering | `lib/rindle/workers/purge_storage.ex` | mitigate | Query live `media_attachments` immediately before destructive delete and short-circuit `:ok` when any surviving attachment exists |
| T-54-06 | Denial of service | repeated purge jobs for the same asset | accept | Duplicate jobs are acceptable because the worker becomes idempotent and cheap on the survivor path; enqueue dedupe remains a secondary aid in Plan 01 |
| T-54-07 | Information disclosure | worker/log-visible asset operations | mitigate | Keep args shape narrow to `asset_id` and `profile`; do not add owner-identifying data to worker payloads or test-visible diagnostics |
| T-54-08 | Elevation of privilege | legacy slot-scoped APIs triggering shared-asset deletion | mitigate | Preserve existing enqueue behavior but enforce final safety only inside the hardened worker boundary before bytes or asset rows are deleted |
</threat_model>
