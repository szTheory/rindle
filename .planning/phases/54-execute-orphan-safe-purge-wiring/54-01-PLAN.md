<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| caller -> `Rindle.erase_owner/2` | Untrusted caller input chooses which owner's attachments are targeted for destructive detach work |
| `Rindle.erase_owner/2` -> DB transaction | Report planning becomes attachment-row mutation and purge enqueue state |
| DB commit -> Oban purge jobs | Async destructive work can become stale before the worker runs |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-54-01 | Tampering | `lib/rindle/internal/owner_erasure.ex` | mitigate | Recompute the owner-erasure plan inside execute per D-06 and derive purge eligibility from live attachment counts instead of trusting a prior preview payload |
| T-54-02 | Denial of service | purge enqueue loop in `lib/rindle/internal/owner_erasure.ex` | mitigate | Use Oban uniqueness keyed by `asset_id` for active states only and treat `%Oban.Job{conflict?: true}` as semantic success per D-15 through D-17 |
| T-54-03 | Information disclosure | public owner-erasure report maps | mitigate | Emit only semantic fields `attachment_id`, `asset_id`, `slot`, `profile`, and `surviving_attachment_count`; do not expose unrelated owner rows, raw schemas, or Oban internals per D-20 through D-23 |
| T-54-04 | Repudiation | public execute/report contract | mitigate | Return stable report buckets and counts for preview and execute so detach and enqueue intent is auditable rather than hidden inside transaction state |
</threat_model>
