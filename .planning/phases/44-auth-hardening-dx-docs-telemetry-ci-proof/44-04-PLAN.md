<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| generated Phoenix app ↔ packaged Rindle tus edge | The package-consumer proof depends on the published package behaving correctly in a fresh adopter app over a real socket, not only inside repo-local unit tests. |
| Node tus client ↔ server create/resume contract | A subtle request-header or protocol mismatch can break upload creation or resume while unit tests still pass. |
| CI smoke wrapper ↔ operator diagnosis | If the lane only reports `socket hang up`, the package-consumer contract becomes expensive to debug and easy to waive incorrectly. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-44-04-01 | D | generated-app tus smoke lane | mitigate | Persist structured failure reports so live-socket failures expose the failing phase and endpoint instead of a bare `ECONNRESET`. |
| T-44-04-02 | T | Node create/resume proof | mitigate | Keep the real `tus-js-client@4.3.1` interrupted-upload + resume flow, but pin and verify the exact request options that the generated Phoenix app accepts. |
| T-44-04-03 | R | merge confidence | mitigate | Keep `bash scripts/install_smoke.sh tus` merge-blocking in CI and require the passing MinIO-backed proof before Phase 44 is re-verified complete. |
</threat_model>
