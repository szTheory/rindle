<objective>
Land ONE additive, reversible change that makes a single stable `CI Summary` aggregate
job the canonical signal of CI health, and collapse the branch-protection required-check
declaration to that one context — all in a single isolated PR, BEFORE any Phase 106 lane
rename (CONTEXT Phase Boundary + D-10). The PR itself performs NO live branch-protection
mutation; the live flip is a separate post-merge human go/no-go step (D-11) documented in a
committed runbook.

Purpose: Branch protection currently requires 12 individual matrix-suffixed contexts that
are fragile to the matrix-name churn Phase 106 will introduce. Collapsing to one aggregate
context (`CI Summary`) makes protection immune to lane renames, closes the fork-PR
pending-forever trap, and — deliberately — brings `brandbook-tokens` into the gate
transitively for the first time (D-09).

- New `ci-summary` job (`name: CI Summary`) appended to `.github/workflows/ci.yml`.
- `scripts/setup_branch_protection.sh` REQUIRED_CHECKS array + heredoc collapsed to `"CI Summary"`.
- `105-FLIP-RUNBOOK.md` documenting the two-step cutover, the D-09 tightening, and the D-13 verification.
- A blocking human-verify checkpoint for the post-merge live flip (NOT automated in CI).
</objective>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CI runner → GitHub Actions `needs` context | The aggregate gate consumes only `toJSON(needs)`, a platform-controlled value — not untrusted user input. |
| Maintainer admin PAT → GitHub branch-protection REST API | The live flip is an admin-scoped write to `/branches/main/protection`; performed by a human, post-merge, never in PR CI. |
| Fork PR → required-check evaluation | Fork PRs skip repo-gated jobs; the aggregate must report success (not hang) for them. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-105-01 | Spoofing | `CI Summary` aggregate result (the "green checkmark lie") | mitigate | Step evaluates `needs.*.result` EXPLICITLY and `exit 1`s on any non-pass (D-06); an `if: always()` job that does not check results would falsely report green. Verified by Task 1 acceptance criteria (skip-as-pass reference + exit-on-failure). |
