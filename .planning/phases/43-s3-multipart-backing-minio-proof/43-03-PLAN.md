<objective>
Close the load-bearing TUS-09 reaper leak. Today every `upload_strategy: "resumable"` session — which now INCLUDES tus sessions — is caught by `resumable_abort_session?/1` and routed to `attempt_resumable_cancel` -> `resolve_resumable_adapter`, which requires the `:resumable_upload_session` capability. S3/Local do NOT advertise that capability, so the resolve fails: the tus session is mis-handled (persisted `aborted` with `resumable_cancel_failed:*`) and the S3 multipart upload is NEVER aborted -> orphaned multipart parts accrue storage cost forever (Pitfall 1).

Fix: convert `expire_session/2` to a `cond` that checks `tus_session?(session)` FIRST and routes tus sessions to a new `expire_tus_session/2` that aborts the S3 multipart (or removes the Local tmp) idempotently, then persists `state: "expired"`. Tighten `resumable_abort_session?/1` to exclude `resumable_protocol: "tus"` so a future query expansion can't double-route.

Purpose: This branch IS the cost-leak / information-disclosure mitigation (T-43-cost-leak). The MinIO `list_multipart_uploads`-empty assertion (Plan 05) proves it.
Output: a reaper that reaps tus sessions correctly without disturbing the gcs_native or legacy lanes.
</objective>
