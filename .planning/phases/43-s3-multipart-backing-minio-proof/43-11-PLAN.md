<objective>
Close **CR-01** completely. Prior plan 43-09 fixed the DELETE happy-path ordering
(abort the backing BEFORE the `aborted` transition) but left the abort-FAILURE
branch a permanent orphan: `abort_delete_backing/2` swallows `{:error, _}` and
returns `:ok`, the row moves to `state: "aborted"` with `failure_reason: nil`, and
NO reaper query ever re-selects it (`fetch_incomplete_timed_out_sessions/0` never
matches `aborted`; `fetch_retryable_abort_sessions/0` requires
`like(failure_reason, "resumable_cancel_failed:%")` which `nil` never satisfies).
A transient `abort_multipart_upload` failure at DELETE time therefore orphans the
S3 multipart upload forever — the exact cost leak TUS-09 exists to close.

**Locked remediation (marker + reaper-query approach):** persist a retryable
`failure_reason` marker (`tus_abort_failed:<reason>`) on the row when the DELETE-time
abort fails, and add a reaper query that re-selects such rows so the next cron
re-aborts the S3 multipart. Rationale and the rejected alternative are documented in
the `<remediation_decision>` block below.

This plan owns the reaper module + the Plug DELETE marker + their two regression
tests. It has NO file overlap with 43-12 (CR-04 / s3.ex), so the two run in the
same wave.

Purpose: make the DELETE abort-failure branch reaper-recoverable so SC4 (DELETE
no-leak) holds even when the remote abort fails transiently — not just on the happy
path 43-09 covered.

Output: a reaper that compensates for a failed DELETE-time abort, a DELETE handler
that records a retryable marker instead of silently orphaning, the WR-03
FSM-transition reconciliation, and two regression tests that fail pre-fix.
</objective>

<remediation_decision>
**Chosen: marker + reaper-query (VERIFICATION.md `missing:` option 2). LOCKED.**

Why the marker approach over the "leave the row in its prior state + return 5xx"
alternative (option 1):

1. **Consistency with existing architecture.** The codebase ALREADY implements this
   exact pattern for the GCS resumable path: `fetch_retryable_abort_sessions/0`
   selects `state == "aborted" AND like(failure_reason, "resumable_cancel_failed:%")`,
   and `persist_resumable_abort_failure/3` writes that marker on a failed cancel. The
   tus abort-failure path is the same shape (a remote abort that may transiently fail
   and must be retried by the reaper). Mirroring the established pattern keeps one
   coherent reaper-compensation model rather than introducing a second, divergent one.
2. **Operator-friendly client semantics (STATE.md decision-making preference).** The
   marker approach keeps DELETE returning 204 to the client: the cancel is accepted,
   and the cost-leak compensation happens transparently in the reaper. Option 1 would
   change client-facing DELETE semantics to a 5xx on a transient backend blip, forcing
   the client to retry a cancel that, from its perspective, already succeeded —
   surprising and worse ergonomics. STATE.md explicitly prefers
   telemetry/metadata-driven compensation over expanding the client-facing error
   surface for advisory/recoverable conditions.
3. **No reliance on `expires_at`.** Option 1 depends on `fetch_incomplete_timed_out_sessions/0`
   reaping the row only AFTER `expires_at` passes — a long-TTL session would leak the
   multipart for the entire remaining TTL window. The marker approach makes the row
   eligible for re-abort on the very next cron, independent of TTL.

**WR-03 reconciliation (mandatory, coupled to this fix):** the FSM declares
`"aborted" => []` (terminal). Routing an aborted tus session back through
`expire_tus_session/2` must NOT then call `gated_expire/2` (which would attempt the
FSM-forbidden `aborted -> expired`, log `invalid_transition`, increment
`abort_errors`, and re-select the row forever — a silent infinite retry). On a
successful re-abort of an already-`aborted` tus session, settle the row via a
dedicated persist that mirrors `persist_resumable_abort_success/3` (clears the
marker; does NOT route through the FSM gate) — exactly how the GCS retryable path
already settles `aborted` rows without an illegal transition.
</remediation_decision>
