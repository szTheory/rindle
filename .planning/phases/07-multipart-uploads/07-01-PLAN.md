<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client→broker | Untrusted multipart initiation, part-number, and ETag input enters the app here |
| broker→storage adapter | Capability-gated server calls create and complete remote multipart uploads |
| broker→repo | Session authority and manifests are persisted on the adopter-owned runtime repo seam |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-07-01 | T | `lib/rindle/upload/broker.ex` multipart completion | mitigate | Accept only integer `part_number` and binary `etag` entries, persist the authoritative manifest server-side, and sort before adapter completion so clients cannot reorder or omit parts silently. |
| T-07-02 | E | `lib/rindle/upload/broker.ex` capability gate | mitigate | Gate multipart behind `adapter.capabilities/0` and return `{:error, {:upload_unsupported, :multipart_upload}}` before any adapter-specific call, preventing unsupported backends from falling into unsafe runtime behavior. |
| T-07-03 | I | `lib/rindle/upload/broker.ex` trust boundary | mitigate | Keep remote completion separate from promotion by invoking the existing `verify_completion/2` path after adapter completion per D-05; do not mark assets promotable directly from remote complete success. |
</threat_model>
