- BLOCKER-2 (translucent-token compositing): `lum(hex)` strips `#` and parseInts 2-char slices,
  so a translucent value like dark `--ck-tint: rgba(16, 185, 129, 0.09)` yields NaN. A
  translucent background also has NO defined contrast ratio without compositing. The resolver
  MUST detect a non-hex (`rgba(...)`/`color-mix(...)`) value and composite it over its REAL
  flat-hex backdrop (`--ck-surface` for the matching theme) to a flat hex BEFORE calling
  `ratio()`. NEVER feed `rgba(...)`/`color-mix(...)` strings into `lum()`. Compositing math:
  for `rgba(r,g,b,a)` over backdrop hex `B`, each channel = `round(r*a + B_channel*(1-a))`.
  (Confirmed: dark `--ck-tint` composited over `#111d18` = `#112b21`; `--ck-brand-strong`
  `#34d399` on it = 7.85:1 — clears the 4.5 large/icon minimum.)
- BLOCKER-3 (theme-invariant token `:root` cascade fallback): `--ck-btn-bg` (cohort.css:38) and
  `--ck-on-brand` (cohort.css:40) — like the other brand tokens `--ck-brand`/`--ck-accent`/
  `--ck-btn-bg-hover` — are defined in the light `:root` block ONLY and are INTENTIONALLY ABSENT
  from the dark `[data-theme="dark"]`/`@media (prefers-color-scheme: dark)` block (white button

  custom-property cascade means a `[data-theme="dark"]` element STILL inherits these `:root`-only

  `--ck-on-brand on --ck-btn-bg` pair. The fallback is NARROW: it only rescues theme-invariant
  tokens that genuinely live in `:root`; a token absent from BOTH blocks still resolves to
  `undefined` and FAILs, so a genuinely-missing dark token is NOT masked. Do NOT add tokens to
  Plan 01 / cohort.css (no value changes — D-96-23). Do NOT drop the dark twin of the pair
  (UI-SPEC line 167 requires it in both themes).
