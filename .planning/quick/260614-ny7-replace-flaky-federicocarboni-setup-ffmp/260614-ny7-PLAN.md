# Quick Task 260614-ny7: Replace flaky setup-ffmpeg action with apt

## Problem

`FedericoCarboni/setup-ffmpeg@v3` downloads ffmpeg from johnvansickle.com on every
run and intermittently fails with `AssertionError: Failed to get latest
johnvansickle ffmpeg release`. Across PR #22 it flaked 3× on different lanes,
blocking merges and forcing manual re-runs — it taxes every PR.

## Scope

`.github/workflows/ci.yml` only. The action is used by 4 jobs: `quality` (both
matrix cells), `integration`, `package-consumer`, `adopter`. Two other lanes
(`adoption-demo-unit`, `adoption-demo-e2e`) already install ffmpeg via apt and pass.

## Constraint

`lib/rindle/av/probe.ex:18` hard-requires ffmpeg **>= 6.0** (raises below it).

**apt is insufficient:** an initial attempt used `apt-get install ffmpeg`, but apt on
the ubuntu-22.04 runners ships **4.4.2** (< 6.0) — the `>= 6` guard correctly caught
this and failed both Quality cells loudly (working as designed). The apt-based
adoption-demo lanes only pass because their tests don't exercise the >= 6 AV path.
So these lanes need a real ffmpeg->=6 source.

## Task

Add `scripts/ci/install_ffmpeg.sh` that downloads johnvansickle's stable
`ffmpeg-release-amd64-static` build directly with `curl --retry` (skipping the
action's brittle "find latest" metadata step that caused the flake) and asserts
major >= 6. Replace each of the 4 `uses: FedericoCarboni/setup-ffmpeg@v3` blocks
(jobs: quality, integration, package-consumer, adopter) with
`run: bash scripts/ci/install_ffmpeg.sh`.

Source note: johnvansickle reports a plain numeric version (`7.0.x`) that the
library's probe (`lib/rindle/av/probe.ex`) parses. apt (4.4) is too old; BtbN's
`n7.1` tag fails the probe regex — both rejected (see SUMMARY).

Leave the 2 existing apt demo steps unchanged. Do NOT touch `lib/` or `test/`.

## Verify

- 0 `uses: FedericoCarboni` lines remain; 6 apt ffmpeg installs total.
- YAML valid.
- Push branch, open PR; `Quality` (1.15/26 + 1.17/27), `Integration`,
  `Package Consumer`, and `Adopter` all green with no ffmpeg-version-gate regression.

## Commit

Single atomic `ci:` commit (NOT `fix:`) so release-please does not cut a patch
release for this CI-only change.
