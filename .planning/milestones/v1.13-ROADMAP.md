# Milestone v1.13: Cancel Direct Upload

**Status:** ✅ SHIPPED 2026-05-27
**Phases:** 64–66
**Total Plans:** 8

## Overview

Give adopters a first-class way to abort an abandoned Mux direct creator upload
before the browser finishes PUT, using the same `asset_id` they received at
create time.

## Phases

**Goal:** Freeze the public cancel boundary before code lands.
**Depends on:** Phase 63 (v1.12)
**Plans:** 4 plans

## Milestone Summary

**Key Decisions:**

- Contract-before-implementation: Phase 64 froze types/FSM/persistence before
  Mux HTTP (reduces integration rework).
- FSM-first orchestration: conditional `update_all` before provider cancel HTTP.
- Mux-only wedge in v1.13; second-provider cancel deferred to explicit demand.
- HTTP-layer 403/404 idempotency at `Mux.HTTP`; orchestration tests use ClientMock.

**Issues Resolved:**

- Mux direct-upload control gap (`cancel_direct_upload/1` was planning-only).
- `create_direct_upload/2` did not persist provider upload handle for cancel correlation.
- Guide/docs parity for cancel semantics and Mux-only scope.

**Issues Deferred:**

- Admin/bulk owner-erasure orchestration (v1.14+ demand wedge).
- Second streaming provider cancel (contract extension when adapter ships).
- Nyquist VALIDATION closure for phases 64 and 66 (partial; non-blocking).

**Technical Debt:**

- `@typedoc` at `lib/rindle/streaming.ex:31` still references "ships Phase 65".
- `ingest_mode :not_cancellable` — freeze test only (D-11 skip by design).
- 403/404 idempotency proven at HTTP layer; orchestration uses ClientMock `:ok`.
