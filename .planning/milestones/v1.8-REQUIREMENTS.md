# Requirements: Rindle v1.8 — Resumable Browser Ingest

**Defined:** 2026-05-22
**Core Value:** Media, made durable.
**Source:** Research-driven locked recommendation in `.planning/research/v1.8/`
(`TUS-RESEARCH.md`, `MUX-DIRECT-UPLOAD-RESEARCH.md`, `STRATEGY-SEQUENCING.md`).
These supersede the now-stale `v1.6-CANDIDATE-TUS.md` and `TUS-CANDIDATE-MEMO.md`,
both of which predate v1.7 — which already shipped ~60% of the "tus foundations"
substrate (broker resumable lane, `"resuming"` FSM, `session_uri` redaction,
resumable telemetry, the reaper, capability machinery).

**Goal:** Make unreliable-network, large-file, browser-origin uploads durable.
Ship the tus 1.0 protocol on a mountable **bare `Plug`** (no `tussle`, no Phoenix
dependency) backed by S3 multipart-per-PATCH and Local tmp-append; pull forward
browser→Mux direct creator upload; so every browser ingest path Rindle exposes is
resume-safe and converges into the one trusted `verify_completion/2` promote lane.

## v1.8 Requirements

### tus Protocol Edge (bare Plug)

- [x] **TUS-01**: Adopter mounts `Rindle.Upload.TusPlug` via `forward "/uploads/tus", Rindle.Upload.TusPlug, ...` in their router without adding a Phoenix dependency to Rindle.
- [x] **TUS-02**: tus client creates a resumable upload and receives `201` + `Location` bound to a broker session created via `Rindle.Upload.Broker.initiate_tus_upload/2`.
- [x] **TUS-03**: tus client reads authoritative offset via `HEAD`, resumes via `PATCH`, gets `204` on success, and gets `409` on offset mismatch.
- [x] **TUS-04**: `OPTIONS` advertises `Tus-Version`, `Tus-Resumable`, `Tus-Extension`, and `Tus-Max-Size`.
- [x] **TUS-05**: tus upload URLs are HMAC-signed, verified on every `HEAD`/`PATCH`/`DELETE`, and stored redacted in `session_uri`.

### Storage Backing (S3 multipart-per-PATCH)

- [x] **TUS-06**: New OPTIONAL adapter callback `upload_part_stream/5` on `Rindle.Storage`; S3 implements one `UploadPart` per `PATCH` ≥ 5 MiB and buffers the sub-5 MiB final chunk under `Rindle.tmp/tus/`.
- [x] **TUS-07**: Adapters advertise the `:tus_upload` capability honestly and `TusPlug` raises at `init/1` when the adapter lacks it.
- [x] **TUS-08**: tus completion converges into the existing `verify_completion/2` lane with zero new completion vocabulary.
- [x] **TUS-09**: tus sessions expire via `expires_at`, `DELETE` terminates them, and the reaper branches on `resumable_protocol`; MinIO proof verifies drop/resume and zero leak behavior.

### Auth, DX, Docs, Proof

- [x] **TUS-10**: Optional resume authorizer can re-validate the resuming request identity against the captured creator identity.
- [x] **TUS-11**: tus errors surface through `Rindle.Error` with tagged reasons and fix-oriented copy.
- [x] **TUS-12**: tus edge telemetry emits through the existing `[:rindle, :upload, :resumable, *]` namespace without leaking forbidden metadata.
- [x] **TUS-13**: `mix rindle.doctor` reports tus capability/config mismatches.
- [x] **TUS-14**: `guides/resumable_uploads.md` documents the contract and the generated-app package-consumer CI proof lane succeeds against MinIO.

### Browser → Mux Direct Creator Upload (sibling — droppable under budget)

- [x] **MUX-20**: The Mux adapter implements `create_direct_upload/2` and advertises `:direct_creator_upload`.
- [x] **MUX-21**: The `video.upload.asset_created` webhook branch links the upload to the provider-asset row via Mux `passthrough`.
- [x] **MUX-22**: `Rindle.Streaming.create_direct_upload/2` creates the local rows and returns only `%{upload_url, asset_id}`.
- [x] **MUX-23**: LiveView and controller/JSON direct-upload paths are documented and covered end-to-end.

### Code-Review Polish (hygiene sub-stream)

- [x] **POLISH-01**: Phase 34 advisory code-review findings were resolved or waived with rationale.
- [x] **POLISH-02**: Phase 35 advisory code-review findings were resolved or waived with rationale.
