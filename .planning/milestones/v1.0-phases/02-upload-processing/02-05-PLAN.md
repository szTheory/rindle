# Plan 02-05: Atomic Promote & Idempotent Purge

**Goal:** Implement atomic attachment logic and an idempotent purge mechanism to prevent race conditions during upload replacements and ensure consistent storage cleanup.

## Success Criteria
- [ ] `Rindle.attach/3` implemented with support for atomic replacements.
- [ ] `Rindle.detach/2` implemented with idempotent behavior.
- [ ] `Rindle.Workers.PurgeStorage` worker implemented for async storage deletion.
- [ ] Atomic Promote Check: Background workers verify that the attachment hasn't changed before final state updates.
- [ ] Idempotent Purge: Storage deletion enqueued transactionally and handles missing objects gracefully.
- [ ] Unit/Integration tests for attach, detach, and purge worker.

## Proposed Changes

### Public API
- Implement `Rindle.attach(asset, owner, slot, opts)` in `lib/rindle.ex`.
- Implement `Rindle.detach(owner, slot, opts)` in `lib/rindle.ex`.

### Workers
- Create `lib/rindle/workers/purge_storage.ex`.
- This worker will call `Rindle.delete/2` for the asset and all its variants.

### Core Logic
- `attach/3`: 
    1. Transactionally check for existing attachment in the slot.
    2. If exists, detach the old one first (triggering purge).
    3. Insert new attachment.
- `detach/2`:
    1. Transactionally remove the attachment record.
    2. Enqueue `PurgeStorage` job for the detached asset (if no other attachments exist).

### Testing
- Create `test/rindle/attach_detach_test.exs`.
- Test concurrent replacement scenario.
