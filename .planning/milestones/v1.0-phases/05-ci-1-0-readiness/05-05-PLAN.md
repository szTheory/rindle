    ```yaml
    name: Release

    on:
      workflow_dispatch:
      push:
        tags:
          - "v*"

    permissions:
      contents: read

    jobs:
      release:
        name: Release Check
        runs-on: ubuntu-latest
        # Blocker 6 / T-05-05-01 mitigation: declare a GitHub Actions Environment
        # named `release`. Once a repo admin configures protection rules on this
        # environment (required reviewers, branch restriction to `main` and tag
        # refs only), any secret bound to the `release` environment (e.g. a
        # future HEX_API_KEY) is gated behind those rules. Fork PRs and
        # unauthorized branch pushes cannot trigger this job in a way that
        # exposes the secret. The environment must be created in repo settings
        # BEFORE any real HEX_API_KEY is added as an environment secret.
        environment: release
        env:
          MIX_ENV: dev

        steps:
          - name: Checkout
            uses: actions/checkout@v4

          - name: Set up Elixir
            uses: erlef/setup-beam@v1
            with:
              elixir-version: "1.17"
              otp-version: "27"

          - name: Restore deps cache
            uses: actions/cache@v4
            with:
              path: deps
              key: deps-${{ runner.os }}-1.17-27-${{ hashFiles('**/mix.lock') }}
              restore-keys: deps-${{ runner.os }}-1.17-27-

          - name: Install dependencies
            run: mix deps.get

          # Hex auth shim — `mix hex.publish --dry-run` may require Hex auth in
          # some CLI versions (Pitfall 5 in 05-RESEARCH.md, A1). Set a dummy
          # HEX_API_KEY so the dry-run never attempts a real publish even if
          # auth IS required. The dry-run should not actually use this key
          # because no PUT to the Hex API occurs. When 1.0 cutover wires a real
          # key, replace this with `${{ secrets.HEX_API_KEY }}` (which becomes
          # available because of the `environment: release` declaration above).
          - name: Set Hex auth shim
            run: echo "HEX_API_KEY=dryrun-placeholder" >> $GITHUB_ENV

          # Step 1: hex.build is the must-pass step (no auth required).
          - name: Build package artifact
            run: mix hex.build --unpack

          - name: Assert required paths present in tarball
            run: |
              set -e
              ls rindle-*/lib/rindle.ex
              ls rindle-*/mix.exs
              ls rindle-*/README.md
              ls rindle-*/LICENSE

          - name: Assert prohibited paths absent from tarball
            run: |
              set -e
              if [ -e rindle-*/_build ];      then echo "FAIL: _build leaked into tarball"; exit 1; fi
              if [ -e rindle-*/.planning ];   then echo "FAIL: .planning leaked into tarball"; exit 1; fi

          # Step 2: hex.publish --dry-run is the additional metadata gate.
          # If it fails due to auth (A1), tolerate the failure with a warning
          # rather than failing the lane — the artifact inspection above is
          # the primary gate.
          - name: Dry-run publish (optional — A1 risk tolerance)

              set +e
              mix hex.publish package --dry-run --yes 2>&1 | tee /tmp/dryrun.log

                fi
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → Hex API (mix hex.publish) | Dry-run does not push, but if a real publish ever runs it would push to Hex.pm with a HEX_API_KEY |
| Tag push → release workflow → release environment | Anyone with push access (or PR merge access via the protected branch) can trigger the release workflow by pushing a `v*` tag; the `release` environment's protection rules gate access to environment secrets |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05-01 | Information Disclosure | Hex API key leaking via release workflow logs (when a real key is configured later at 1.0 cutover) | mitigate | Concrete enforcement mechanism (Blocker 6): the release job declares `environment: release` so any HEX_API_KEY secret must be bound to that environment (NOT to repo-level secrets). | T-05-05-02 | Tampering | Supply-chain compromise of Hex.pm publishing infrastructure | accept | Trust delegated to Hex.pm team. Same trust model used by the entire Elixir ecosystem. No project-level mitigation possible. |
| T-05-05-03 | Information Disclosure | `.planning/`, internal config, or secrets accidentally shipped in the published tarball | mitigate | Explicit `files:` allowlist in mix.exs `package/0` (Task 1) excludes all non-allowlisted paths.
