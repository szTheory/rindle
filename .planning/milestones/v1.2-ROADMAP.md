# Milestone v1.2: First Hex Publish

**Status:** ✅ SHIPPED 2026-04-29
**Phases:** 10–14
**Total Plans:** 11

## Overview

Milestone v1.2 proved Rindle's first real `Hex.pm` publication path end to end
so future releases can reuse a practiced, documented, test-backed workflow. The
roadmap stayed deliberately narrow: Phase 10 closed the remaining publish
readiness gaps around metadata, owner/auth setup, and preflight visibility.
Phase 11 upgraded the existing release lane from dry-run-only proof to a real,
protected publish path. Phase 12 proved the public package from the outside in
and captured the maintainer runbook so future releases reuse the same path
instead of rediscovering it. Phase 13 closed audit traceability and runbook
drift. Phase 14 finished the remaining Nyquist validation artifacts needed for
a clean milestone closeout.

## Milestone Summary

**Key Decisions:**

- First public Hex publish should be scoped narrowly and exercised before broader API cleanup (the release path was the remaining trust gap and should become routine before new surface-area bets)
- Keep Phase 10 preflight-only by routing release checks through `scripts/release_preflight.sh` and leaving live Hex credentials for Phase 11
- Swapped out dry-run publish step for a live publish step guarded by real `HEX_API_KEY` environment variable logic in Phase 11-01
- Ensured publish pipeline fails fast if the Git tag does not match the mix.exs version (version drift gate added in Phase 11-02)
- Moved the previously local/manual dry-run validation into a fully automated CI test to avoid manual verification (Phase 11-03)
- Normalize all Phase 11 and Phase 12 summaries to `requirements-completed` rather than mixed requirement:/requirements: fields that blocked the strict three-source milestone audit
- Encode workflow contract as both positive assertions (step names/commands must appear) and refutation assertions (stale deferred language must be absent) in parity test

**Issues Resolved:**

- Requirement trace metadata inconsistency across phase summaries (resolved in Phase 13-01)
- Stale "live HEX_API_KEY automation is deferred to Phase 11" language in release runbook (resolved in Phase 13-02)
- Phase 10 and Phase 11 VALIDATION artifacts left in draft/pending Nyquist state (resolved in Phase 14)
- Release-doc parity test did not assert against the live workflow contract (resolved in Phase 13-02)

**Technical Debt Incurred:**

- Live Hex.pm publish proof (RELEASE-08 live confirmation) delegated to the next real tag; wiring is in place and tested locally but not exercised against a real published version
