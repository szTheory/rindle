# Milestone v1.10: Owner Account Erasure

**Status:** ✅ SHIPPED 2026-05-26
**Phases:** 53-55
**Total Plans:** 6

## Overview

Milestone `v1.10` turned owner/account deletion from a hand-rolled detach-loop
workaround into a supported, auditable lifecycle contract. The shipped wedge
adds one public preview/execute facade, explicit reporting buckets, orphan-only
purge semantics, retained shared-asset behavior, and proof/docs parity that
keeps the support boundary honest.

## Key Accomplishments

- Froze one owner-erasure contract on the public `Rindle` facade with stable
  `attachments_to_detach`, `assets_to_purge`, and
  `retained_shared_assets` report buckets.
- Added a shared preview/execute owner-erasure seam that detaches attachment
  rows transactionally and only reuses async purge when assets become newly
  orphaned.
- Hardened the destructive purge boundary so stale purge work skips deletion if
  any surviving attachment still exists.
- Closed the proof posture with hermetic and canonical adopter-facing coverage
  for preview, execute, orphan purge, retained shared assets, and idempotent
  reruns.
- Replaced workaround-first account-deletion guidance with canonical owner
  erasure docs and aligned roadmap, requirements, state, and audit truth.

### Milestone Goal

Give adopters one auditable account-deletion flow that reports what will be
detached and purged, executes that work through a public facade, and preserves
shared assets unless they become newly orphaned.
