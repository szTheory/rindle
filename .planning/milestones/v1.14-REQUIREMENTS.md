# Requirements — Milestone v1.14 Bulk Owner-Erasure Orchestration

**Milestone:** v1.14 — started 2026-05-27
**Goal:** Extend v1.10 single-owner erasure with batch preview/execute and operator
surface for GDPR/compliance scale.

## Requirements

### Bulk Erasure Policy & Contract

- [x] **BULK-01**: Operator can preview erasure for a bounded batch of owners and receive
      an aggregate report with per-owner `owner_erasure_report()` entries plus batch totals.

- [x] **BULK-02**: Batch preview enforces a configurable maximum owner count (default bounded)
      and returns a tagged error when the batch exceeds the limit.

### Batch Implementation

- [x] **BULK-03**: Batch execute processes each owner through the existing single-owner
      erasure planner with transactional per-owner isolation.

- [x] **BULK-04**: Adopter can execute batch owner erasure through one public API call that
      reuses `Rindle.Internal.OwnerErasure` and preserves the v1.10 report vocabulary.

- [x] **BULK-05**: Re-running batch erasure for the same owner set is idempotent and returns
      stable no-op/report results for already-cleared owners.

### Operator Surface

- [x] **OPS-02**: Operator can run batch owner-erasure preview or execute from
      `mix rindle.batch_owner_erasure` with documented CLI contract.

### Proof & Support Truth

- [x] **PROOF-05**: Hermetic proof covers batch preview aggregation, per-owner isolation on
      execute, partial failure handling, idempotent rerun, and retained shared-asset semantics.

- [x] **TRUTH-03**: Guides document batch erasure as the supported multi-owner orchestration
      surface and explicitly defer force-delete, admin UI, and scheduler workflows.

## Future Requirements

| Bulk Mux/streaming/tus operations | Unrelated to lifecycle erasure wedge |
