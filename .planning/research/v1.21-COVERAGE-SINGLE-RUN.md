# v1.21 Research — Single-Run Coverage (eliminate the double ExUnit suite run)

**Milestone:** v1.21 "CI/DX Reliability Tail" — Research Area 1 of 5
**Seed:** SEED-004 (CI `:epipe` flake from double suite run)
**Researched:** 2026-06-26
**Confidence:** HIGH (grounded in the ExCoveralls 0.18.5 source vendored in `deps/`, cross-checked against the official README + HexDocs)
**Posture:** ONE locked recommendation. Alternatives are rationale / rejected-path only.

---

## 1. Problem & evidence

Every CI job that runs the default ExUnit suite executes the **entire suite twice**: once to gate, once to emit `cover/excoveralls.json` for artifact upload. This doubles test wall-clock and **doubles exposure to the non-deterministic `:epipe` (broken-pipe) failures** in the ffmpeg/subprocess-spawning tests. When the flake lands on the *second* run, the job reds even though the gating run was clean — observed twice on 2026-06-26 (PR #45 `Integration` / `ffmpeg_test.exs:32`; post-merge `Quality (1.17,27)` / `lifecycle_repair_test.exs:122`), both cleared on rerun. It is a low-grade but real trust/velocity tax that also momentarily blocks release-please.

### The exact ci.yml steps (3 lanes affected)

There are **three** coverage-step sites, and they are **not** all structured the same way — this asymmetry matters for the fix:

| Lane | Gating run (1st execution) | JSON run (2nd full execution) | ci.yml anchor |
|------|----------------------------|-------------------------------|----------------|
| **`quality`** (matrix: 1.15/26, 1.17/27 — the PR gate) | `mix coveralls --slowest 20` (line ~187–196) — **this is the merge-blocking gate** | `mix coveralls.json` (`if: always()`, line ~200–202) | `.github/workflows/ci.yml` 184–211 |
| **`integration`** (MinIO) | `mix test <files> --include integration/minio --slowest 20` (line ~373–383) — **gate is plain `mix test`, NOT `mix coveralls`** | `mix coveralls.json` (`if: always()`, line ~387–389) | 370–399 |
| **`adoption-demo` / install-smoke proof job** | (install-smoke + version checks; not a unit-suite gate) | `MIX_ENV=test mix coveralls.json` (`if: always()`, line ~649–651) | 646–662 |

All three feed the same artifact step: `actions/upload-artifact` with `path: cover/excoveralls.json` + `_build/test/junit/rindle-junit.xml` (`if-no-files-found: warn`).

**Repo-grounded facts that constrain the fix:**

- `mix.exs:43` → `test_coverage: [tool: ExCoveralls]`. `mix.exs:47–61` (`cli/0`) sets `preferred_envs` so `coveralls`, `coveralls.json`, etc. run under `MIX_ENV=test`.
- **No coveralls config exists** — there is no `coveralls.json` config file and no `:excoveralls` block in `config/`. So `minimum_coverage` defaults to `0` (verified: `ExCoveralls.Stats.ensure_minimum_coverage/1` → `coverage_options["minimum_coverage"] || 0`, and `if minimum_coverage > 0` guards the check). **Today the "coverage gate" is purely: tests pass (ExUnit non-zero exit) + coverage computed.** There is no threshold being enforced yet.
- excoveralls is pinned at **0.18.5** (`mix.lock`).
- The `quality` lane comment explicitly states *"The gating step STAYS `mix coveralls` (NOT replaced by coveralls.json)."* Section 7 proves WHY that comment is correct and how the recommendation honors it rather than violating it.

### Why "just run coveralls.json and drop the gate run" is unsafe (the load-bearing fact)

Reading the vendored 0.18.5 source directly:

- `ExCoveralls.Local.execute/2` (the `local` / `mix coveralls` analyzer) ends with **`ExCoveralls.Stats.ensure_minimum_coverage(stats)`** → `check_coverage_threshold` → `exit({:shutdown, 1})` on breach. (`deps/excoveralls/lib/excoveralls/local.ex:24`, `stats.ex:237–250`.)
- `ExCoveralls.Json.execute/2` **does NOT call `ensure_minimum_coverage`** — it only writes the file then prints a summary. (`deps/excoveralls/lib/excoveralls/json.ex:12–16`.)

The official docs corroborate: *"minimum_coverage … causes the `mix coveralls` and `mix coveralls.html` tasks to exit with status code 1 … Only two tasks enforce this: `mix coveralls` and `mix coveralls.html`."* **`mix coveralls.json` is a non-gating analyzer.** Swapping the gate to `coveralls.json`'s exit code would silently drop threshold enforcement the moment a `minimum_coverage` is ever introduced — exactly the footgun the ci.yml comment guards against. This rules out option (b).

## 2. Options

All snippets verified against the 0.18.5 source path: `Mix.Tasks.Coveralls.do_run/2` → `Runner.run("test", ["--cover"] ++ args)` → ExCoveralls cover-callback → `Enum.each(types, &analyze/3)` over **one** stat set.

### (a) `mix coveralls.multiple --type ... ` — ONE run, console gate + JSON ✅ RECOMMENDED

**Mechanism.** `Mix.Tasks.Coveralls.Multiple` (`deps/excoveralls/lib/mix/tasks.ex:329–362`) parses repeated `--type` flags and calls `do_run(args, type: ["html", "json", ...])`. `ExCoveralls.execute/3` then does `types = List.wrap(options[:type]); Enum.each(types, &analyze(stats, &1, options))` (`excoveralls.ex:64–69`) — i.e. **all requested formatters run off a single test execution**. Crucially, including `local` in the type list routes through `Local.execute/2`, which still calls `ensure_minimum_coverage` — so the gate is byte-identical to today's `mix coveralls`. Added upstream in PR #303 ("Add coveralls.multiple command", CHANGELOG line 63). Its stated purpose: *"execute multiple coveralls task at once without re-running tests."*

**Snippet (drop-in for the `quality` lane):**
```yaml
- name: Run tests with coverage (single run → console gate + JSON artifact)
  run: |
    set -o pipefail
    mix coveralls.multiple --type local --type json --slowest 20 2>&1 | tee /tmp/test.out
    {
      echo "## Test timing — ${{ github.job }} (${{ matrix.elixir }}/${{ matrix.otp }})"
      echo '```'
      grep -E 'Randomized with seed|slowest|^\s+\* test ' /tmp/test.out | tail -30 || true
      echo '```'
    } >> "$GITHUB_STEP_SUMMARY"
```
- `--type local` = exactly what `mix coveralls` does (prints the `[TOTAL] xx%` table **and** runs `ensure_minimum_coverage` → the gate).
- `--type json` = writes `cover/excoveralls.json` (same path the upload step already expects).
- `--slowest 20` survives: `do_run` strips only ExCoveralls' own switches and forwards the rest to `mix test` (`parse_common_options`, `tasks.ex:65–95`).
- The old separate `Generate coverage JSON artifact` step is **deleted** — the JSON now comes from the gating run itself.

**Gate-semantics verdict: PRESERVED EXACTLY.** Same `do_run` → `mix test --cover` path, same `Local.execute` threshold `exit({:shutdown,1})`, same ExUnit-failure exit propagation. The console gate is still the merge-blocking signal; JSON is a side-output of the same run, not a second source of truth.

**Pros:** one suite run (halves wall-clock for the test phase AND halves `:epipe` exposure — the SEED-004 (a) win); zero new config; no `lib/` change; gate unchanged; smallest diff; symmetrical across lanes; the JSON is now *guaranteed consistent* with the gated numbers (today the two runs could in principle diverge under flake/seed differences).
**Cons:** `coveralls.multiple` is **not documented in the README** (it exists and is tested upstream, but is a lesser-known task) — a future maintainer reading only HexDocs might not recognize it; mitigated by an inline comment. `--type local` prints the full file table to the log (slightly noisier than today's `coveralls` default, which is the same table — so effectively no change).

### (b) Run only `mix coveralls.json`, derive the gate from its exit code ❌ REJECTED

**Mechanism.** Single `mix coveralls.json`; treat its exit code as the gate.
**Snippet:** `run: mix coveralls.json --slowest 20`
**Gate-semantics verdict: BROKEN.** `Json.execute` never calls `ensure_minimum_coverage` (source above). Today (no threshold) it would *appear* to work because ExUnit failures still exit non-zero — but it silently drops threshold enforcement, and the moment Area-adjacent work or a maintainer adds `minimum_coverage: N`, the gate would pass below threshold with no signal. Directly contradicts the ci.yml invariant comment. **Reject.**

### (c) `mix test --cover` for the gate + ExCoveralls only for JSON ❌ REJECTED

**Mechanism.** Built-in `mix test --cover` gates; a second pass emits JSON.
**Why reject.** `mix test --cover` uses ExCoveralls as the configured `test_coverage` tool, so it would print a summary — but to also get `cover/excoveralls.json` you still need a JSON analyzer pass, i.e. **you're back to two runs** unless you use the `--export-coverage`/`--import-cover` bridge (that's option (d), and it's strictly more moving parts than (a)). No advantage over (a); loses the unified single-task ergonomics. **Reject.**

### (d) Two tasks, but the 2nd imports coverdata instead of re-running tests ⚠️ VIABLE FALLBACK ONLY

**Mechanism.** `mix test` (or `mix coveralls`) with **`--export-coverage NAME`** writes `cover/NAME.coverdata`; a second `mix coveralls.json --import-cover cover` reads it and emits JSON **without re-running tests** (`ExCoveralls.start/2` honors `options[:import_cover]` → `Cover.import/1`, `excoveralls.ex:46–49`; `--import-cover` is a real switch, `tasks.ex:36`).

**Snippet:**
```yaml
- name: Run tests with coverage (gate + export coverdata)
  run: mix coveralls --slowest 20 --export-coverage default
- name: Emit coverage JSON from exported coverdata (no re-run)
  if: always()
  run: mix coveralls.json --import-cover cover --no-start
```
**Caveats / footguns:** `--export-coverage` is a `mix test`/cover flag whose interaction with the `coveralls`-task wrapper is fiddlier than the single `multiple` task; the import pass still boots the app/compiles and runs `mix test` with an **empty/zero-test** filter unless carefully muted (it does not necessarily *skip* test execution cleanly — the clean "import only" story is better-supported in the umbrella/partitioned-coverage workflow than for a single-run dual-output). More steps, more surface for drift, and `if: always()` on the 2nd step can mask a missing coverdata file.
**Gate-semantics verdict: PRESERVED** (gate is still `mix coveralls`), but **strictly more complex than (a)** for the same outcome. Hold as contingency if `coveralls.multiple` ever proves insufficient.

### (e) Ecosystem-idiomatic baseline

The most common single-run dual-output pattern in Elixir CI is exactly **`mix coveralls.json`** (when the only consumer is Coveralls.io / Codecov upload, the JSON run *is* the gate because people set `minimum_coverage` and accept that JSON doesn't enforce it — or they use `mix coveralls.github` to post). Rindle's situation is slightly unusual: it wants **both** a local console gate **and** a JSON artifact for upload **from one run**. `coveralls.multiple` is the purpose-built tool for precisely that "multiple outputs, one execution" need.

## 3. Idiomatic-for-Elixir analysis

- **One `mix test` execution per lane is the norm.** Running the suite twice purely to change the *output formatter* is non-idiomatic; ExCoveralls' formatter list (`analyze/3` fan-out) exists specifically so you choose outputs without re-executing.
- **`mix coveralls` is the canonical gate task** (it carries `ensure_minimum_coverage`); `mix coveralls.json`/`.html`/`.cobertura`/`.lcov` are *reporters*. The idiomatic move is "gate with `coveralls` (or `coveralls.multiple` including `local`), report with the same run."
- **`MIX_ENV=test` via `preferred_envs`** is already correctly wired (`mix.exs cli/0`), so `coveralls.multiple` inherits the right env with no extra `MIX_ENV=test` prefix (the install-smoke lane's explicit `MIX_ENV=test mix coveralls.json` is belt-and-suspenders and can stay or be dropped).
- **`--slowest 20` + `tee` for the run summary** (OBS-02 from v1.20) is a Rindle-local convention; (a) preserves it verbatim.

## 4. Peer-lib lessons

| Project | What they do | Lesson |
|---------|--------------|--------|
| **ExCoveralls' own CI** (parroty/excoveralls) | Runs the suite once; posts via `mix coveralls.github`. Does not double-run. | **Emulate:** one execution; choose the formatter, don't re-run. |
| **Oban, Ecto, Phoenix, Plug, Bandit, Req** | Predominantly a single `mix test`/`mix coveralls` (or `coveralls.github`) invocation per matrix cell; coverage upload is a side-output, never a second full suite run. None re-run the suite to change output format. | **Emulate:** coverage artifact is a side-effect of the gating run, not a separate job step that re-executes tests. |
| **Codecov/Coveralls Elixir guides** | Recommend `mix coveralls.json` then upload the file; for projects needing a local gate too, `minimum_coverage` on `mix coveralls`. | **Footgun to avoid:** assuming `coveralls.json` enforces `minimum_coverage` — it doesn't. (Confirmed in source + docs.) |
| **Ruby SimpleCov (transferable)** | SimpleCov hooks `at_exit` to emit *all* configured formatters (HTML + JSON + …) from **one** test run via `SimpleCov.formatters = [...]`. Re-running RSpec twice for two report formats is universally treated as a mistake. | **Transferable principle:** "many formatters, one run" is the cross-language standard. `coveralls.multiple --type local --type json` is the direct Elixir analogue of `SimpleCov::Formatter::MultiFormatter`. |

**FOOTGUN to avoid (Rindle-specific):** the `integration` lane currently gates with plain `mix test` (no coverage) and then runs a *full second* `mix coveralls.json`. That second run is **100% redundant test execution** that produces coverage numbers nobody gates on — it's pure `:epipe` surface for zero gate value. The fix should fold JSON into the single integration run (or, if integration coverage isn't wanted in the aggregate, drop the JSON step there entirely — see §7 note).

## 5. DX / contributor experience

- **Local↔CI parity.** `mix ci` (`mix.exs aliases`) ends in a plain `test` task, and `RUNNING.md`/CONTRIBUTING describe `mix coveralls`. After the change, a contributor reproduces the *gate* with `mix coveralls --slowest 20` (unchanged) and the *full CI step* with `mix coveralls.multiple --type local --type json --slowest 20`. Recommend documenting the latter near the coverage docs so CI is reproducible verbatim. No change to `mix ci`'s final `test` is required (the merge-blocking gate alias stays simple); the dual-output is a CI-only concern.
- **Error legibility.** On failure, contributors see one ExUnit failure block (today they can see it twice — once per run — which is actively confusing, e.g. "the gate passed but the job is red"). Collapsing to one run makes the red/green signal honest: if the suite is green, the job is green.
- **Least surprise.** The single visible change is *"tests run once now."* The console gate output (`[TOTAL] xx%` table) is identical. `cover/excoveralls.json` lands at the same path. The artifact upload step is unchanged.
- **One legibility cost to mitigate:** `coveralls.multiple` is README-undocumented. **Mitigation:** an inline ci.yml comment ("single-run dual-output: `local` = console gate incl. minimum_coverage; `json` = artifact; one suite execution — see ExCoveralls `Mix.Tasks.Coveralls.Multiple`") so the next maintainer isn't puzzled.

## 6. Design-pillar pass

| Pillar | How the recommendation serves it |
|--------|----------------------------------|
| **Determinism / reliability** | Halves `:epipe` exposure by removing the redundant second suite execution per lane — the primary SEED-004 win. JSON numbers now provably match the gated numbers (one run, one stat set). |
| **Performance / wall-clock** | Removes a full second ExUnit suite run from every default-suite job (`quality` ×2 matrix cells, `integration`). Directly cuts test-phase wall-clock on the PR critical path that v1.20 worked to shrink. |
| **Maintainability** | Net **fewer** ci.yml steps (3 "Generate coverage JSON" steps collapse into the gating step). Zero new config files. No `lib/` change. |
| **Parity (local↔CI)** | Single documented command reproduces the CI coverage step locally; gate reproduction (`mix coveralls`) is unchanged. |
| **Security / supply-chain** | No new deps, no version bump, no Action changes; `excoveralls 0.18.5` already vendored. Smaller CI surface. |

## 7. LOCKED recommendation

**Adopt option (a): replace the gating-run + separate-JSON-run pair with a single `mix coveralls.multiple --type local --type json` invocation in every default-suite lane.** One ExUnit execution emits both the merge-blocking console gate (`local` analyzer → `ensure_minimum_coverage`, byte-identical gate semantics to today's `mix coveralls`) and `cover/excoveralls.json` for artifact upload. Delete the three standalone `Generate coverage JSON artifact` steps. This halves test wall-clock and `:epipe` exposure on every default-suite job, requires zero new config and zero `lib/` change, and the ci.yml invariant ("gate STAYS `mix coveralls`, not `coveralls.json`") is *honored, not superseded* — the gate is still the `local` analyzer's threshold/exit, never `coveralls.json`'s non-gating exit code.

### Why this safely supersedes the ci.yml comment's literal wording
The comment says "gating step STAYS `mix coveralls` (NOT replaced by `coveralls.json`)" because **`coveralls.json` does not enforce `minimum_coverage`** (proven in §1/§2 from source). `coveralls.multiple --type local --type json` keeps the **`local`** analyzer in the gate path, so `ensure_minimum_coverage` still runs. The comment's *intent* (don't let the non-gating JSON formatter become the gate) is fully preserved; only the literal task name changes. Update the comment to reflect this.

### Exact changes — all 3 lanes

**Lane 1 — `quality` (`.github/workflows/ci.yml` ~184–211):**
- Replace the `Run tests with coverage` step's `mix coveralls --slowest 20` with `mix coveralls.multiple --type local --type json --slowest 20` (keep the `tee` + run-summary block verbatim).
- **Delete** the `Generate coverage JSON artifact` step (~198–202).
- Update the leading comment to the §7 wording.

**Lane 2 — `integration` (~370–399):**
- The gate is currently plain `mix test <files> --include …`. Two clean sub-options:
  - **(2a, preferred)** Convert each gating `mix test <file> --include <tag>` to `mix coveralls.multiple --type local --type json --include <tag> <file> --slowest 20` so the JSON is produced from the same run; **delete** the standalone `Generate coverage JSON artifact` step.
  - **(2b, if integration coverage is not wanted in the aggregate)** Simply **delete** the redundant `mix coveralls.json` step entirely and keep the gate as `mix test`. The integration JSON today is a full redundant re-run feeding an artifact nobody gates on. *Escalation-light decision — see flag below.*
- **Recommendation: 2b** unless a downstream consumer actually merges integration coverage. The integration lane's value is the MinIO/integration behavior, not its coverage %; dropping the second run is the cleanest `:epipe`-surface reduction.

**Lane 3 — `adoption-demo` / install-smoke proof (~646–662):**
- This job does **not** run the unit suite as its gate; its `MIX_ENV=test mix coveralls.json` is a *standalone full suite run* purely to produce an artifact. **Delete** the `Generate coverage JSON artifact` step here. (If a coverage artifact from this job is genuinely consumed, fold it into an existing test step via `coveralls.multiple`; otherwise drop it — it is a full redundant suite execution with no gate.)
- Keep the `Upload JUnit + coverage artifacts` step with `if-no-files-found: warn` (it already tolerates a missing `cover/excoveralls.json`).

**`mix.exs`:** **No change required.** `coveralls.multiple` is preferred-env-inherited via the existing `coveralls`/`coveralls.json` entries running under `:test`; `mix test` env applies. (Optional polish: add `"coveralls.multiple": :test` to `cli/0 preferred_envs` for explicitness — harmless, additive. **Recommended** as a one-line clarity add.)

### Coherence with the other 4 v1.21 areas and v1.20 invariants
- **SEED-004 (b) "harden subprocess against `:epipe`":** complementary, not conflicting — (a) removes the *redundant* exposure; the (b) area fixes the *remaining* single run's broken-pipe race. Ship together.
- **v1.20 invariants:** ✅ no `ci.yml` rename / `name: CI` unchanged; ✅ `CI Summary` aggregate untouched (`skipped`==pass preserved); ✅ release full-verification gate unchanged; ✅ **zero `lib/` change** (CI/mix-config only — confirmed); ✅ gate remains the `mix coveralls`-equivalent console signal.

### Blast-radius / escalation flag
- **Blast radius: LOW.** CI-config-only, reversible in one commit, no public API, no deps, no release coupling touched. Resolve by the agent per the decide-by-default contract.
- **One soft escalation to surface (not block):** *integration-lane coverage drop (2b)* — confirm no dashboard/aggregate consumes the `integration` job's `cover/excoveralls.json` before deleting it. If a consumer exists, use 2a (fold into single run) instead of dropping. This is a "name the boundary, proceed unless a consumer is found" call, not a maintainer-arbitration gate.

### Requirement-ready bullets (lift directly into REQ-IDs)
- The `quality` lane runs the default ExUnit suite **exactly once** per matrix cell and emits both the merge-blocking console coverage gate and `cover/excoveralls.json` from that single run (`mix coveralls.multiple --type local --type json --slowest 20`).
- The merge-blocking coverage gate continues to run the **`local`** analyzer (i.e. `ensure_minimum_coverage` is still exercised); the gate's pass/fail is never derived from `mix coveralls.json`'s exit code.
- The standalone `Generate coverage JSON artifact` step (the redundant second `mix coveralls.json` suite run) is **removed from all three lanes** (`quality`, `integration`, `adoption-demo`/install-smoke).
- The `integration` lane no longer executes the suite a second time for coverage JSON (either folded into its single gating run, or the redundant JSON step is dropped after confirming no consumer).
- `cover/excoveralls.json` continues to be produced at the same path and uploaded by the existing `Upload JUnit + coverage artifacts` step, with `if-no-files-found: warn` preserved.
- A contributor can reproduce the CI coverage step locally with one documented command (`mix coveralls.multiple --type local --type json --slowest 20`) and the gate alone with `mix coveralls` (parity preserved; documented near the coverage docs / `RUNNING.md`).
- No `lib/` source changes and no new dependencies are introduced by the single-run coverage change (CI + `mix.exs` config only).
- The ci.yml comment is updated to explain that `--type local` preserves the gate and `--type json` is the artifact side-output of the same run.
