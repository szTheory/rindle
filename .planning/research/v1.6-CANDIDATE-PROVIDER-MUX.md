# v1.6 Candidate: Provider Boundary + Mux Reference Adapter — Locked Recommendation

**Date:** 2026-05-06
**Confidence:** HIGH for provider-boundary + Mux scope; MEDIUM for "v1.6 now" sequencing — provider-boundary is genuinely the highest-leverage breadth bet for video credibility, but it widens the public abstraction more than GCS does. Verdict is locked: ship it as v1.6 only if the synthesizer values *strategic positioning for video adopters* over *additive breadth across all media kinds*.
**Recommendation score:** **8/10** for "v1.6 now"

This memo refreshes and supersedes `.planning/research/PROVIDER-ADAPTERS-MILESTONE-MEMO.md` (2026-05-05). Versions re-verified 2026-05-06 against Hex.pm.

## 1. Locked scope (one provider only, additive contract)

**In scope (locked):**

1. Promote `Rindle.Streaming.Provider` from reserved behaviour to a real adapter contract (Section 4).
2. Ship **one** reference adapter: `Rindle.Streaming.Provider.Mux`. No second provider, no parity, no shim. Cloudflare Stream is mentioned only as a contrast oracle to keep the abstraction honest.
3. `Rindle.Delivery.streaming_url/3` gains a deterministic provider-vs-progressive dispatch rule (Section 5.1).
4. Profile DSL gains a `:streaming` key (Section 5.2). Adopters opt in per profile; no per-variant streaming knob.
5. New `media_provider_assets` Ecto table for durable provider state (Section 6).
6. Mountable signed-webhook plug (`Rindle.Delivery.WebhookPlug`) (Section 5.3).
7. Oban workers for provider sync, webhook ingestion, and dead-letter (Section 7).
8. Profile-aware mapping from named Rindle presets to Mux playback policy + asset settings — never raw Mux knobs in the DSL.
9. Five new locked error atoms; extend the v1.4 `Rindle.Error` vocabulary (Section 8.4).
10. Telemetry events extend (do not break) the frozen v1.4 delivery contract (Section 8.5).
11. CI proof: VCR-style cassette tests against the Mux SDK + a real-network "soak" lane behind a `MUX_TOKEN_ID` secret, gated like the live publish lane.

**Out of scope (deferred to v1.7+ with reason):**

- Second provider (Cloudflare Stream / Bunny Stream / Cloudinary Video). One adapter only — see Section 13.
- Live streaming, RTMP/WebRTC ingest, DRM (Widevine/FairPlay/PlayReady), captions, multi-region failover.
- Replacing `Rindle.Processor.AV`. Mux is *additive*; FFmpeg-driven progressive delivery stays intact and is the fallback when a profile does not opt into a provider.
- Direct-to-Mux upload from the browser (Mux's authenticated PUT URL flow). v1.6 ships only the *server-pushed* asset path: `Rindle.Processor.AV` produces a 720p mp4 → Rindle uploads it to Mux from server context (or `Mux.Video.Assets.create(%{input: signed_storage_url})` against the existing Rindle storage URL). Browser→Mux direct-upload is a v1.7 add (Section 13).
- tus protocol. tus is *not* a precondition for provider work; Mux's resumable PUT is its own protocol.
- Public Cloudflare Stream / Bunny Stream / Transloadit adapters in core. Pluggable, not bundled.

## 2. Phase breakdown (concrete, with names + REQ-IDs + success criteria)

Five phases. Each phase ends with a single observable success criterion that maps to a CI assertion.

### Phase 33 — Provider Boundary + State Schema (`STREAM-01..STREAM-09`)

**Goal:** Lock the public seam without adding any Mux code. Land the Ecto migration, behaviour, capability vocabulary, profile DSL key, dispatch rule, and the `:streaming_not_configured` plus four sibling error atoms — so downstream adapter work has a stable contract.

- `STREAM-01`: New `Rindle.Streaming.Capabilities` module (mirrors `Rindle.Storage.Capabilities` and `Rindle.Processor.Capabilities`). Vocabulary: `:signed_playback`, `:public_playback`, `:webhook_ingest`, `:server_push_ingest`, `:direct_creator_upload`. Direct-creator-upload is *reserved* for v1.7.
- `STREAM-02`: Promote `Rindle.Streaming.Provider` from reserved behaviour to runtime contract. Locked callbacks per Section 4.
- `STREAM-03`: Ecto migration `create_media_provider_assets` adds the `media_provider_assets` table per Section 6. Idempotent and additive.
- `STREAM-04`: `Rindle.Domain.MediaProviderAsset` schema + changeset + state FSM (`pending → uploading → processing → ready | errored | deleted`).
- `STREAM-05`: Profile DSL extension per Section 5.2 — `:streaming` key validated through NimbleOptions; image-only and AV-only profiles compile unchanged.
- `STREAM-06`: `Rindle.Delivery.streaming_url/3` dispatch rule per Section 5.1; provider lookup is a single `repo.get_by/2` keyed by `(asset_id, profile, status: :ready)`. Provider-asset-not-ready returns `{:error, :provider_asset_not_ready}`; no fallback to progressive (locked rule, Section 8.1).
- `STREAM-07`: Extend `Rindle.Error` with the five locked atoms (Section 8.4).
- `STREAM-08`: Extend `Rindle.Capability.report/0` to include detected streaming providers and signed-playback configuration status.
- `STREAM-09`: ExUnit parity gate — assert exact message text and reason atom for the five new error variants (matches the AV-06-05 freeze pattern).

**Success criterion:** `mix test` passes with the new behaviour, schema, DSL key, and error vocabulary in place; an image-only profile and an AV-only profile both still compile and exercise the existing v1.4 lifecycle byte-for-byte. No Mux code merged yet.

### Phase 34 — Mux REST Adapter + Server-Push Sync (`MUX-01..MUX-08`)

**Goal:** First real adapter. Server pushes a finished mp4 to Mux from existing `Rindle.Processor.AV` output; durable provider state tracks Mux asset id + playback id; signed-playback URLs work.

- `MUX-01`: Add `{:mux, "~> 3.2"}` (currently `3.2.2`, 2024-07-02) and `{:jose, "~> 1.11"}` (currently `1.11.12`, 2025-11-20) as **optional** deps. Adopters that don't enable streaming pay zero transitive cost. (Note: `mux` already pulls `tesla ~> 1.0` and `jose ~> 1.9`. Tesla `1.17.0` (2026-04-18) is current and Phoenix-compatible.)
- `MUX-02`: `Rindle.Streaming.Provider.Mux` implements the locked behaviour. REST calls dispatch through `Mux.Video.Assets.create/2`, `Mux.Video.Assets.get/2`, `Mux.Video.Assets.delete/2`. Authentication uses Mux's HTTP basic auth (`MUX_TOKEN_ID:MUX_TOKEN_SECRET`); credential resolution is `Application.get_env(:rindle, Rindle.Streaming.Provider.Mux)` keys per Section 8.6.
- `MUX-03`: `Rindle.Streaming.Provider.Mux.create_asset/3` is invoked by a new `Rindle.Workers.MuxIngestVariant` Oban worker. Worker accepts `(asset_id, profile, variant_name)`, fetches the variant's storage URL via existing `Rindle.Delivery.url/3` (private signed URL with TTL ≥ 30 minutes), calls `Mux.Video.Assets.create/2` with `%{input: signed_url, playback_policies: [policy_for_profile(profile)]}`, persists the resulting `provider_asset_id` + `playback_id` into `media_provider_assets`, and transitions state `pending → uploading → processing`.
- `MUX-04`: Signed playback URL minting goes through Mux's own `Mux.Token.sign/2` helper (uses JOSE under the hood). Rindle owns the wrapper `Rindle.Streaming.Provider.Mux.signed_playback_url/3` that builds `https://stream.mux.com/{playback_id}.m3u8?token={jwt}` with `aud: "v"`, `exp: now + signed_url_ttl_seconds(profile)`, `kid: signing_key_id`. Token TTL respects the existing `signed_url_ttl_seconds` profile policy from v1.4.
- `MUX-05`: Idempotent server-push: worker is `unique: [period: 86_400, fields: [:worker, :args], states: [:scheduled, :executing, :retryable, :completed]]` keyed on `(asset_id, profile, variant_name)`. Re-running yields the same `media_provider_assets` row.
- `MUX-06`: Atomic-promote on flip-to-`ready`: re-fetch source asset before transitioning the provider-asset row, abort if `recipe_digest` or `storage_key` changed (mirrors `AV-03-10`). Re-upload-during-transcode race must not strand a provider asset.
- `MUX-07`: Polling fallback Oban worker `Rindle.Workers.MuxSyncProviderAsset` runs at `unique: [period: 60]` for any `processing` `media_provider_assets` row older than `provider_polling_floor_seconds` (default 30). This is *defensive* — webhooks remain primary. If a provider asset stays in `processing` longer than `provider_stuck_threshold_seconds` (default 7200, the v1.4 max-duration cap × 4 to allow Mux queue depth), worker transitions to `errored` with reason `:provider_asset_stuck` and emits `[:rindle, :provider, :sync, :stuck]` telemetry.
- `MUX-08`: Telemetry: `[:rindle, :provider, :ingest, :start | :stop | :exception]` and `[:rindle, :provider, :sync, :resolved | :stuck]`. Documented measurement / metadata schemas (Section 8.5).

**Success criterion:** Cassette-based ExUnit suite drives a 720p sample through `MuxIngestVariant`; the resulting `media_provider_assets` row reaches `ready` state via simulated webhook (Phase 35); `streaming_url/3` returns a Mux-signed playback URL whose JWT verifies against the test signing-key fixture.

### Phase 35 — Signed-Webhook Plug + Idempotent Ingest (`MUX-09..MUX-14`)

**Goal:** Webhooks become the primary readiness signal. Cryptographically verified, replay-protected, secret-rotation-aware, and idempotently processed via Oban.

- `MUX-09`: `Rindle.Delivery.WebhookPlug` is a mountable `Plug` (router macro: `Rindle.Delivery.WebhookPlug.routes(:rindle_mux, "/webhooks/rindle/mux")`). Plug is provider-aware (`provider: Rindle.Streaming.Provider.Mux`), reads the raw body via the canonical Phoenix raw-body cache pattern (a `Rindle.Delivery.WebhookBodyReader` is shipped, plus copy-pasteable `endpoint.ex` snippet). Bypasses `Plug.Parsers` JSON decode for the webhook scope.
- `MUX-10`: Signature verification delegates to `Mux.Webhooks.verify_header/4` (HMAC-SHA256, `t=…,v1=…` header format, 300s default tolerance — matches Stripe's posture). Tolerance is config-overridable via `:webhook_tolerance_seconds` (default 300, max 900). Constant-time comparison is provided by the SDK; Rindle does not roll its own.
- `MUX-11`: Multi-secret rotation. `:webhook_secrets` accepts a list `[primary | rotating]`; Plug tries each in order; first match wins; metric `[:rindle, :provider, :webhook, :secret_used]` records which secret matched (so operators can confirm rotation completed before retiring the old secret). Aligns with Stripe's 24h grace-period precedent.
- `MUX-12`: On verified webhook, the plug enqueues `Rindle.Workers.IngestProviderWebhook` with the *raw* event payload (so the worker, not the request, owns processing latency). Plug returns `202 Accepted` after enqueue commit. Failed signature returns `400` with `:provider_webhook_invalid`. Replay-window failure returns `400` with `:provider_webhook_invalid` (single tagged atom — operators distinguish via telemetry metadata, not error variants — keeps the public vocabulary tight).
- `MUX-13`: `IngestProviderWebhook` worker is `unique: [period: 86_400, fields: [:args], states: [:scheduled, :executing, :retryable]]` keyed on `event.id` (Mux event UUID). Idempotent re-delivery is a no-op. Worker dispatches on `event.type`:
  - `video.asset.ready` → flip `media_provider_assets` to `ready`, persist `playback_ids[]`, broadcast `:provider_asset_ready` PubSub.
  - `video.asset.errored` → flip to `errored`, store `last_sync_error`, broadcast `:provider_asset_errored`.
  - `video.upload.asset_created` → link upload-id to asset-id (only relevant after Phase 36 in v1.7+).
  - `video.asset.deleted` → flip to `deleted`.
  - All other events: persist to `media_provider_assets.last_event_at`, no state change. Unknown-event silence is intentional (Mux ships new events; library shouldn't crash on novelty).
- `MUX-14`: Dead-letter: workers exceeding `max_attempts: 5` (Oban default backoff) leave the row in last known state with `last_sync_error` set. `mix rindle.runtime_status` (shipped v1.5) gains a `--provider-stuck` filter that lists rows where `state in (processing, uploading)` AND `updated_at < now() - provider_stuck_threshold`.

**Success criterion:** Bypass-driven ExUnit test posts a fixture `video.asset.ready` payload with a real HMAC signature against the Plug, the worker idempotently flips the matching `media_provider_assets` row, and a second identical post is a no-op (unique constraint passes the assertion). Replay attack (timestamp 600s old) returns 400.

### Phase 36 — Public DX, Onboarding, CI Proof (`MUX-15..MUX-19`)

**Goal:** Lock the adopter onboarding path and prove the package-consumer story matches v1.5's bar.

- `MUX-15`: Profile preset: `Rindle.Profile.Presets.MuxWeb` (alongside the existing `Rindle.Profile.Presets.Web` from v1.4). Demonstrates `:streaming` opt-in with named-preset playback policy (`:signed`).
- `MUX-16`: `mix rindle.doctor` extends to validate streaming configuration: presence of `MUX_TOKEN_ID`, signing key id + RSA private key, webhook secret(s), and that `Mux.Video.Assets.list/1` returns a 200 (smoke ping with a 5s timeout). Reports per-profile streaming status with PASS / FAIL.
- `MUX-17`: New guide `guides/streaming_providers.md` with a Mux-only section. Copy-pasteable: env vars, signing-key creation steps, webhook secret rotation workflow, runtime gotchas (raw body cache, ngrok-style local webhook tunnel guidance, doctor smoke).
- `MUX-18`: Generated-app package-consumer proof gains a `mux-enabled` lane (alongside the existing `image-only` and `av-enabled` lanes from v1.5). Lane uses cassette-based Mux fixtures by default; gated `mux-soak` lane behind a `MUX_TOKEN_ID` GitHub secret runs against real Mux every PR labeled `streaming`.
- `MUX-19`: README + getting-started gain a "Streaming with Mux" subsection that points at the new guide; image and AV onboarding paths remain the canonical first-run story.

**Success criterion:** Fresh `mix phx.new` adopter app installs Rindle, declares `Rindle.Profile.Presets.MuxWeb`, runs `mix rindle.doctor`, uploads a sample mp4, and renders a `<video>` tag whose `src` resolves to a Mux-signed HLS URL — all from CI, all from the published artifact. Mirrors v1.5's `package-consumer-proof` bar.

### Phase 37 (optional, pull-forward only if time) — Browser→Mux Direct Upload (`MUX-20..MUX-23`)

**Goal:** Adopters who don't want server-side ingest cost can let the browser PUT directly to Mux. Skip this phase if the milestone is running long; it's a clean v1.7 addition.

- `MUX-20`: New `Rindle.Streaming.Provider.Mux.create_direct_upload/2` returns `%{upload_url, upload_id, asset_id}` after creating a `media_provider_assets` row in `pending` state with `direct_creator_upload: true`.
- `MUX-21`: `video.upload.asset_created` webhook handler links upload-id to asset-id via `Rindle.Workers.IngestProviderWebhook`.
- `MUX-22`: Capability gate: `Rindle.Streaming.Capabilities.require_streaming(adapter, :direct_creator_upload)`.
- `MUX-23`: LiveView helper `Rindle.LiveView.subscribe(:provider_asset, id)` extends the v1.4 PubSub vocabulary with `:provider_asset_created`, `:provider_asset_ready`, `:provider_asset_errored`.

**Success criterion:** End-to-end LiveView test creates a direct upload, simulates client PUT, posts the `video.upload.asset_created` and `video.asset.ready` webhooks, and the LiveView receives both PubSub events.

## 4. Provider behaviour contract (locked Behaviour with @callback signatures)

```elixir
defmodule Rindle.Streaming.Provider do
  @moduledoc """
  Behaviour for streaming providers. v1.6 ships one reference implementation
  (Mux). Adopters MAY implement this behaviour to plug in additional providers
  in v1.7+; v1.6 explicitly does NOT promise multi-provider parity.
  """

  @typedoc "Provider-side opaque asset identifier."
  @type provider_asset_id :: String.t()

  @typedoc "Public playback locator. For Mux, the playback_id."
  @type playback_id :: String.t()

  @typedoc "Provider state mirrored into media_provider_assets.state."
  @type provider_state ::
          :pending | :uploading | :processing | :ready | :errored | :deleted

  @typedoc "Provider event normalized from a verified webhook payload."
  @type provider_event :: %{
          required(:event_id) => String.t(),
          required(:event_type) => String.t(),
          required(:provider_asset_id) => provider_asset_id() | nil,
          required(:occurred_at) => DateTime.t(),
          required(:raw) => map()
        }

  @typedoc "Capabilities atom set; subset of Rindle.Streaming.Capabilities vocabulary."
  @type capability ::
          :signed_playback
          | :public_playback
          | :webhook_ingest
          | :server_push_ingest
          | :direct_creator_upload

  @callback capabilities() :: [capability()]

  @callback create_asset(profile :: module(), source_url :: String.t(), opts :: keyword()) ::
              {:ok,
               %{
                 provider_asset_id: provider_asset_id(),
                 playback_id: playback_id() | nil,
                 state: provider_state()
               }}
              | {:error, term()}

  @callback get_asset(provider_asset_id()) ::
              {:ok,
               %{
                 provider_asset_id: provider_asset_id(),
                 playback_id: playback_id() | nil,
                 state: provider_state(),
                 raw: map()
               }}
              | {:error, term()}

  @callback delete_asset(provider_asset_id()) :: :ok | {:error, term()}

  @callback signed_playback_url(profile :: module(), playback_id(), opts :: keyword()) ::
              {:ok, %{url: String.t(), kind: :hls | :dash, mime: String.t()}}
              | {:error, term()}

  @callback verify_webhook(raw_body :: binary(), headers :: %{optional(String.t()) => String.t()}, secrets :: [String.t()]) ::
              {:ok, provider_event()}
              | {:error, :provider_webhook_invalid}

  @callback create_direct_upload(profile :: module(), opts :: keyword()) ::
              {:ok, %{upload_url: String.t(), upload_id: String.t(), provider_asset_id: provider_asset_id() | nil}}
              | {:error, term()}
end
```

**Locked guarantees:**

- Every callback returns either `:ok`-tuple or `:error`-tuple. No raised exceptions on the happy path.
- `create_direct_upload/2` is part of the behaviour but is **optional via `@optional_callbacks`**; v1.6 Mux adapter does *not* need to implement it. Reserves shape for v1.7.
- `verify_webhook/3` returns a *normalized* `provider_event`, not a Mux-specific struct. Adapters do the translation. This is the single boundary that prevents Mux-isms from leaking into core.
- `signed_playback_url/3` returns `kind: :hls` for Mux. Cloudflare Stream would also return `:hls`. Returning `:dash` is reserved for v1.7+.
- `capabilities/0` is the runtime introspection point — `mix rindle.doctor` and `Rindle.Capability.report/0` consume it.

## 5. Public API shape

### 5.1 `streaming_url/3` dispatch rule (locked)

Single decision tree, no implicit fallback to progressive when a provider is configured:

```
streaming_url(profile, asset_or_key, opts):
  1. If profile.delivery_policy().streaming is nil:
       -> existing v1.4 progressive path (no behavior change)
  2. If profile.delivery_policy().streaming is configured AND asset is a binary key:
       -> {:error, :streaming_provider_requires_asset_struct}
       (provider dispatch is asset-row-keyed; binary keys are storage-only)
  3. If asset has a media_provider_assets row in (:processing, :uploading, :pending):
       -> {:error, :provider_asset_not_ready}
  4. If asset has a media_provider_assets row in :errored:
       -> {:error, :provider_sync_failed}
  5. If asset has a media_provider_assets row in :ready:
       -> provider.signed_playback_url(profile, playback_id, opts)
  6. If asset has NO media_provider_assets row:
       -> existing progressive path (provider not yet ingested; original FFmpeg variant serves)
       -> emits [:rindle, :delivery, :streaming, :resolved] with kind: :progressive (v1.4 contract preserved)
```

**Rule 6 is the critical adopter-facing decision.** It says: *progressive output remains the safety net while ingest is in flight or the profile newly opted in.* Once a provider asset row exists and is `ready`, the provider takes over. Adopters who want **strict provider-only behavior** can pass `opts: [strict: true]`, which converts step 6 into `{:error, :provider_asset_not_ready}`. Default is non-strict (progressive fallback) because adopters with mixed legacy assets need the gradual migration path.

### 5.2 Profile DSL extension (locked)

```elixir
defmodule MyApp.VideoProfile do
  use Rindle.Profile,
    storage: Rindle.Storage.S3,
    allow_mime: ["video/mp4", "video/quicktime"],
    max_bytes: 500_000_000,
    delivery: %{
      public: false,
      signed_url_ttl_seconds: 7200,  # 2h, v1.4 VOD recommendation
      streaming: %{
        provider: Rindle.Streaming.Provider.Mux,
        playback_policy: :signed,    # :signed | :public  (Rindle named, never raw Mux atoms)
        ingest_mode: :server_push,   # :server_push | :direct_creator_upload (v1.7)
        source_variant: :web         # which Rindle variant feeds the provider; usually 720p preset
      }
    },
    variants: %{
      web: %{kind: :video, format: :mp4, width: 1280, height: 720, bitrate: 2_500_000},
      poster: %{kind: :image, from: :web, scene_detect: true}
    }
end
```

**Locked DSL keys:** `:provider`, `:playback_policy`, `:ingest_mode`, `:source_variant`.
**Forbidden:** raw Mux config (no `:max_resolution_tier`, no `:input` shape, no `:mp4_support`, no `:passthrough`). Translations live inside the adapter, not the profile. Adopters who need provider-specific knobs hand-roll a custom processor.

### 5.3 Mountable webhook plug (locked, yes)

```elixir
# In adopter's MyAppWeb.Endpoint:
plug Plug.Parsers,
  parsers: [:urlencoded, :multipart, :json],
  pass: ["*/*"],
  body_reader: {Rindle.Delivery.WebhookBodyReader, :read_body, []},
  json_decoder: Phoenix.json_library()

# In adopter's MyAppWeb.Router:
forward "/webhooks/rindle/mux", Rindle.Delivery.WebhookPlug,
  provider: Rindle.Streaming.Provider.Mux,
  secrets: {:system, "RINDLE_MUX_WEBHOOK_SECRETS"}  # comma-separated for rotation
```

**Why mountable Plug, not generated controller:** Active Storage's mistake was generating controllers into adopter apps; library-owned Plug means we can patch signature-verification bugs library-side without adopter migrations. Stripe's `Stripe.WebhookPlug` (community pattern) confirms this is the right shape. Locked: yes.

## 6. Migration / Ecto shape (additive — provider state columns / table)

New table; nothing on `media_assets` or `media_variants` changes. Operator queries against existing tables stay byte-identical.

```elixir
defmodule MyApp.Repo.Migrations.CreateMediaProviderAssets do
  use Ecto.Migration

  def change do
    create table(:media_provider_assets, primary_key: false) do
      add :id, :binary_id, primary_key: true
      add :asset_id, references(:media_assets, type: :binary_id, on_delete: :delete_all), null: false
      add :profile, :string, null: false
      add :provider_name, :string, null: false   # "mux" — opaque adapter identifier
      add :provider_asset_id, :string            # Mux asset_id; nullable until create_asset succeeds
      add :playback_ids, {:array, :string}, default: []  # one or more (signed/public)
      add :playback_policy, :string, null: false  # "signed" | "public"
      add :ingest_mode, :string, null: false      # "server_push" | "direct_creator_upload"
      add :state, :string, default: "pending", null: false
      add :last_event_id, :string                  # last processed Mux event UUID (idempotency belt)
      add :last_event_at, :utc_datetime_usec
      add :last_sync_error, :string                # truncated to 4096 chars
      add :raw_provider_metadata, :map, default: %{}  # opaque, never rendered into public paths
      timestamps()
    end

    create unique_index(:media_provider_assets, [:provider_name, :provider_asset_id],
      where: "provider_asset_id IS NOT NULL")
    create unique_index(:media_provider_assets, [:asset_id, :profile, :provider_name])
    create index(:media_provider_assets, [:state])
    create index(:media_provider_assets, [:state, :updated_at])  # for stuck-row sweeper
  end
end
```

**Locked invariants:**

- `provider_asset_id` is a string, never exposed in public-facing paths (Section 9 invariant 14).
- `playback_ids` is an array because Mux supports multiple policies per asset.
- `raw_provider_metadata` stores the opaque last-known provider response; never rendered.
- `unique_index` on `(asset_id, profile, provider_name)` — one provider asset per (asset, profile, provider). Enables clean re-ingest on profile bump.
- All columns are additive; pre-v1.6 adopters running `mix ecto.migrate` see one new table and zero schema drift on existing tables.

## 7. Oban + webhook ingestion contract (idempotency, replay window, retry, dead-letter)

**Workers:**

| Worker | Queue | Concurrency default | Max attempts | Unique | Timeout |
|---|---|---|---|---|---|
| `Rindle.Workers.MuxIngestVariant` | `:rindle_provider` | 2 | 5 | `[fields: [:args], period: 86_400, states: [:scheduled, :executing, :retryable, :completed]]` keyed on `(asset_id, profile, variant_name)` | 5 min |
| `Rindle.Workers.IngestProviderWebhook` | `:rindle_provider` | 4 | 5 | `[fields: [:args], period: 86_400, states: [:scheduled, :executing, :retryable]]` keyed on `event.id` | 30s |
| `Rindle.Workers.MuxSyncProviderAsset` | `:rindle_provider` | 1 | 3 | `[fields: [:args], period: 60]` keyed on `provider_asset_id` | 30s |

**Idempotency:** Every webhook event carries a Mux event UUID. The `IngestProviderWebhook` worker is unique on that UUID for 24 hours. Re-delivery from Mux during outage is a no-op.

**Replay window:** 300s default (Stripe parity). Tolerable up to 900s via `:webhook_tolerance_seconds`. Below 60s is rejected at config-validation time (clock skew is the dominant failure mode).

**Retry curve:** Oban default exponential backoff (`base * 2 ^ (attempt - 1)`). For webhook ingest, default `base: 5s` over 5 attempts ≈ 5s, 10s, 20s, 40s, 80s — total ≤ 3 minutes. Beyond that, the *next* webhook redelivery from Mux (default 1h interval, then exponential) becomes the recovery channel.

**Dead-letter:** Workers exceeding `max_attempts` leave the `media_provider_assets` row with `last_sync_error` populated and `state` unchanged from its last known good state. Operators see the row via `mix rindle.runtime_status --provider-stuck` (Phase 35 / `MUX-14`). No new dead-letter queue — the durable `media_provider_assets` row IS the dead letter.

**Sync polling:** Defensive, low-frequency. `MuxSyncProviderAsset` runs unique-per-minute against rows in `processing`/`uploading` whose `updated_at` is older than 60s. This is the safety net for the rare case where a webhook is missed entirely (e.g. the adopter's webhook endpoint was down for 24h). Polling is *not* the primary path.

## 8. DX / UX / principle-of-least-surprise rules

### 8.1 Locked behavioral rules

1. `streaming_url/3` dispatch rule (Section 5.1) is the **only** decision tree. No magic fallthroughs.
2. `:strict` opt converts the progressive-fallback rule (step 6) into an error. Default is non-strict for migration friendliness.
3. Provider opt-in is **per profile**, not per variant or per asset. Adopters who want mixed-mode run two profiles.
4. Webhook ingest is **always Oban-deferred**. The plug never does work in-request.
5. Webhook signature failures return `400` and emit telemetry; they do not enqueue work. Replay-window failures return the same `400` (operators distinguish via telemetry metadata, not error variants — keeps the public vocabulary tight, matches the v1.4 freeze posture).
6. Raw provider IDs (`provider_asset_id`) **never** appear in URLs, logs, or telemetry metadata exposed to adopters. Only `playback_id` (which is the public-side identifier) does.

### 8.2 Failure-mode vocabulary (locked, additive to v1.4)

The v1.4 `Rindle.Error` vocabulary already includes `:streaming_not_configured` (AV-04) and 7 sibling AV atoms. v1.6 adds **five** atoms — exhaustive, no expansion mid-milestone:

| Atom | Cause | Operator action |
|---|---|---|
| `:provider_asset_not_ready` | `media_provider_assets.state in (pending, uploading, processing)` | Wait for webhook or check `mix rindle.runtime_status --provider-stuck` |
| `:provider_webhook_invalid` | Signature mismatch OR replay window exceeded OR malformed body | Verify `RINDLE_MUX_WEBHOOK_SECRETS` env var; check clock skew |
| `:provider_sync_failed` | `media_provider_assets.state == :errored` | Inspect `last_sync_error`; consider re-ingest via `Rindle.regenerate_variants/2` (v1.5 surface) |
| `:provider_quota_exceeded` | Provider returned 429 / rate-limit error during ingest | Tune Oban `concurrency` for `:rindle_provider`; check provider plan |
| `:streaming_provider_requires_asset_struct` | Adopter passed a bare `key` to `streaming_url/3` while a streaming profile is configured | Pass the `MediaAsset` struct, or remove the `:streaming` profile key |

Plus the existing v1.4 `:streaming_not_configured` is reused (no semver shift).

### 8.3 Configuration burden (locked)

```elixir
config :rindle, Rindle.Streaming.Provider.Mux,
  token_id: System.get_env("RINDLE_MUX_TOKEN_ID"),
  token_secret: System.get_env("RINDLE_MUX_TOKEN_SECRET"),
  signing_key_id: System.get_env("RINDLE_MUX_SIGNING_KEY_ID"),
  signing_private_key: System.get_env("RINDLE_MUX_SIGNING_PRIVATE_KEY"),  # PEM
  webhook_secrets:
    "RINDLE_MUX_WEBHOOK_SECRETS"
    |> System.get_env("")
    |> String.split(",", trim: true),
  webhook_tolerance_seconds: 300,
  provider_polling_floor_seconds: 30,
  provider_stuck_threshold_seconds: 7200
```

Five env vars (`RINDLE_MUX_TOKEN_ID`, `RINDLE_MUX_TOKEN_SECRET`, `RINDLE_MUX_SIGNING_KEY_ID`, `RINDLE_MUX_SIGNING_PRIVATE_KEY`, `RINDLE_MUX_WEBHOOK_SECRETS`). Three optional tunables. `mix rindle.doctor` validates presence at boot.

### 8.4 Telemetry events (additive to frozen v1.4 contract)

```
[:rindle, :provider, :ingest, :start | :stop | :exception]
  measurements: %{system_time, duration?}
  metadata: %{profile, provider, asset_id, variant_name}

[:rindle, :provider, :sync, :resolved | :stuck]
  measurements: %{system_time}
  metadata: %{profile, provider, asset_id, provider_state, age_ms}

[:rindle, :provider, :webhook, :verified | :rejected | :secret_used]
  measurements: %{system_time}
  metadata: %{provider, event_type, secret_index, reason?}
```

The v1.4-frozen `[:rindle, :delivery, :streaming, :resolved]` event is **preserved unchanged**. When the provider path resolves, it fires with `kind: :hls` (new value) instead of `kind: :progressive`. This is additive on the metadata `kind` field only; consumers asserting `kind == :progressive` need to update — but that exact assertion is so narrow we accept it as the only deliberate v1.4 contract extension. Documented in the v1.6 upgrade guide.

### 8.5 What NOT to ship (DX guardrails)

- No `Rindle.Streaming.url_for_asset/2` shorthand — `streaming_url/3` is the single entrypoint.
- No automatic `:provider` activation based on `kind: :video`. Profile must opt in explicitly.
- No silent fallback from signed playback to public on token failure.
- No raw Mux client exposure (`Rindle.Streaming.Provider.Mux.client/0` is private). Adopters who need raw Mux call the SDK directly.

## 10. Lessons from peer libs — DO / AVOID per peer

### Mux Ruby SDK (`mux_ruby`)
- **DO**: copy the *server-push from URL* pattern (`Mux::Video::Assets.create(input: presigned_url)`). It's the cleanest ingest path; Rindle's existing signed `Rindle.Delivery.url/3` is a perfect input source.
- **AVOID**: using the SDK to parse webhook payloads. Mux explicitly documents that the generated REST SDK is unsuitable for webhook event modeling; their JS/Elixir SDKs ship dedicated `Webhooks` modules for this reason. We follow that — `Mux.Webhooks.verify_header/4` only.

### Mux Node SDK (`@mux/mux-node`)
- **DO**: copy the playback-ID + signing-key separation. Signing keys are managed out-of-band (one-time CLI / Dashboard creation). Rindle expects them via env, never auto-creates.
- **AVOID**: shipping a "create signing key" automation. Lifecycle of the signing private key is the adopter's secret-management problem, not the library's.

### Mux Elixir SDK (`muxinc/mux-elixir`)
- **DO**: rely on `Mux.Webhooks.verify_header/4` — it already implements the constant-time comparison + 300s tolerance + multi-`v1=` parsing correctly. Re-implementing this is gratuitous risk.
- **AVOID**: depending on `Mux.Token` defaults — its 7-day expiration default is way longer than Rindle's `signed_url_ttl_seconds` policy. Pass `expiration:` explicitly on every signed-playback call.

### Cloudflare Stream (contrast oracle)
- **DO**: stay with `:hls` as the only `kind`. Both Mux and Cloudflare Stream serve HLS; DASH support is post-v1.6.
- **AVOID**: leaking the `cors_origin` / `maxDurationSeconds` Mux/CF differences into the profile DSL. The translation lives in the adapter; the DSL stays provider-agnostic.

### Rails Active Storage + community Mux integrations
- **DO**: ship `media_provider_assets` as a first-class table (Rails Active Storage learned this lesson too late — analyzers were originally serialized into `metadata` JSONB and became unmanageable).
- **AVOID**: generating a webhook controller into the adopter app (Active Storage style). Library-owned Plug means library can patch signature bugs without adopter migrations.

### Shrine + Shrine derivatives plugin
- **DO**: copy the *atomic promotion* posture for provider state — re-fetch source asset before flipping `media_provider_assets.state` (Shrine's `AttachmentChanged` pattern). Rindle's AV-03-10 already does this for variants; extending to provider rows is direct.
- **AVOID**: a flat-blob `derivatives` JSONB field for provider info. Operator queries on `state == :errored` need typed columns, not JSONB extraction.

### Spatie Media Library + community Mux/Bunny integrations
- **DO**: ship a real reference adapter (Spatie's Mux community plugins drift because Spatie itself ships only docs).
- **AVOID**: punting webhook ingest as "adopter responsibility." Spatie does this; it's why every Spatie+Mux integration looks different. We own the Plug.

### Transloadit Robots / SDK (contrast only)
- **AVOID**: turning Rindle into a workflow engine. Transloadit's "robot graph" is the wrong shape for a lifecycle library — we ship one preset that points at one provider, not a DAG.

### Stripe webhooks (the gold standard)
- **DO**: copy posture verbatim — `t=…,v1=…` header format, 300s tolerance, multi-secret rotation with a 24h grace, constant-time compare. Mux already mirrors Stripe; our plug just confirms it.
- **AVOID**: skipping the multi-secret rotation. Webhook secret rotation is *mandatory* for ops hygiene; shipping single-secret-only kicks the can six months down the road.

### Vimeo OTT / Bitmovin (only if relevant)
- Not relevant for v1.6. Both target adaptive streaming + DRM, which are out-of-scope.

## 11. Pros / cons / risks specific to Rindle

**Pros (HIGH confidence):**

- Closes the v1.4 reservation cleanly. `streaming_url/3` already exists as a no-op delegate; v1.6 makes it real without breaking v1.4 adopters.
- One-provider scope keeps the abstraction honest. Mux is the right choice for v1 because their SDK already implements webhook verification + JWT signing — the two highest-risk pieces.
- Adopter-owned Repo + Oban contract extends naturally (new table, new workers, no architectural mutation).
- Strategic positioning: gives Rindle a credible video-at-scale answer without becoming a video platform.

**Cons (MEDIUM confidence):**

- Webhook ingest plumbing is high-fidelity work — three Oban workers, raw-body cache, multi-secret rotation, replay protection, idempotency. ~40% of the milestone effort lives in Phase 35.
- New Ecto table + new error vocabulary + new telemetry events is a real API surface. v1.4 already taught us to freeze before adoption pressure grows; v1.6 starts the freeze clock for *streaming* surfaces.
- CI burden grows: cassette-based fixtures + a real-network "soak" lane behind a `MUX_TOKEN_ID` secret. Mirrors the v1.2 protected-publish lane discipline.

**Risks (and how to mitigate):**

| Risk | Severity | Mitigation |
|---|---|---|
| Wrong abstraction locks early | HIGH | Single provider only; profile DSL refuses raw provider knobs; v1.7 second adapter is the abstraction test |
| Webhook drift / Mux event-shape changes | MEDIUM | Worker tolerates unknown event types (logs + persists `last_event_at`, no crash); Mux's webhook contract is stable |
| Token expiry surprises | MEDIUM | Reuse v1.4 `signed_url_ttl_seconds` policy; `mix rindle.doctor` smoke-pings signing key; per-content-type TTL guidance from v1.4 carries over |
| Provider-side stuck processing | MEDIUM | `MUX-07` polling fallback + `MUX-14` runtime_status filter |
| Multi-secret rotation gets skipped → outage during rotation | MEDIUM | Land it in Phase 35 from day one; not a v1.7 add |
| `media_provider_assets` state machine drift from `media_assets` FSM | LOW | Separate FSM; only crosspoint is the existing `MediaAsset` `:transcoding` → `:ready` flow remaining unchanged |
