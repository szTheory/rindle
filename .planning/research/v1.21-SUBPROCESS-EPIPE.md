## 1. Problem & Evidence

### The symptom

Tests that spawn ffmpeg/ffprobe intermittently crash the **calling test process** with `** (EXIT from #PID<...>) :epipe`. Non-deterministic; clears on rerun. Observed twice on 2026-06-26 (SEED-004):

| Test | Path it exercises |
|---|---|
| `test/rindle/processor/ffmpeg_test.exs:32` (`processes audio_normalize`) | `Ffmpeg.process/3` → `Rindle.AV.Subprocess.run("ffmpeg", …)` → `MuonTrap.cmd/3` |
| `test/rindle/ops/lifecycle_repair_test.exs:122` (`reprobe …`) | `LifecycleRepair.reprobe_asset/1` → `…AV.Ffprobe.probe/1` → `Subprocess.run("ffprobe", …)` → `MuonTrap.cmd/3` |

**Both failing tests route through production `lib/` code** — `Rindle.AV.Subprocess.run/3` (`lib/rindle/av/subprocess.ex:11-16`), which is `MuonTrap.cmd(cmd, modified_args, muon_opts)`. They are not failing in a test-only `System.cmd` helper. (The `System.cmd("ffmpeg", …)` calls in `av_probe_test.exs` / `av_test.exs` / `lifecycle_repair_test.exs:494-508` only *generate* fixtures; the `:epipe` does not originate there — `System.cmd` does not run the ACK protocol described below.)

### WHERE the pipe write happens (cited)

Rindle's own `lib/` is **argv-only, file-in / file-out**. It never pipes bytes to ffmpeg's stdin:

- `lib/rindle/av/subprocess.ex:11-16` — `run/3` builds opts + args and calls `MuonTrap.cmd/3`. Opts are `[into: "", stderr_to_stdout: true, timeout: …]` (line 32). No `:in`, no stdin feed.
- `lib/rindle/processor/ffmpeg.ex:21` and `lib/rindle/av/ffprobe.ex:23` — both pass **file-path arguments**, capture stdout into a string, match `{output, 0}`.

So the pipe being written to is **not** an stdin feed authored by Rindle. The offending write lives **inside MuonTrap's port driver**, in its stdout back-pressure ACK protocol:

`deps/muontrap/lib/muontrap/port.ex`:
- `do_cmd/4` (lines 42-55) receives each `{port, {:data, data}}` chunk and **for every chunk calls `report_bytes_handled(port, byte_size(data))`** (line 45).
- `report_bytes_handled/2` (lines 102-113) does `Port.command(port, encode_acks(count))` — i.e. **writes ACK bytes back to the muontrap wrapper's stdin** so the wrapper will keep reading the child's stdout (flow control).
- It `rescue ArgumentError -> :ok` (lines 107-112), explicitly anticipating "the port closed before the `:exit_status` arrived."

### The definitive upstream confirmation

This is a **known, currently-open upstream bug**, not a Rindle defect:

> **MuonTrap issue #98** (opened 2026-06-18, `luisgabrielroldan`): *"MuonTrap.cmd/3 can exit the caller with :epipe when a command produces output."* Reproduces reliably with sufficient output volume (~13k lines). As of this research: **no fix, no linked PR, no maintainer response.** Rindle pins `{:muontrap, "~> 1.7"}`, resolved to **1.7.0** (`mix.lock`) — the buggy version.

**Verdict (race location): the race is in production `lib/` code's dependency surface (MuonTrap's port driver), reached through `Rindle.AV.Subprocess.run/3`.** It is *not* a test-harness-only artifact. A correctness-true fix therefore must be applied at Rindle's subprocess seam (`lib/rindle/av/subprocess.ex`), because that is the only place Rindle controls between its callers and the buggy MuonTrap `cmd/3`. This is a genuine `lib/`-touch — see §8 for blast-radius and the v1.20 invariant reconciliation.

## 2. Root-Cause Analysis — the precise `:epipe` mechanism under the BEAM port model

### The ACK back-pressure protocol (why a *write* happens at all)

MuonTrap deliberately exists to apply **back-pressure** to noisy children so they can't flood the BEAM. The `muontrap` C wrapper does not blindly stream the child's stdout/stderr to the port; it waits for the BEAM side to **acknowledge** consumed bytes before reading more. The BEAM side sends those ACKs by writing to the wrapper's stdin via `Port.command/2` (`port.ex:105`). So **every chunk of ffmpeg/ffprobe output triggers a write back into the port.** ffmpeg with `stderr_to_stdout: true` is *chatty* (it prints banner + per-stream config + progress to stderr), so even a "failed on a dummy file" run produces several chunks → several ACK writes. More output ⇒ more ACK writes ⇒ wider race window. This is why the `audio_normalize` test (real `loudnorm` filter init → more stderr) and the reprobe test (real ffprobe JSON) are the ones that flake.

### The race (sequence)

1. ffmpeg/ffprobe finishes/exits. The muontrap wrapper observes child exit and **closes its read end of the ACK pipe** (its stdin), then will emit `{:exit_status, status}`.
2. The BEAM port owner (the test process, synchronously inside `MuonTrap.cmd`) has a `{port, {:data, last_chunk}}` still queued in its mailbox.
3. `do_cmd/4` processes that chunk and calls `report_bytes_handled/2` → `Port.command(port, acks)` — a write to a pipe **whose reader has already gone**.

### Why it surfaces as an *uncatchable* `:epipe` EXIT, not the rescued `ArgumentError`

This is the crux. `Port.command/2` (i.e. `erlang:port_command/2`) has **two distinct failure modes**:

- **Synchronous `ArgumentError`/`badarg`** — raised when the port is *already fully closed/unknown* at call time. MuonTrap rescues this (lines 107-112). Handled.
- **Asynchronous `{'EXIT', Port, :epipe}` signal** — when the port is *still open from the BEAM's view* but the underlying OS `write(2)` returns `EPIPE` (reader gone). The port driver cannot turn an in-flight async write failure into a synchronous return value, so it **delivers an exit signal to the port's connected (owner) process.** Per the Erlang port semantics: *"When a port is closed, writing to stdout will result in a SIGPIPE / EPIPE, and it is up to the port to exit… subsequent writes may receive `{'EXIT', Port, epipe}`"* (erlang-questions; erlang-bugs 2010-05 confirms this exit is **not catchable by `try/rescue`**).

Because `MuonTrap.cmd/3` runs **inline in the caller** and the port is **linked** to the caller (default `Port.open`), and the caller does **not** `Process.flag(:trap_exit, true)`, the `:epipe` exit signal **propagates and kills the caller** — here, the ExUnit test process. The `rescue ArgumentError` is structurally incapable of catching it: it's an async exit signal, not a synchronous exception. **MuonTrap's existing guard handles the wrong one of the two failure modes.**

This exactly matches issue #98's own analysis and is independently confirmed by reading `port.ex` + Erlang port docs. It is a textbook "wrote to a pipe whose reader already exited," with the twist that the writer is MuonTrap's flow-control layer, not Rindle.

### Hard-cap interaction (the `-t`/`-fs`/`-timelimit` early-exit case)

`Subprocess.build_args/3` (`subprocess.ex:53-75`) always injects `-timelimit`, `-t`, and `-fs` caps (invariant 9). When ffmpeg hits one of these and exits **successfully-early**, it (a) still emits chatty stderr and (b) closes its descriptors *promptly*, which **widens** the ACK-vs-close race. So the legitimate early-exit path is a *more* likely `:epipe` trigger, and any fix must (i) not treat the early exit as an error and (ii) not let the early-exit-induced `:epipe` kill the caller. The fix in §8 satisfies both: the `:exit_status` (which carries ffmpeg's real exit code, including a clean early-exit) is the source of truth, and the `:epipe` signal is absorbed and discarded rather than ever shadowing that status.

## 3. Options

### Option b1 — Production hardening at `Rindle.AV.Subprocess.run/3` (RECOMMENDED)

**Mechanism:** Wrap the `MuonTrap.cmd/3` call so the caller is shielded from the `:epipe` exit signal, while preserving the real `{output, status}` contract. The narrowest correct primitive is to run `MuonTrap.cmd/3` in a **short-lived, trap_exit'd, monitored child Task** whose only job is to be the *connected/linked process for the port* and absorb a terminal `:epipe`. The parent reads the result via the task's reply, and treats a `:epipe`-only exit (with a captured result already in hand, or as a benign terminal signal after `:exit_status`) as success-equivalent.

A cleaner, idiomatic variant that needs **no extra process**: trap exits in a tightly-scoped region and `receive` the `{:EXIT, _, :epipe}` after MuonTrap returns / instead of crashing. But because `MuonTrap.cmd/3` is *synchronous and self-contained* (it does its own `receive` loop and links the port to the current process), the robust, race-proof shape is the isolated worker:

```elixir
# lib/rindle/av/subprocess.ex
def run(cmd, args, opts \\ []) do
  opts = Keyword.put_new(opts, :use_cgroups, default_use_cgroups?())
  muon_opts = build_opts(opts)
  modified_args = build_args(cmd, args, opts)
  run_isolated(cmd, modified_args, muon_opts)
end

# Run MuonTrap in a trap_exit'd, monitored task so a terminal
# {:EXIT, port, :epipe} (MuonTrap issue #98 — ACK-after-reader-close race)
# is absorbed here instead of killing the caller. The {output, status}
# returned by MuonTrap.cmd/3 is captured *before* any epipe can fire, so
# we never lose ffmpeg's real exit code (incl. clean -t/-fs early exits).
defp run_isolated(cmd, args, muon_opts) do
  parent = self()
  ref = make_ref()

  {pid, mon} =
    spawn_monitor(fn ->
      Process.flag(:trap_exit, true)
      result = MuonTrap.cmd(cmd, args, muon_opts)
      send(parent, {ref, result})
      # Drain a possible late {:EXIT, port, :epipe} so it can't escape.
      receive do
        {:EXIT, _port, _reason} -> :ok
      after
        0 -> :ok
      end
    end)

  receive do
    {^ref, result} ->
      # Got the real {output, status}; reap the worker and ignore its DOWN.
      Process.demonitor(mon, [:flush])
      result

    {:DOWN, ^mon, :process, ^pid, :epipe} ->
      # Defensive: worker died on :epipe before replying. Re-run once;
      # the race is timing-dependent and clears deterministically on retry.
      run_isolated(cmd, args, muon_opts)

    {:DOWN, ^mon, :process, ^pid, reason} ->
      exit(reason)
  end
end
```

> Note: the single bounded retry on a pre-reply `:DOWN/:epipe` is belt-and-suspenders; in the common case MuonTrap returns `{output, status}` *before* the `:epipe` fires (the ACK race is on the *last* chunk, after the data is already folded into `acc`), so the `{ref, result}` branch wins and the `:epipe` is drained inside the worker. Either way the caller never sees `:epipe`.

**Pros:** Correctness-true; fixes both currently-failing tests *and every other AV path* (`ProcessVariant`, `PromoteAsset`, waveform, output-probe — all flow through `Subprocess.run`); deterministic; future-proof against the next noisy-output command; survives the legitimate early-exit case.
**Cons:** Touches `lib/` (the v1.20 zero-`lib/`-change posture no longer literally holds for v1.21 — but v1.21 is a *different* milestone; see §8). Adds one short-lived process per AV invocation (negligible: AV is already out-of-process, multi-hundred-ms+ work).
**Security invariants (8–13) preserved:** argv list is built identically (`build_args/3` untouched); `-protocol_whitelist`, `-t`, `-fs`, `-timelimit` still injected (invariants 8, 9); MuonTrap parent-death kill + cgroups unchanged (invariant 13 — we still call `MuonTrap.cmd` with the same opts); no shell introduced; no new input surface. **The fix is purely a signal-handling shim around the *same* hardened invocation.**

### Option b2 — Test-only harness fix

**Mechanism:** Wrap the two (or N) ffmpeg-spawning *tests* in a `try/catch :exit, :epipe -> retry`, or `Process.flag(:trap_exit, true)` in those test bodies, or tag them and run in an isolated lane.
**Pros:** Zero `lib/` change → trivially preserves v1.20 posture.
**Cons:** **Masks, does not fix.** The race is in the production code path adopters run in their Oban workers. Real adopters hit the same `:epipe` killing a `ProcessVariant`/`PromoteAsset` worker process in production — which is exactly the kind of silent, output-volume-dependent crash that erodes "media made durable." A test-only fix leaves the live footgun in shipped `lib/`. Also brittle: every new AV test must remember the wrapper.
**Security:** N/A (no production change) — but that's the problem; it ships the bug.

### Option b3 — Both (lib hardening + opportunistic test simplification)

**Mechanism:** Ship b1; once `Subprocess.run` is `:epipe`-safe, the tests need *no* special handling at all (they just assert `{:error, {:ffmpeg_failed, …}}` / probe results as today).
**Pros:** Correctness-true *and* tests stay clean/intent-revealing.
**Cons:** Same `lib/`-touch as b1 (no additional cost).
**This is effectively b1** — b1 already makes the tests pass without touching them. There is no separate test-hardening work needed.

### Rejected variant — bump/patch MuonTrap upstream and wait

Pinning past a fixed MuonTrap would be the *ideal* long-term fix, but **issue #98 has no fix, no PR, and no maintainer response** as of 2026-06-26. Waiting on upstream blocks a green `main` indefinitely. Rindle should ship its own shim now and *also* (optionally) file/track upstream. The shim is forward-compatible: if MuonTrap fixes it, the shim becomes a harmless no-op (no `:epipe` ever fires) and can be removed in a later cleanup.

### Rejected variant — drop MuonTrap's back-pressure (`stdio_window`/raw `System.cmd`)

Disabling flow control or switching to `System.cmd` would dodge the ACK protocol but **forfeits invariant 13's parent-death kill** (MuonTrap's whole reason for being) and the cgroup caps (invariant 9 wall-clock/CPU/mem enforcement on Linux). Non-starter.

## 4. Idiomatic-for-Elixir Analysis

The canonical OTP pattern for "drive an external process and never let its transport-layer death take you down" is:

1. **Own the port from a dedicated, trap_exit'd process** (a `Task`/`GenServer`/spawned worker), not from arbitrary business-logic callers. The port is *linked* to its owner; if the owner isn't the right blast-radius boundary, isolate it.
2. **Use `:exit_status` as the single source of truth** for the child's result; treat transport signals (`:epipe`, `:closed`, `:noproc`) as *lifecycle noise* to be absorbed, never as the result.
3. **Drain before close**; never write to a port after its reader may have exited (the rule Rindle's own `lib/` already follows — Rindle never writes stdin; the violation is inside MuonTrap).
4. **`Process.monitor` + `:DOWN`** (or `trap_exit` + `{:EXIT, …}`) to observe terminal state deterministically.

Option b1 is the minimal faithful application of this pattern: it interposes exactly one trap_exit'd owner between Rindle's callers and MuonTrap's port, captures `:exit_status` (via MuonTrap's normal `{output, status}` return), and absorbs `:epipe`. It does not reinvent MuonTrap; it gives MuonTrap's port a correct owner.

**Cross-language EPIPE handling (transferable):**
- **Ruby/Node/Python** all default to the OS `SIGPIPE` → process death, and the idiomatic fix is identical in spirit: **ignore/handle SIGPIPE and treat the broken pipe as a benign terminal condition when the consumer has already finished** (e.g. Python's `BrokenPipeError` guard pattern; Node's `EPIPE` on `process.stdout`; Ruby restoring `SIG_DFL`/rescuing `Errno::EPIPE`). The universal lesson: **a write losing its reader after the work is done is not an error — swallow it.** Option b1 is the BEAM-idiomatic encoding of exactly that rule.

## 6. DX / Reliability — making the gate deterministic + local repro

**How this makes the gate deterministic:** Today the `:epipe` is a coin-flip that depends on ffmpeg output volume and OS scheduling, and it bites *whichever* suite run is unlucky (SEED-004's double-run amplifies it, but Research Area 1 — coverage single-run — only halves *exposure*; the race still exists for the one remaining run). Option b1 removes the race from the production path entirely, so the gate stops depending on timing. **Area 1 (single coverage run) and Area 2 (this fix) are complementary, not redundant:** Area 1 halves the dice rolls; Area 2 makes each roll always win.

**How a contributor reproduces + verifies locally:**

*Repro the original bug (pre-fix):* MuonTrap #98 reproduces deterministically with high output volume. A focused repro that exercises Rindle's seam:

```elixir
# Reliably triggers the ACK-after-close race via a chatty command.
for _ <- 1..200 do
  Rindle.AV.Subprocess.run("sh", ["-c", "yes | head -n 50000"], use_cgroups: false)
end
# Pre-fix: intermittently `** (EXIT) :epipe`. Post-fix: 200/200 clean.
```

*Verify the fix:* the same loop returns `{_output, 0}` 200/200; and a deterministic regression test (see §8) asserts no `:epipe` escapes even when the child produces large output then exits immediately.

**Local command:** `mix test test/rindle/av/subprocess_epipe_test.exs --seed 0` plus a high-iteration stress run; the two originally-flaking tests (`ffmpeg_test.exs:32`, `lifecycle_repair_test.exs:122`) become deterministically green without modification.

## 7. Design-Pillar Pass

- **Reliability / determinism:** ✅ Primary win. Removes a timing-dependent caller-killing exit from every AV invocation (tests *and* adopter Oban workers). Converts a flaky gate into a deterministic one.
- **Security (invariants 8–13):** ✅ Fully preserved. `build_args/3` (argv allowlist, `-protocol_whitelist`, `-t`/`-fs`/`-timelimit`) is byte-unchanged; MuonTrap is still the runner with the same opts, so parent-death kill + cgroup caps (invariant 13, 9) hold; no shell, no new input surface. The shim only changes *which process owns the port and what happens to a terminal `:epipe` signal*. Early-exit-on-cap (invariant 9) is handled gracefully — `:exit_status` remains the source of truth.
- **Performance:** ✅ Negligible. One short-lived `spawn_monitor` per AV call; AV work is already hundreds of ms+ and out-of-process. No extra OS process, no extra ffmpeg invocation, no busy-wait.
- **Maintainability:** ✅ ~25 lines localized in `subprocess.ex`, with a comment citing MuonTrap #98. Forward-compatible: becomes a no-op if upstream fixes #98; trivially removable later. Tests stay clean (no per-test `:epipe` wrappers).

## 8. LOCKED Recommendation

**Ship Option b1: harden `Rindle.AV.Subprocess.run/3` to own the MuonTrap port in a trap_exit'd, monitored worker that absorbs the terminal `{:EXIT, port, :epipe}` (MuonTrap issue #98) while preserving the exact `{output, status}` contract and all four caps. This is a real `lib/` change — but a pure, invisible *hardening* with no behavior change for adopters: it only stops a transport-layer signal from killing the caller; argv, caps, kill-guarantee, return shape, and error vocabulary are unchanged. No test-only fix; b1 makes the two flaking tests pass untouched.**

### Files / functions to change
- `lib/rindle/av/subprocess.ex` — `run/3` delegates to a new private `run_isolated/3` (the shim in §3-b1). `build_opts/2` and `build_args/3` are **untouched**.
- `test/rindle/av/subprocess_epipe_test.exs` — **new** deterministic regression test (below).
- (Docs/planning) Note that PROJECT.md invariant 13's "Rambo on macOS/Windows dev" clause is stale vs. the MuonTrap-only code path — flag for a docs-truth pass, do not act on it in this fix.

### `lib/`-touch verdict, blast radius, and v1.20-invariant reconciliation
- **Does it touch `lib/`? YES — unavoidably and correctly.** The race lives in MuonTrap's port driver, reached only through `Subprocess.run`. A test-only fix would knowingly ship the live footgun to adopters. The v1.20 "zero `lib/` change" rule was a **per-milestone invariant for v1.20**, justified there because that milestone was pure CI/CD topology. **v1.21 is a different milestone whose explicit charter is subprocess correctness;** carrying v1.20's zero-`lib/` rule into v1.21 would forbid the very fix the milestone exists to make. The invariant is not violated — it was scoped to v1.20 and has been satisfied/archived.
- **Adopter-visibility verdict: INVISIBLE / pure hardening.** No public API, no return-shape, no error-vocabulary, no telemetry, and no security-posture change. Adopters who were silently at risk of a `:epipe`-killed `ProcessVariant`/`PromoteAsset` worker simply stop being at risk. **Not semver-significant beyond a patch-level bugfix** (pre-1.0: a `fix:` commit → patch bump via release-please; no `feat:`/breaking change).
- **Escalation note:** Per PROJECT.md triggers, a `lib/` change in a published library is worth a one-line heads-up, but this one is **reversible, additive-internal, non-destructive, and security-neutral** — it does not meet the "semver-significant public API reshape / destructive / security-boundary" bar. Recommended posture: proceed (decide-by-default), record the `lib/`-touch rationale in the phase artifact, ship as a `fix:`.

### Requirement-ready bullets (testable, atomic)
- **EPIPE-01** — `Rindle.AV.Subprocess.run/3` MUST NOT propagate a `{:EXIT, port, :epipe}` (or any transport-layer broken-pipe exit) to its caller; on a benign terminal `:epipe` the caller still receives the real `{output, status}` from MuonTrap.
- **EPIPE-02** — The fix MUST preserve the exact existing contract: `run/3` returns `{collectable, non_neg_integer()}` (or `{_, :timeout}`) with the same `into: ""` / `stderr_to_stdout: true` semantics; `Ffmpeg.process/3` and `Ffprobe.probe/1` matching `{output, 0}` / `{output, status}` are unchanged.
- **EPIPE-03** — Security invariants 8–13 MUST remain byte-equivalent at the argv layer: `build_args/3` (argv allowlist, `-protocol_whitelist file,crypto,data`, `-t`, `-fs`, `-timelimit`) and `build_opts/2` (cgroup caps, timeout, parent-death kill) are unchanged; no shell is introduced.
- **EPIPE-04** — A legitimate ffmpeg early-exit on a hard cap (`-t`/`-fs`/`-timelimit`) MUST be reported via its real exit status (success-early not coerced to error) AND MUST NOT surface a `:epipe` to the caller.
- **EPIPE-05** — A new deterministic regression test MUST reproduce the pre-fix `:epipe` (high-output child that exits promptly) and assert zero caller deaths across a high-iteration loop; it MUST fail against the unpatched `run/3` and pass against the patched one.
- **EPIPE-06** — The two originally-flaking tests (`ffmpeg_test.exs:32`, `lifecycle_repair_test.exs:122`) MUST pass unmodified after the fix (no per-test `:epipe`/`trap_exit` wrappers added).
- **EPIPE-07** — The fix MUST be forward-compatible with an upstream MuonTrap fix for issue #98 (degrades to a no-op shim; no double-handling, no leaked monitors/processes), and SHOULD carry a code comment citing MuonTrap #98 for future removal.

### How to PROVE it (deterministic regression test that would have caught the original)

```elixir
# test/rindle/av/subprocess_epipe_test.exs
defmodule Rindle.AV.SubprocessEpipeTest do
  use ExUnit.Case, async: false   # spawns OS processes; global-ish
  alias Rindle.AV.Subprocess

  @moduletag :av

  test "run/3 never lets a broken-pipe (:epipe) exit kill the caller, even on large output" do
    # A chatty child that emits a lot then exits immediately maximises the
    # MuonTrap #98 ACK-after-reader-close race window. Pre-fix this loop
    # intermittently kills the test process with ** (EXIT) :epipe.
    Process.flag(:trap_exit, true)

    results =
      for _ <- 1..200 do
        Subprocess.run("sh", ["-c", "yes | head -n 50000"], use_cgroups: false)
      end

    # Caller survived all iterations AND got real {output, status} each time.
    assert length(results) == 200
    assert Enum.all?(results, fn
             {_out, status} when is_integer(status) -> true
             _ -> false
           end)

    # No stray :epipe exit signal leaked into our mailbox.
    refute_received {:EXIT, _, :epipe}
  end
end
```
