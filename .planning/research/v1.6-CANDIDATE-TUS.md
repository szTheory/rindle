# v1.6 Candidate: tus Resumable Upload Protocol — Locked Recommendation

**Date:** 2026-05-06
**Confidence:** MEDIUM-HIGH. Protocol surface, idiomatic Plug shape, peer-library
lessons, tusd warnings, and Rindle's existing seams are all directly verifiable.
The single biggest unknown is real adopter demand vs. GCS/provider boundary
work; that is a synthesis-step decision, not this document's.
**Recommendation score:** 6/10 for "v1.6 now" — up from 4/10 in the
2026-05-05 memo. Three things changed: (1) **Tussle v0.3.1 shipped 2026-03-24**
with an active maintainer, real CloudFlare HEAD-to-GET fix, and configurable
`read_body` options — this is no longer an ecosystem-thin candidate. (2) The
v1.5 ship gave Rindle exactly the tagged-error and capability vocabulary tus
needs to land additively. (3) v1.4 AV's "smartphone-source large video
upload" wedge is exactly the case multipart cannot solve and tus does.

The 6/10 (not 8/10) reflects that this is still the most boundary-expanding
of the three v1.6 candidates: Rindle becomes a server that runs the upload
hot path, not just a lifecycle library. Pursue if synthesis confirms adopter
demand for resumable mobile-AV ingest is the top unblocked pain.

---

## 1. Locked scope (in-process Plug vs external tusd — pick one)

**Lock: in-process mountable Plug, vendored on top of `tussle ~> 0.3` as a
hard runtime dependency, with a clean Rindle-owned facade.**

Not external tusd. Not a hand-rolled Plug. Not "support both modes."

Rationale, in order of weight:

1. **Adopter ownership.** v1.1 made adopter-owned `Repo` the contract.
   External tusd forces adopters to operate a Go process, configure its
   metadata store separately from `MyApp.Repo`, and reconcile two sources of
   truth at the post-finish hook. That breaks the v1.1 ownership decision.
2. **Tagged-error vocabulary.** v1.5 locked Rindle's error vocabulary. An
   in-process Plug can return `{:error, {:upload_unsupported,
   :resumable_upload}}` and friends directly. An external tusd hook returns
   HTTP status codes Rindle would have to translate.
3. **Capability honesty.** v1.4/v1.5 promised that capabilities Rindle
   advertises are real. An external-tusd posture means Rindle advertises
   `:resumable_upload` only when the adopter has correctly stood up tusd.
   That is a deployment-correctness claim Rindle cannot verify at boot.
4. **Tussle's actual state.** Verified live 2026-05-06: tussle v0.3.1
   released 2026-03-24, jvantuyl as active maintainer, supports core +
   creation + termination + expiration extensions, pluggable cache and
   storage, mountable Plug routes via `Tussle.Routes.add_tus_routes/1`. It
   is the one library in the Elixir tus ecosystem that meets the bar.
   Exotus (v0.1.0, 2020) and gottfrois/tus (~8 stars, dormant) do not.
5. **Failure mode containment.** A bug in Rindle's tus path crashes the
   adopter's BEAM the same way any other bug does. A bug in external tusd
   crashes a separate Go service the adopter's existing supervision and
   observability stack does not own. The Rindle posture is "your runtime
   owns Rindle"; an external server breaks that.

**What we explicitly do not ship:**

- A bundled tusd binary or container image. Rindle is a Hex library.
- Concatenation, checksum, or checksum-trailer extensions in v1.6.
- IETF draft-ietf-httpbis-resumable-upload (tus 2.0 / RUFH) support.
  Draft-11 as of 2026-03 is still a draft; tus-js-client v4.3+ supports
  both `tus-v1` and `ietf-draft-05`, so adopters using the canonical client
  get a forward path naturally without Rindle leading.
- A Rindle-owned tus client. `tus-js-client` (v4.3.x) and Uppy `@uppy/tus`
  are the documented client paths.

**What is in scope for v1.6:**

- Tus core 1.0.0 protocol on a mountable Rindle Plug.
- Creation, expiration, and termination extensions (the same three Tussle
  ships and tus-ruby-server treats as table stakes).
- A broker-owned tus session row converging into the existing
  `verify_completion` lane.
- Two storage-backing modes locked: local-tmp accumulation, or S3 multipart
  per PATCH chunk for the S3 adapter (see §7).
- Oban expiry sweep (reuse the v1.1 `AbortIncompleteUploads` worker shape).
- `:resumable_upload` capability un-reserved on `Rindle.Storage.Local` and
  `Rindle.Storage.S3` (when MinIO-proven). Capability stays unsupported on
  any adapter that has not been proven against MinIO + tus-js-client.

## 2. Phase breakdown (concrete, with names + REQ-IDs + success criteria)

Five phases. All phase numbers continue from v1.5 (which closed at Phase 32).

### Phase 33 — TUS Foundations (capability, session schema, dependency seam)

**REQ-IDs:** TUS-01, TUS-02, TUS-03, RESUMABLE-01

**Goal:** Lock Rindle's tus seam in code without serving any HTTP yet:
session schema, capability un-reservation, dependency pinning, and the
broker entrypoints that other phases will wire to.

- TUS-01: Add `tussle ~> 0.3.1` as a runtime dep behind a documented
  capability gate.
- TUS-02: Migration adds `tus_*` columns to `media_upload_sessions`
  (additive, see §6) and a new `upload_strategy: "tus"` value.
- TUS-03: `Rindle.Upload.Broker.initiate_resumable_upload/2` and
  `cancel_resumable_upload/1` shipped as broker entrypoints; both fail
  with `{:error, {:upload_unsupported, :resumable_upload}}` on adapters
  that do not advertise `:resumable_upload`.
- RESUMABLE-01: `Rindle.Storage.Capabilities` lifts the "reserved" status
  from `:resumable_upload` and `:resumable_upload_session`; capability
  guide updated; release notes warn that adapters silently advertising
  these atoms in adopter-private forks must now back them.

**Success:** mix test green; `Rindle.Storage.Local` and `Rindle.Storage.S3`
explicitly do **not** advertise `:resumable_upload` yet; broker returns
the tagged unsupported tuple; no HTTP route lands in this phase.

### Phase 34 — TUS Plug & Endpoint Wiring

**REQ-IDs:** TUS-04, TUS-05, TUS-06, TUS-07

**Goal:** Mount the protocol edge, with adopter-controlled path and
configurable body limits. Tussle handles HEAD/PATCH/OPTIONS/POST/DELETE
mechanics; Rindle owns auth, session resolution, and post-finish handoff.

- TUS-04: `Rindle.Upload.TusPlug` macro mounts tus routes under an
  adopter-chosen scope. Locks the public router macro.
- TUS-05: `Rindle.Upload.TusController` (or equivalent — Tussle expects
  a controller module via `add_tus_routes/1`) implements pre-create and
  post-finish callbacks that bind tus uploads to broker sessions.
- TUS-06: Endpoint configuration is documented exactly (Plug.Parsers
  `:pass`, body length, read length, read timeout, allowed content type
  `application/offset+octet-stream`, parser pass-through).
- TUS-07: `Rindle.Storage.Local` advertises `:resumable_upload` and
  `:resumable_upload_session`; integration test uploads a >32MiB file via
  tus-js-client across simulated PATCH retries against the local adapter.

**Success:** the existing mix test suite runs unmodified, an opt-in
integration suite proves tus-js-client → Rindle tus Plug → local storage
→ verify/promote → ready asset, end-to-end.

### Phase 35 — S3 Storage Backing & MinIO Proof

**REQ-IDs:** TUS-08, TUS-09, TUS-10, TUS-11

**Goal:** Connect the tus Plug to the S3 adapter so chunks land directly
in S3 (or MinIO) using S3 multipart UploadPart per PATCH, not local-tmp.
This is the only way to keep BEAM RAM and BEAM disk constant for any
upload size.

- TUS-08: `Rindle.Storage.S3` advertises `:resumable_upload` only after
  the MinIO proof passes; the tussle storage adapter for S3 lives behind
  `Rindle.Upload.Tus.Storage.S3`.
- TUS-09: Each tus PATCH ≥5MiB triggers `UploadPart`; PATCH <5MiB lands
  in a temporary S3 staging key (or buffered locally, configurably) until
  it can be flushed as a part — see §7.
- TUS-10: Completion calls `complete_multipart_upload/4` and converges
  into `Rindle.Upload.Broker.verify_completion/2` — the same lane as
  presigned PUT and S3 multipart.
- TUS-11: MinIO-backed integration: 1GiB tus upload, simulated mid-flight
  network drop with tus-js-client retry, simulated abandonment + Oban
  reaper proves both partial parts and tus session row are cleaned.

**Success:** v1.1's MinIO CI lane gains a tus-js-client smoke and a
3-PATCH-retry resume scenario. S3 cleanup is provably idempotent.

### Phase 36 — Oban Expiry, Termination, and Auth

**REQ-IDs:** TUS-12, TUS-13, TUS-14, TUS-15

**Goal:** Make the protocol safe in production: server-owned URL secrecy,
explicit expiry, termination semantics, and Oban-driven cleanup of
abandoned uploads — including S3 multipart parts.

- TUS-12: Tus session URLs are signed with an adopter-configurable secret
  (config `:rindle, :tus_url_secret`) and bound to the broker session ID.
  Resume requires HMAC-validated path; raw guessing the URL is
  insufficient. Closes the tusd-documented "no built-in same-user
  resume" footgun (§9).
- TUS-13: Expiration extension: `Upload-Expires` header populated from
  session `expires_at`; expired sessions return `410 Gone`; resumes
  beyond expiry return `410 Gone`, never 200.
- TUS-14: Termination extension: `DELETE` on a tus session URL transitions
  the broker session to `aborted`, aborts S3 multipart (idempotent), and
  emits the existing abort telemetry. No new telemetry vocabulary.
- TUS-15: `Rindle.Workers.AbortIncompleteUploads` extended to terminate
  abandoned tus uploads (not just multipart). Reuses the v1.1 cron shape
  exactly. Documents the recommended cron cadence.

**Success:** A signed-URL contract test proves tampered tus URLs return
401/404 (never 200). 410 returns proven by integration test. MinIO proof
shows abandoned tus uploads reaped within one cron cycle.

### Phase 37 — DX, Docs, Telemetry, and CI Proof Lane

**REQ-IDs:** TUS-16, TUS-17, TUS-18, TUS-19

**Goal:** Adopters can ship tus from copy-paste docs, the `mix
rindle.doctor` lane reports honestly, telemetry is locked, and a generated
package-consumer app proves the smartphone-source-AV-via-tus story end-to-end.

- TUS-16: Public guide `guides/resumable_uploads.md` covers endpoint
  config, router macro, Phoenix `endpoint.ex` settings,
  `Plug.Parsers` `:pass` guidance, recommended client config
  (`@uppy/tus` and `tus-js-client`), and the security checklist.
- TUS-17: `mix rindle.doctor` learns a `:resumable_upload` profile
  check: if a profile names an adapter without `:resumable_upload` but
  the adopter mounted `Rindle.Upload.TusPlug`, doctor reports a
  configuration mismatch.
- TUS-18: Telemetry contract locked (§10). Events:
  `[:rindle, :upload, :resumable, :start | :patch | :stop | :abort |
  :expire]`.
- TUS-19: Generated-app proof lane: a `mix phx.new` adopter installs
  Rindle, mounts the tus Plug, and a CI script (using tus-js-client in
  Node 20 against MinIO) uploads a 200MB sample MP4, simulates one
  network drop, resumes, and asserts the resulting `MediaAsset` is in
  `ready` state with the expected `byte_size` and `content_type`.

**Success:** adopters can copy-paste the guide and ship; `mix
rindle.doctor` is honest; CI proves the killer mobile-AV ingest path.

## 3. Idiomatic Elixir/Phoenix/Ecto/Plug fit (with pinned hex versions)

Verified live on 2026-05-06.

| Library | Version | Confidence | Notes |
|---------|---------|------------|-------|
| `tussle` | `~> 0.3.1` | HIGH (Hex 2026-03-24) | jvantuyl-maintained fork of `tus`, mountable Plug routes, S3 storage via `tus_storage_s3`, Memory and Redis caches; `Tussle.Routes.add_tus_routes/1` is the supported macro; v0.3.1 added configurable `read_body` options and the CloudFlare HEAD-to-GET fix |
| `tus_storage_s3` | latest from `tussle` org | MEDIUM | Optional companion package; vendor-pin only after Phase 35 confirms the integration |
| `tus_cache_redis` | n/a in v1.6 | MEDIUM | Skip; default `Tussle.Cache.Memory` is wrong for production, but Rindle replaces the cache with an Ecto-backed cache (see §6) |
| `plug` | `~> 1.16` (existing) | HIGH | `Plug.Parsers` `:pass: ["application/offset+octet-stream"]` is the documented seam |
| `phoenix` | `~> 1.7` (existing) | HIGH | No new dependency needed |
| `oban` | existing | HIGH | Reuse `:rindle_maintenance` queue and the v1.1 cron pattern |
| `tus-js-client` (client side) | `4.3.x` | HIGH | Defaults: `chunkSize: Infinity`, `retryDelays: [0,1000,3000,5000]`, `parallelUploads: 1`; supports `protocol: 'tus-v1' | 'ietf-draft-05'` |
| `@uppy/tus` (client side, optional) | latest | MEDIUM | Documented as canonical Uppy client path |

**Body-streaming lock-in:** Phoenix Endpoint must be configured with
`http: [protocol_options: [max_request_line_length: ..., max_header_value_length: ...]]`
and Plug.Parsers must `:pass` the tus content-type. PATCH bodies are read
chunk-by-chunk by Tussle via `Plug.Conn.read_body/2` with explicit
`:length`, `:read_length`, and `:read_timeout` (Plug docs verified
2026-05-06: defaults are 8MB / 1MB / 15s respectively). Rindle locks
`:read_length: 1_048_576` and `:length: 16_777_216` per call as defaults
in TUS-06; this caps per-call BEAM heap pressure at 16 MiB regardless of
client `chunkSize`.

**No NIF, no port, no Go binary** — pure BEAM. Matches the v1.4 decision
that AV ships via FFmpeg subprocess (not NIFs) and the project constraint
"Tech stack: Elixir/Phoenix/Ecto only in core."

## 4. Public API + mountable Plug shape (locked function arities, router macro, endpoint config)

### Broker entrypoints (locked)

```elixir
@spec initiate_resumable_upload(profile :: module(), opts :: keyword()) ::
        {:ok, %{
          session: MediaUploadSession.t(),
          resumable: %{
            location: String.t(),     # the tus upload URL adopters return to clients
            expires_at: DateTime.t(),
            max_size: pos_integer()
          }
        }}
        | {:error, {:upload_unsupported, :resumable_upload}}
        | {:error, term()}
def initiate_resumable_upload(profile_module, opts \\ [])

@spec cancel_resumable_upload(session_id :: binary()) ::
        {:ok, MediaUploadSession.t()} | {:error, term()}
def cancel_resumable_upload(session_id)
```

`HEAD`/`PATCH` operations are NOT exposed as Elixir functions. They are
HTTP-only, owned by the Plug. Programmatic access to session state goes
through the existing `repo.get(MediaUploadSession, id)` path; there is no
need for a `head_resumable_upload/1` Elixir API and inventing one creates
a parallel surface for no gain.

`complete_resumable_upload/2` is also NOT exposed. Tus completion is
triggered automatically when the final PATCH lands; Rindle's post-finish
callback enqueues `Rindle.Upload.Broker.verify_completion/2` into Oban —
the same lane multipart uses. There is one completion vocabulary, not two.

### Mountable Plug (locked)

```elixir
defmodule MyAppWeb.Router do
  use MyAppWeb, :router
  import Rindle.Upload.TusPlug, only: [tus_routes: 1]

  scope "/uploads/tus", MyAppWeb do
    pipe_through [:api, :require_authenticated_user]
    tus_routes Rindle.Upload.TusController
  end
end
```

`tus_routes/1` is a thin wrapper over `Tussle.Routes.add_tus_routes/1`
that wires the Tussle controller to Rindle's pre-create / post-finish
callbacks. Adopters mount it under whatever scope and pipeline they want;
the auth pipeline is theirs — Rindle does not invent an auth contract.

### Phoenix endpoint config (locked, MUST set)

```elixir
plug Plug.Parsers,
  parsers: [:urlencoded, :multipart, :json],
  pass: ["application/offset+octet-stream"],   # MUST
  json_decoder: Phoenix.json_library(),
  length: 20_000_000                             # MUST raise from Plug default if larger uploads expected at the parser layer; Tussle reads the body itself with read_body/2
```

Adopters who mount under `/api` and use Phoenix's `:api` pipeline must
ensure Plug.Parsers does not reject the tus content-type. This is
non-negotiable; without `:pass`, every PATCH gets rejected at the parser.

### Capability gate

```elixir
:ok = Rindle.Storage.Capabilities.require_upload(adapter, :resumable_upload)
```

No silent fallback to presigned PUT or multipart. Adapters that do not
advertise `:resumable_upload` return `{:error, {:upload_unsupported,
:resumable_upload}}`.

### Locked failure-mode atoms (additive to v1.5 vocabulary)

| Atom | Meaning | HTTP status |
|------|---------|-------------|
| `{:upload_unsupported, :resumable_upload}` | adapter does not support resumable | 501 (or returned from Elixir API only; Plug never sees this) |
| `:tus_session_not_found` | tus URL points at a session not in DB | 404 |
| `:tus_session_expired` | session past `expires_at` | 410 |
| `:tus_offset_conflict` | PATCH `Upload-Offset` ≠ server offset | 409 |
| `:tus_size_exceeded` | upload exceeds `Tus-Max-Size` or session `max_size` | 413 |
| `:tus_unsupported_extension` | client requested an extension Rindle does not implement | 501 |
| `:tus_storage_unavailable` | underlying storage adapter call failed | 500 |
| `:tus_url_signature_invalid` | tampered or stale signed URL | 401 |

These compose with v1.5's `Rindle.Error` struct. Bang variants (`!`) on
the broker raise `Rindle.Error{action: :resumable_upload, reason: ...}`.

## 5. Capability vocabulary changes (reserved → real)

v1.1 reserved `:resumable_upload` and `:resumable_upload_session`. v1.6
makes them real.

| Adapter | Before v1.6 | After v1.6 |
|---------|-------------|------------|
| `Rindle.Storage.Local` | `[:local, :presigned_put]` | `[:local, :presigned_put, :resumable_upload, :resumable_upload_session]` |
| `Rindle.Storage.S3` | `[:presigned_put, :head, :signed_url, :multipart_upload]` | `[:presigned_put, :head, :signed_url, :multipart_upload, :resumable_upload, :resumable_upload_session]` once MinIO proof passes |
| Cloudflare R2 (via S3 adapter) | identical to S3 | identical to S3, with **explicit guide note that R2 has its own tus surface** and Rindle's tus uses the S3 multipart path; adopters who want R2-native tus point at `https://<account>.r2.cloudflarestorage.com` and accept the R2 chunk-size minimums |
| Future GCS | unchanged (deferred to GCS milestone) | unchanged |

`:resumable_upload` advertises that the adapter can back tus PATCH chunks.
`:resumable_upload_session` advertises that the adapter exposes durable
in-flight session state separate from the final object.

R2 honesty rule (preserved from v1.5): if a vendor's S3-compat surface
does not honor `UploadPart` semantics within the 5MiB minimum constraint,
the guide says so. Rindle does not silently downgrade.

## 6. Migration / Ecto shape (additive — broker tus session)

Reuse `media_upload_sessions`. Do not invent a parallel table. Tus
sessions are upload sessions; the existing FSM already has the right
states (`initialized | signed | uploading | uploaded | verifying |
completed | aborted | expired | failed`).

### Additive columns (TUS-02)

```elixir
defmodule Rindle.Repo.Migrations.AddTusFieldsToMediaUploadSessions do
  use Ecto.Migration

  def change do
    alter table(:media_upload_sessions) do
      add :tus_offset, :bigint, default: 0
      add :tus_size, :bigint                          # the negotiated total upload size
      add :tus_metadata, :map, default: %{}           # tus Upload-Metadata (filename, type, ...)
      add :tus_url_signature, :string                 # HMAC binding the URL to this row
      add :tus_protocol_version, :string, default: "1.0.0"
    end

    create index(:media_upload_sessions, [:upload_strategy, :state, :expires_at])
  end
end
```

`upload_strategy` extends from `"presigned_put" | "multipart"` to add
`"tus"`. The FSM is unchanged. `multipart_upload_id` and `multipart_parts`
are reused: when the S3 backing path is active, the underlying S3 multipart
upload's ID lives in `multipart_upload_id` exactly as it does for direct
multipart sessions today. This is the convergence point that lets the
v1.1 `AbortIncompleteUploads` worker terminate abandoned tus uploads
without new code paths.

### Tussle's cache, replaced

Tussle ships `Tussle.Cache.Memory` and `tus_cache_redis`. Both are wrong
for Rindle: the source of truth must be the adopter Repo. Rindle ships
`Rindle.Upload.Tus.Cache.Ecto` implementing the `Tussle.Cache` behaviour
on top of `media_upload_sessions`. Every PATCH writes its new offset to
the row inside a transaction (same Repo, no in-memory drift). This is
non-negotiable; without it, two BEAM nodes will return inconsistent
offsets on `HEAD` and break tus-js-client retries.

## 7. Storage backing strategy (chunk to tmp+PUT vs S3 UploadPart per chunk — lock one)

Two locked modes, one per adapter family:

### `Rindle.Storage.Local`: accumulate to tmp, no `PUT` (just rename)

PATCH chunks are appended directly to a temp file under
`Rindle.Config.tmp_root() <> "/tus/<session_id>.part"`. On post-finish,
the file is renamed (cheap atomic on same filesystem) into the final
storage key path. No PUT, no copy. Disk usage is bounded by
`session.tus_size`.

### `Rindle.Storage.S3` (and S3-compatible incl. MinIO, Cloudflare R2): S3 multipart, one UploadPart per PATCH ≥ 5MiB

This matches what tusd does internally for its S3 backend (verified
2026-05-06: tusd buffers small chunks on disk to satisfy the 5MiB
minimum, then flushes them as a part).

Concrete rule:

- `initiate_resumable_upload/2` calls `S3.initiate_multipart_upload/3`
  immediately. The S3 `upload_id` lands in
  `media_upload_sessions.multipart_upload_id`. **Tus session and S3
  multipart session are 1:1.**
- Each PATCH whose body length ≥ 5MiB streams directly into one S3
  `UploadPart`. The returned ETag is appended to
  `media_upload_sessions.multipart_parts`. The session's `tus_offset`
  advances by the PATCH length.
- PATCHes < 5MiB (which can only legally be the final PATCH per S3 rules)
  buffer to a temp file under the BEAM node's local disk; on the next
  PATCH (or completion), the buffer is flushed as a single part. This is
  the only place local disk gets touched on the S3 path.
- Completion calls `S3.complete_multipart_upload/4` and converges to
  `verify_completion/2`. `verify_completion/2` already calls
  `adapter.head/2`, which already validates final byte size and
  `content_type` against profile constraints. **This is the locked
  convergence point that preserves the v1.1 verification lane unchanged.**

This is **not** "buffer the whole upload locally then PUT to S3." That
mode is rejected: 1GB mobile videos would consume 1GB of BEAM disk per
in-flight upload, multiplied by every concurrent uploader. Unbounded.

**Why not abuse `:multipart_upload` capability and call this
"multipart"?** Because the broker session FSM, the adopter API surface,
and the failure-mode vocabulary are all distinct. Rindle keeps three
strategies (`presigned_put`, `multipart`, `tus`) and one storage backend
(`S3`) that backs both `multipart` and `tus`. This is the same pattern
tusd uses: tusd-with-S3-backend is a tus protocol implementation that
happens to use S3 multipart internally. Adopters never see "multipart" in
the tus API surface.

## 8. Oban / cleanup / expiry contract

Reuse the v1.1 two-step cleanup lane. No new workers, only extended
behavior.

```elixir
config :my_app, Oban,
  queues: [rindle_maintenance: 1],
  plugins: [
    {Oban.Plugins.Cron,
     crontab: [
       {"0 1 * * *", Rindle.Workers.AbortIncompleteUploads},      # marks signed/uploading + tus past TTL as expired
       {"0 2 * * *", Rindle.Workers.CleanupOrphans,
        args: %{"dry_run" => false}}                              # deletes expired sessions + storage objects
     ]}
  ]
```

### Locked retention windows (defaults)

- `tus_session_ttl_seconds`: 24 hours (default)
  Justification: smartphone uploads on flaky LTE in foreign roaming get
  paused overnight; 24h covers the realistic resume window. Adopters can
  override via `config :rindle, :tus_session_ttl_seconds, 7 * 24 * 3600`
  for very large multi-day uploads.
- `tus_partial_storage_grace_seconds`: 0 (immediate cleanup once session
  is in `expired`). Hanging onto S3 multipart parts past expiry is a cost
  vector; we do not gate cleanup behind a separate grace window.
- `tus_max_upload_size_bytes`: 5 GiB default (configurable). Above this,
  S3 multipart maxes out at 10000 parts × 5GiB = 5TiB anyway, but Rindle
  caps lower to make the smartphone-AV case the explicit target, not
  archival storage.

### Cleanup on `expired`

`Rindle.Workers.CleanupOrphans` extends one path: when the session has
`upload_strategy == "tus"` and a non-null `multipart_upload_id`, it calls
`adapter.abort_multipart_upload/3` (idempotent — `:not_found` is success)
before deleting the row. This converges with the v1.1 multipart cleanup
exactly.

### Termination via `DELETE`

When a client calls `DELETE` on the tus URL (termination extension),
Rindle transitions the session to `aborted` and triggers the same
cleanup synchronously (off the request thread, via Oban) — `aborted` and
`expired` go through the same reaper.

### Closing the tusd footgun

tusd documents that "post-processing retries are not built in. You should
use another task management system." Rindle's answer: the post-finish
callback enqueues a durable `Oban.insert(PromoteAsset, ...)` inside the
same `Ecto.Multi` that transitions the session to `verifying`. If the
transaction commits, the job is durably enqueued. If it does not commit,
neither is the session marked completed. **No best-effort post-finish
hook.**

## 9. Auth + bearer-URL handling rules

This is the most security-critical section.

### tusd's documented gap

> "Currently, there is no mechanism to ensure that the upload is resumed
> by the same user that created it." (tusd advanced-topics/hooks docs,
> verified 2026-05-06)

tusd compensates with "the upload URL is randomly generated and only
short-lived." That posture is weaker than Rindle's v1.5 contract. Rindle
must do better.

### Rindle's locked posture

1. **Auth at creation** (POST /uploads/tus): the adopter's pipeline runs
   first. Rindle's `pre-create` callback receives the authenticated
   conn, captures user identity into `media_upload_sessions` via a
   broker-supplied callback, and signs the resulting tus URL with HMAC.
2. **Auth at resume** (HEAD/PATCH /uploads/tus/:id): Rindle validates
   the HMAC signature on the tus URL path. If the signature is missing
   or invalid → `:tus_url_signature_invalid` → 401.
3. **Adopter rebind hook**: optional but recommended. The adopter can
   provide `config :rindle, :tus_resume_authorizer, MyApp.TusAuth` (a
   callback module) that re-validates the resuming user against the
   captured user identity. If the hook returns `:reject`, Rindle returns
   401 regardless of HMAC validity. Default authorizer is a no-op (HMAC
   alone is enforced).
4. **Logging rule**: tus URLs are bearer-equivalent. The full URL is
   never logged. `Logger.metadata` is scrubbed. `inspect/1` on a
   `MediaUploadSession` returns the session id but not the signed URL —
   we add a `@derive {Inspect, except: [:tus_url_signature]}` to the
   schema. (The signature alone is not the full URL, but treating it as
   sensitive is the right default.)
5. **No Authorization header trust by Rindle's tus endpoint**. The
   adopter's pipeline owns auth headers; Rindle's tus endpoint owns
   session resolution. Mixing the two would let buggy adopter pipelines
   accidentally bypass session checks.

### Cookie / token rule

Adopters using session cookies route their auth pipeline through
`pipe_through :browser` (or equivalent) before `tus_routes`. Adopters
using bearer tokens route through `:api`. Rindle does not pick. The guide
documents both patterns; the security checklist requires one or the
other.

## 10. DX / UX / principle-of-least-surprise rules

### Adopter onboarding (copy-paste)

1. Add `{:rindle, "~> 1.6"}` to deps.
2. Add to endpoint: `pass: ["application/offset+octet-stream"]` in
   `Plug.Parsers`.
3. Mount in router: `tus_routes Rindle.Upload.TusController` under your
   chosen scope.
4. Initiate from your controller:
   `Rindle.Upload.Broker.initiate_resumable_upload(MyApp.MediaProfile,
   filename: "input.mp4", max_size: 5_368_709_120)` returns the location
   to hand to `tus-js-client`.
5. Wire the client (`tus-js-client` v4.3.x or `@uppy/tus`) to the
   returned `location`.
6. Run `mix rindle.doctor` and confirm `:resumable_upload` is reported
   honestly.

That is the entire surface.

### Telemetry contract (locked)

| Event | Measurements | Metadata |
|-------|--------------|----------|
| `[:rindle, :upload, :resumable, :start]` | `system_time` | `profile`, `adapter`, `session_id`, `max_size` |
| `[:rindle, :upload, :resumable, :patch]` | `bytes`, `duration_native` | `profile`, `adapter`, `session_id`, `offset_after` |
| `[:rindle, :upload, :resumable, :stop]` | `system_time`, `total_bytes` | `profile`, `adapter`, `session_id`, `asset_id` |
| `[:rindle, :upload, :resumable, :abort]` | `system_time` | `profile`, `adapter`, `session_id`, `reason` |
| `[:rindle, :upload, :resumable, :expire]` | `system_time` | `profile`, `adapter`, `session_id` |

This mirrors v1.4/v1.5 vocabulary. `[:rindle, :upload, :stop]` continues
to fire from `verify_completion/2` after the resumable session finishes
— so existing dashboards do not break.

### Error guidance via `Rindle.Error.message/1`

Each new atom from §4 gets an explicit, fix-oriented message in
`Rindle.Error.message/1`, matching v1.4's pattern:

```
:tus_offset_conflict ->
  "Resumable upload offset mismatch. The client's Upload-Offset header
  did not match Rindle's recorded offset. The client must HEAD the
  upload URL and resume from the server-reported offset."
```

### Anti-DX traps (locked-out)

- No silent downgrade to presigned PUT or multipart. Tagged error.
- No tus URL appearing in `inspect(session)` output.
- No background re-attempt of failed PATCHes from server side. Only
  client retries (per tus protocol).
- No new public function on `Rindle` facade beyond
  `Rindle.Upload.Broker.initiate_resumable_upload/2` and
  `cancel_resumable_upload/1`. The facade does not gain
  `Rindle.upload/3`-style sugar for tus; tus requires a client and is
  inherently advanced.

| # | Invariant | tus impact | Status |

## 12. Lessons from peer libs — DO / AVOID per peer

### tusd (Go reference server)

- **DO**: Treat upload handling as its own server concern; expose
  pre-create / post-finish hooks; use S3 multipart with on-disk buffering
  for sub-5MiB chunks; advertise `expiration` extension as first-class.
  (Verified at tus.github.io/tusd/storage-backends/aws-s3 and
  /advanced-topics/hooks.)
- **AVOID**: Best-effort post-processing hooks. tusd's own docs:
  "hooks are usually not retried." Rindle replaces this with a durable
  Oban enqueue inside the same Ecto.Multi as the state transition.

### tus-node-server / `@tus/server` v2.0

- **DO**: Pluggable datastore pattern (`@tus/file-store`, `@tus/s3-store`,
  `@tus/gcs-store`, `@tus/azure-store`). Rindle mirrors this via
  `Rindle.Storage` adapter capability advertisement.
- **AVOID**: A constructor-options soup. `@tus/server`'s API surface is
  large and fragmented across runtimes (Next.js, Lambda, Cloudflare,
  Bun, Deno). Rindle's "one Phoenix Plug, one router macro" is more
  opinionated and deliberately so.

### tus-ruby-server (Janko Marohnić)

- **DO**: Mountable Roda app with hook system (`before_create`,
  `after_finish`); pluggable metadata stores; explicit warning that
  Puma/Unicorn workers tie up for the duration of the upload. Rindle
  copies the mountable shape and the Ecto-backed metadata store.
- **AVOID**: Recommending Falcon-equivalent (non-blocking) servers as a
  hard dependency. The BEAM is naturally non-blocking; Bandit and Cowboy
  both handle long-lived PATCHes correctly. Rindle does not invent a
  preferred web server; it documents `read_length` and `read_timeout`
  knobs and trusts the BEAM.

### Shrine + tus-ruby-server (the integration boundary)

- **DO**: Treat tus as glue around the attachment lifecycle, not the
  lifecycle itself. Shrine-tus's "Approach C — unified storage" (same
  storage backs both tus and Shrine) is exactly what Rindle does
  natively: tus PATCHes land directly in the Rindle storage adapter via
  S3 multipart; promotion is metadata-only.
- **AVOID**: Shrine-tus's "Approach A — download through tus server"
  (re-uploading the file from tus to permanent storage). That doubles
  bandwidth and creates a second source of truth. Rindle's tus session
  IS the multipart upload; completion is metadata-only.

### Uppy + `@uppy/tus`

- **DO**: Default `chunkSize: Infinity` is correct for HTTP/2 streams
  but adopters on HTTP/1.1 with proxies that buffer must override.
  Rindle's guide says explicitly: "On HTTP/2 leave `chunkSize` alone;
  on HTTP/1.1 with body-buffering proxies, set `chunkSize` to 5 MiB
  to match S3's minimum part size."
- **AVOID**: Hard-coding `parallelUploads > 1`. Tus parallel uploads
  use the concatenation extension, which Rindle does not implement in
  v1.6. Document explicitly: `parallelUploads: 1` only.

### tus-js-client

- **DO**: Default retry delays `[0, 1000, 3000, 5000]` are sound; status
  409/423 are auto-retried. Rindle's 409 response on offset conflict is
  the documented contract.
- **AVOID**: `removeFingerprintOnSuccess: false` is the client default,
  meaning resumable URLs persist in browser localStorage indefinitely.
  Rindle's guide tells adopters to set `removeFingerprintOnSuccess: true`
  in production to avoid stale URL accumulation.

### Cloudflare Stream tus

- **DO**: 5,242,880-byte minimum chunk size and 256 KiB-divisibility
  rule are exactly the constraints S3 multipart imposes. Rindle reuses
  this as the documented guidance.
- **AVOID**: Cloudflare's pattern of leaking upload identity through the
  `stream-media-id` response header. Rindle returns the asset id only
  via the post-finish completion callback (and only after auth); the
  client never gets the asset id from a tus header.

### Mux UpChunk (the contrast case)

- **AVOID**: Mux chose NOT to use tus. They built their own resumable
  PUT with `Content-Range` headers because tus's auth model was
  insufficient for their use case. Rindle's HMAC-signed tus URL +
  optional rebind authorizer is the explicit answer to that critique;
  Rindle does not concede the point that tus is unauthable.

### Vimeo (historical)

- **DO**: Vimeo replaced their Resumable PUT API with tus in 2018.
  Validates tus as a stable choice for video ingest at scale.
- **AVOID**: Vimeo's tus-android-client 404 issue (tus-android-client
  GitHub #38) traces to URL TTL drift between client localStorage and
  server expiry. Rindle's `Upload-Expires` header + 410 response on
  expired sessions makes the client behavior deterministic.

### AWS S3 multipart (contrast only)

- **DO**: Lock S3 multipart as the storage backing for tus on S3.
- **AVOID**: Exposing S3 multipart APIs to clients. Tus is for clients
  that cannot or should not get presigned S3 URLs — typically because
  the client is a phone on a flaky connection, or because the adopter
  wants server-side validation before S3 sees the bytes. The two flows
  serve different adopters; Rindle keeps both.

## 13. Pros / cons / risks specific to Rindle

### Pros

- **Mobile-AV killer case.** Smartphone uploads of 100MB–5GB videos on
  LTE are exactly the scenario presigned-PUT and even multipart cannot
  recover gracefully. v1.4 shipped AV; v1.5 shipped adopter hardening;
  v1.6 with tus completes the "trust Rindle for production AV" arc.
- **Reuses every existing seam.** Broker, FSM, Oban cleanup, capability
  vocabulary, telemetry, error vocabulary — all extend additively. There
  is no new concurrent abstraction to maintain.
- **Closes the v1.1 reservation cleanly.** `:resumable_upload` was
  reserved on principle. v1.6 makes it real on the same principle:
  capability honesty.
- **The Elixir ecosystem is now ready.** Tussle v0.3.1 (March 2026,
  active maintainer, real bug fixes, configurable read_body) is a
  meaningful change from the 2026-05-05 memo's "ecosystem thin" stance.

### Cons

- **Boundary expansion.** Rindle becomes a server that runs the upload
  hot path. This is the largest scope expansion since v1.4 AV. Five
  phases is realistic; two phases is fantasy.
- **Operational dimension grows.** Adopters must know about
  `Plug.Parsers` `:pass`, `read_length`, `read_timeout`, proxy PATCH
  behavior, and S3 multipart minimums. The guide is non-trivial.
- **Tussle is one maintainer.** Active, but a single human. Bus-factor
  risk. Mitigation: the dependency is small (~700 LOC), Rindle could
  fork in a worst case, and the tus protocol surface is stable
  (1.0.0 since 2016).
- **Provider-tus parity is honest, not heroic.** R2 has its own tus
  surface; GCS has its own resumable surface. Rindle's tus uses the S3
  multipart backing for S3-compat providers and does NOT speak to
  provider-native tus (R2's, Mux's, Cloudflare Stream's). The guide is
  explicit. Adopters who want provider-native tus do not need Rindle in
  the request path.

### Risks

- **Risk 1: BEAM RAM blowup on naive `read_body` config.** Mitigation:
  TUS-06 hard-locks `:read_length: 1MiB, :length: 16MiB` per call.
  Property test under load.
- **Risk 2: HMAC URL signing implemented incorrectly.** Mitigation:
  TUS-12 requires a security review checklist and tampered-URL contract
  tests. Use `Plug.Crypto.MessageVerifier` (already a Phoenix dep), not
  hand-rolled HMAC.
- **Risk 3: Cleanup leaks S3 multipart parts on edge cases.**
  Mitigation: TUS-15 + MinIO integration that deliberately abandons an
  upload, runs the reaper, and asserts `list_multipart_uploads` returns
  empty.
- **Risk 4: Adopter mounts the tus Plug under an unauthenticated
  pipeline.** Mitigation: the guide's security checklist explicitly
  says "if `:require_authenticated_user` is not in your pipeline, your
  storage costs are now bound only by the universe of internet
  attackers"; `mix rindle.doctor` reports the configured pipeline
  scope where possible.

## 15. Open decisions deliberately deferred

These are not "we should think about it"; they are explicit "v1.7+,
documented as deferred" calls.

- **GCS-native resumable uploads.** Out of scope for v1.6. GCS is its
  own milestone candidate. Rindle's tus implementation does not speak
  GCS resumable; if adopters use GCS, they use the GCS milestone or
  external tusd. The capability advertisement on a future
  `Rindle.Storage.GCS` adapter would be `:resumable_upload_gcs_native`,
  not the same atom.
- **IETF draft-ietf-httpbis-resumable-upload (tus 2.0 / RUFH).** Draft-11
  as of 2026-03; still a draft. Adopters using `tus-js-client` v4.3+
  can opt into `protocol: 'ietf-draft-05'` against future Rindle
  versions. Not v1.6.
- **Concatenation extension** (parallel chunked uploads merged
  server-side). Useful but adds complexity. Not v1.6.
- **Checksum extension.** SHA-1 per chunk. Defer until adopter demand
  shows up — `verify_completion/2` already validates final size and
  content_type; per-chunk SHA-1 mostly catches transit corruption that
  TLS already prevents.
- **LiveView upload component for tus.** v1.4 shipped LiveView progress
  for AV processing; a tus-aware uploader component is natural for
  v1.7. Not v1.6.
- **External tusd adapter** (the rejected option from §1). If adopters
  end up wanting it later, it becomes `Rindle.Upload.Tus.External` — a
  separate seam. Not in v1.6 to keep one shipping target.
- **R2-native tus surface.** Cloudflare R2 ships its own tus endpoint.
  Rindle's docs describe it but Rindle does not proxy or wrap it.
  Adopters who want R2-native tus point the tus client at R2 directly
  and use Rindle for everything post-completion. Out of scope for v1.6.
