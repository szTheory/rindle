# v1.5 Candidate Memo: Adopter Hardening After AV Ship

**Project:** Rindle
**Date:** 2026-05-05
**Candidate:** post-v1.4 adopter hardening and feedback closure
**Recommendation confidence:** HIGH for scope/rationale, MEDIUM for exact adopter pain ranking until real post-ship issue volume arrives

## 1. Candidate Summary

Recommended v1.5 shape: **Adopter Runtime Hardening & Lifecycle Repair**.

This should be a real milestone, not a polish bucket. Scope it around four bounded deliverables:

1. **Outside-in adopter proof matrix**
   - Expand package-consumer smoke beyond the current happy path to cover image-only and AV-enabled installs, local dev playback, signed delivery, and one real S3-compatible path.
   - Keep generated-app proof primary. Fixture apps remain supporting evidence.
   - Add docs-parity gates so README, getting-started, AV onboarding, and ops guides all describe the same proved path.

2. **Lifecycle repair operations**
   - Ship first-class repair surfaces for the states v1.4 introduced: re-probe asset, requeue failed/cancelled variants, regenerate derivative sets after preset changes, and sweep orphans/stale temp files on demand.
   - Expose these as explicit `Rindle.Ops.*` entrypoints and `mix` tasks, not hidden internal helpers.
   - Require idempotency, auditability, and tagged failure reasons.

3. **Capability-drift and runtime diagnostics**
   - Extend `mix rindle.doctor` from install preflight into runtime ownership checks: FFmpeg drift, storage capability drift, missing queues, delivery plug misconfiguration, and stale migration detection.
   - Add frozen telemetry and operator-facing failure categories for repair paths and runtime refusals.

4. **Upgrade and migration safety**
   - Add package-consumer upgrade smoke from a pre-v1.4 app into the current schema/runtime shape.
   - Verify additive migrations, backfill behavior, and operational recovery after interrupted AV processing.
   - Publish a durable upgrade/runbook guide instead of assuming greenfield installs.

In roadmap terms, this is the milestone that turns the AV wedge from “feature-complete” into “safe to adopt and safe to live with.”

## 2. Pros For Rindle Specifically

- It compounds directly on Rindle’s strongest differentiator: durable lifecycle ownership, not raw media feature count.
- It matches the project’s proven wins from v1.1 and v1.4: package-consumer-first proof, capability honesty, and explicit day-two operations.
- It reduces the biggest post-AV trust risk: adopters can now ingest, probe, transcode, and serve media, but failure recovery and upgrade ergonomics are still thinner than the core happy path.
- It keeps Rindle focused on its library identity. Provider adapters and tus push the project toward integration breadth; this milestone deepens the core lifecycle contract instead.

## 3. Cons/Risks For Rindle Specifically

- It is less marketable than “GCS support” or “tus support,” so the user-visible headline is weaker.
- Real adopter feedback may fragment into many small asks; without strict phase discipline this can devolve into generic cleanup work.
- Hardening milestones can uncover API boundary mistakes in `Rindle.Ops` or diagnostics surfaces, which risks accidental public-surface growth.
- The milestone only pays off if verification stays outside-in. If it becomes mostly internal refactors, it will consume time without increasing adopter trust.

## 4. Tradeoffs Versus Other Likely Candidates

**Versus provider adapters / streaming boundary**
- Provider adapters are strategically important, but they widen the support matrix before the current AV wedge has enough operational proof.
- The better sequence is: harden `streaming_url/3`, delivery telemetry, and failure semantics now; land provider adapters after those seams are boring.

**Versus GCS adapter**
- GCS expands cloud reach, but it is mostly breadth. Rindle still has more leverage tightening lifecycle repair and upgrade safety across the storage shapes it already supports.
- GCS becomes cheaper after this milestone because capability drift, docs parity, and repair conventions will already exist.

**Versus tus protocol support**
- tus is a larger protocol boundary than it looks: client choreography, resumable state semantics, cancellation, expiry, and Plug/Phoenix integration all widen the public contract.
- Multipart already covers the nearer large-upload need. Hardening the current broker, maintenance, and recovery story is the more disciplined next step.

Net: this candidate delays breadth in exchange for making the current product survivable under real adopter load. That trade is correct immediately after a major wedge ship.

## 5. Idiomatic Fit For Elixir/Phoenix/Ecto/Plug

- **Elixir library fit:** strong. Explicit tagged errors, `mix` diagnostics, telemetry contracts, and small public seams are idiomatic.
- **Phoenix fit:** strong. Generated-app smoke and guide parity align with Phoenix’s “real app first” ergonomics.
- **Ecto fit:** very strong. Repair flows, additive migrations, and explicit backfills belong naturally in Ecto/Oban, not in implicit storage-side magic.
- **Plug fit:** strong if delivery/diagnostic instrumentation stays explicit and span-shaped rather than magical. `Plug.Telemetry` is the right model for boring observability.

This candidate is more idiomatic than tus or provider-adapter work right now because it stays inside Rindle’s existing library boundary instead of introducing a larger integration surface.

## 6. Lessons From Comparable Libraries/Frameworks

**Rails Active Storage**
- Right: variant tracking, built-in GCS support, and mirror-service migration posture are practical patterns.
- Footguns: mirror writes are not atomic; lazy processing and variant tracking can create surprise latency and N+1 behavior.
- Copy: explicit migration/runbook posture and cached derivative knowledge.
- Avoid: hiding operational caveats behind a “just attach files” surface.

**Shrine**
- Right: atomic promotion and attachment-change concurrency protection are exactly the kind of lifecycle rigor Rindle should emulate.
- Footguns: plugin composition is powerful but can make the operational contract harder to see.
- Copy: explicit concurrency-safe promote/replace semantics.
- Avoid: making repair behavior depend on too many opt-in combinations.

**Spatie Media Library**
- Right: queued conversions after DB commit, regeneration tools, and strong day-two ergonomics.
- Footguns: sync-vs-queued toggles and model-instance-dependent conversions can create performance and surprise issues.
- Copy: regeneration/repair as first-class supported operations.
- Avoid: options that silently switch lifecycle semantics or hide extra query cost.

**Phoenix LiveView / Plug**
- Right: public upload progress/cancel contracts and explicit telemetry are simple and teachable.
- Copy: narrow, explicit event vocabularies and composable hooks.
- Avoid: expanding Rindle’s public LiveView contract faster than it can freeze and document it.

## 7. DX Implications

**API shape**
- Prefer additive, ops-oriented APIs: `reprobe`, `requeue`, `regenerate`, `repair`, `audit`.
- Do not redesign upload or AV facade calls in this milestone.

**Docs**
- README should stay the shortest proven install path.
- Deep guides should add upgrade, repair, and runtime-drift playbooks with parity tests.

**Installation friction**
- Slightly higher for maintainers, lower for adopters. More verification lanes and doctor checks add work in CI, but they remove hidden setup mistakes.

**Observability**
- This milestone should freeze the operational telemetry needed to answer “what is stuck, why, and how do I recover it?” That is the current gap.

**Migration safety**
- High payoff. v1.4 added schema and lifecycle breadth; v1.5 should prove upgrades, backfills, and post-failure recovery before adding more protocols.

**Principle of least surprise**
- Strong fit if every repair action is explicit, idempotent, and documented with exact side effects.
- Bad fit if Rindle starts auto-healing in the background without telling operators what changed.

## 8. Recommendation Score

**8.5/10 for “next milestone now.”**

Why not higher:
- It is less adoption-marketable than GCS or tus.
- Its exact shape should still be constrained by the first round of real adopter feedback.

Why it still wins now:
- It best matches Rindle’s identity.
- It protects the newly shipped AV wedge before breadth work multiplies the blast radius.
- It is the most cohesive option with package-consumer-first verification, docs parity, and durable lifecycle operations.

## Bottom-Line Recommendation

Make v1.5 an **adopter-hardening milestone centered on lifecycle repair, runtime diagnostics, and upgrade proof**. Do not let it sprawl into generic cleanup, and do not mix in GCS or tus. Provider-adapter work should follow once the current AV wedge has a boring, operator-grade recovery story.
