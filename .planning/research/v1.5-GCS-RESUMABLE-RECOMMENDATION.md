# Candidate Milestone Memo: GCS Adapter + Resumable Uploads

**Project:** Rindle  
**Date:** 2026-05-05  
**Recommendation:** Worth planning, but not the automatic next milestone without clear adopter pull.

## 1. Candidate Summary

A meaty but bounded `v1.5` would add **one new storage adapter** (`Rindle.Storage.GCS`) plus an **explicit resumable-upload family** that mirrors GCS reality instead of pretending it is S3 multipart.

Recommended scope:

- `Rindle.Storage.GCS` with truthful capabilities for:
  - signed/private delivery URL
  - object metadata `head`
  - resumable upload session bootstrap
  - resumable upload status query
  - resumable upload cancel
- Additive broker/API surface, separate from current PUT and multipart APIs:
  - `initiate_resumable_upload/2`
  - `resumable_upload_status/2`
  - `cancel_resumable_upload/2`
  - reuse existing `verify_completion/2` after client finishes upload
- Additive upload-session persistence for GCS-specific state:
  - session URI
  - content type / expected length hints
  - last known offset / status metadata
- Maintenance integration:
  - abort stale resumable sessions through Oban
  - never delete local session rows before remote cancel succeeds or is proven idempotent
- CI/docs proof:
  - preserve current presigned PUT + multipart S3 proofs unchanged
  - add GCS-specific proof lane and docs that explain session-URI secrecy, status checks, and cancel semantics

Do **not** scope in:

- generic tus support
- auto-fallback from resumable to PUT
- provider-agnostic “unified resumable” magic
- any change that rewires existing S3 entrypoints

## 2. Pros for Rindle Specifically

- It is an explicit carry-over from `v1.1`, so it closes a known deferred gap instead of inventing a new lane.
- It fits the existing **capability-honesty** posture well if kept as a separate API family.
- It reuses Rindle’s strongest seam: broker-owned upload sessions converging back into the same trusted `head` verification path.
- It expands provider coverage without violating the adopter-owned Repo or Oban-required job model.
- It is narrower than tus and narrower than a full streaming/provider milestone.

## 3. Cons / Risks for Rindle Specifically

- GCS resumable is **not** S3 multipart with different names. It introduces session-URI, offset, status, and cancel semantics that want their own contract.
- The session URI is effectively a bearer token; logging, telemetry, inspect output, and DB visibility become more sensitive than current `upload_id` handling.
- GCS has a real footgun around geography: Google notes resumable uploads are pinned to the region of the initial request, so server-initiated sessions can be suboptimal for distant clients.
- Capability naming is currently only half-designed (`:resumable_upload`, `:resumable_upload_session` reserved). Locking the wrong public semantics here would create cleanup work.
- Proof burden is meaningfully higher than S3/MinIO because Rindle needs confidence in status, resume, and cancel behavior, not just object existence.

## 4. Tradeoffs vs Other Likely Candidates

**Vs provider adapters / streaming boundary**

- GCS is more bounded and closer to existing seams.
- Streaming/provider work is likely more strategic after `v1.4`, but it is a bigger abstraction bet: webhooks, playback IDs, provider lifecycle, and non-storage delivery semantics.
- If the goal is “ship one contained storage expansion,” GCS wins.
- If the goal is “capitalize on the fresh AV wedge,” streaming/provider work may have higher upside.

**Vs tus protocol support**

- GCS is much smaller.
- tus is cleaner as a protocol, but it pulls Rindle toward being an upload server/protocol implementation, not just a lifecycle library.
- tus also wants a broader client/server contract than Rindle currently exposes.
- Recommendation: do GCS before tus if provider demand exists; do not do both in one milestone.

**Vs adopter hardening**

- Adopter hardening is lower novelty but higher trust.
- Right after `v1.4`, hardening is the safer “protect the core” choice if real adopter feedback is already surfacing.
- GCS is the better pick only if the milestone goal is deliberate storage-surface expansion, not stabilization.

## 5. Idiomatic Fit for Elixir / Phoenix / Ecto / Plug

- **Good fit:** Ecto session rows, additive migration, explicit FSM transitions, and Oban cleanup/status work all match Rindle’s current architecture.
- **Good fit:** Plug/Phoenix only need to broker session bootstrap and status endpoints; bytes still flow directly to GCS.
- **Mixed fit:** Elixir has usable GCS pieces (`google_api_storage`, `goth`, `gcs_signed_url`), but not the same clean, battle-tested “ExAws-for-this-exact-flow” feeling Rindle has on S3.
- **Important fit constraint:** keep Google-specific auth/client wiring optional and adapter-local. Do not leak it into the core happy path for S3 adopters.

## 6. Lessons from Comparable Libraries / Frameworks

**Active Storage**

- Good: one app-facing direct-upload story across providers.
- Footgun: its GCS service documents that some response parameters are not safely enforced by GCS signed URLs, so it stores critical disposition/type data as object metadata instead.
- Copy: hide provider weirdness in the adapter.
- Avoid: pretending provider semantics are more uniform than they are.

**Shrine**

- Good: protocol/provider-specific power lives in plugins (`presign_endpoint`, S3 multipart, tus) rather than being blurred into one vague upload API.
- Footgun: direct-upload metadata can be tampered with unless the server re-extracts trusted metadata.
- Copy: keep advanced flows opt-in and explicit.
- Avoid: trusting client-reported upload facts.

**tus**

- Good: explicit offset/status/termination semantics.
- Lesson: resumable upload is a protocol surface, not just a storage capability flag.
- Copy: if Rindle adds resumable APIs, make offset/status/cancel explicit.
- Avoid: shipping a fake-generic resumable abstraction that hides state transitions.

## 7. DX Implications

Recommended API/DX posture:

- Add a **separate resumable family**. Do not overload `sign_url/2` or multipart entrypoints.
- Keep capability failures explicit: `{:error, {:upload_unsupported, :resumable_upload_session}}`.
- Document that the returned session URI is secret and must not be logged or exposed outside the uploading client.
- Reuse `verify_completion/2` as the final convergence point so adopters still reason about one promotion path.

Expected DX burden:

- Higher configuration burden than S3: bucket config, Google auth, signing/delivery choices, CORS, and resumable-session behavior all need docs.
- More surprising failure modes than S3 PUT:
  - stale session URI
  - offset mismatch / partial persistence
  - session cancellation
  - one-week expiry
  - region-pinned initiation
- Principle-of-least-surprise rule: Rindle should never silently choose PUT when resumable was requested, and never silently choose resumable on GCS when the adopter asked for plain PUT semantics.

## 8. Recommendation Score

**Score: 7/10 for “next milestone now.”**

Why not higher:

- It is a valid, bounded carry-over, but it adds a new public upload contract with security-sensitive session handling.
- Right after the `v1.4` AV ship, adopter hardening may still be the more prudent default if feedback is already arriving.
- The milestone is strong if the intent is storage expansion; weaker if the intent is consolidation.

**Bottom line:** plan it before tus, keep it separate from streaming/provider work, and only make it the immediate next milestone if there is concrete GCS adopter demand. Otherwise, do one hardening milestone first and take GCS as the next expansion milestone.
