## 1. Problem & Evidence (repo-grounded)

### 1.1 The global-config mechanism is confirmed

`Rindle.Config.repo/0` is process-**global** — it reads a single application-env key with no process scoping:

```elixir
# lib/rindle/config.ex:9-12
@spec repo() :: module()
def repo do
  Application.get_env(:rindle, :repo, Rindle.Repo)
end
```

It is read on **hot paths all over `lib/`** (≈40 call sites): `lib/rindle.ex` (×5), `lib/rindle/streaming.ex:64,137`, `lib/rindle/delivery.ex:248` (`get_by`), `lib/rindle/live_view.ex:329`, `lib/rindle/admin/queries.ex` (×11), all `lib/rindle/workers/*`, `lib/rindle/ops/*`. Every one of those calls resolves the **same global** for **every concurrently-running test process**.

### 1.2 The mutation site

`test/support/counting_failing_txn_repo.ex:6-21`:

```elixir
def with_counting_repo(fail_after, fun) when is_integer(fail_after) and is_function(fun, 0) do
  previous_repo = Application.get_env(:rindle, :repo)
  ...
  Application.put_env(:rindle, :repo, __MODULE__)          # ← GLOBAL mutation
  Application.put_env(:rindle, :counting_failing_txn_repo, fail_after: fail_after)
  reset_count()
  try do
    fun.()
  after
    restore_env(:repo, previous_repo)                       # window closes only after fun.()
    ...
  end
end
```

For the entire duration of `fun.()`, `:rindle, :repo` globally points at the double. Any process anywhere in the BEAM that calls `Rindle.Config.repo()` in that window dispatches through `CountingFailingTxnRepo` — and gets a **force-failed transaction** if its call happens to be the Nth, or (historically) an `UndefinedFunctionError` for any un-proxied function. The 2026-06-26 symptom fix (the `Rindle.Repo.__info__(:functions)` passthrough generator, lines 66-73) made the proxy **complete** so it can no longer raise `UndefinedFunctionError`, but it did **not** stop the double from intercepting unrelated tests' transactions — completeness papers over the crash, not the cross-pollution.

### 1.3 The consumers are ALREADY `async: false` — and so is the victim

| Module | Role | Flag | Source |
|---|---|---|---|
| `Rindle.OwnerErasureBatchProofTest` | mutator (calls `with_counting_repo`) | `async: false` | `owner_erasure_batch_proof_test.exs:7` |
| `Rindle.BatchOwnerErasureTaskTest` | mutator | `async: false` | `batch_owner_erasure_task_test.exs:2` |
| `Rindle.Delivery.StreamingDispatchTest` | **victim** (reads `Config.repo()` on Branch 5/5b) | `async: false` | `streaming_dispatch_test.exs:26` |

The victim's own header documents the workaround and explicitly defers the root cause to a later milestone:

```elixir
# streaming_dispatch_test.exs:21-25
# async: false — this suite is sensitive to global-state pollution from concurrently-running
# async tests (a test that swaps the global `:rindle, :repo` or force-fails a transaction in its
# window makes Branch 5/5b dispatch return the wrong value → intermittent `==` failures).
# Serializing these 17 tests is cheap and makes the dispatch matrix deterministic; the broader
# root-cause cleanup (eliminate the global mutations) is tracked separately.   ← THIS MILESTONE
```

**This is the load-bearing observation.** The fix that already shipped was to **demote the innocent victim to serial**, not to fix the mutator. That is whack-a-mole: every future async test that happens to read `Config.repo()` is one merge away from the same intermittent failure, and the only "cure" on offer is to keep demoting victims to `async: false` — a steadily-growing serialization tax (currently 56 of 130 test modules are already `async: false`).

### 1.4 The v1.20 async-safety guard has a real GAP here — and it is structural, not an escape-hatch leak

I verified there is **no** `@async_safety_allow [:application_put_env]` anywhere in the suite (grep: the only occurrences are the guard's own docstring/parser, plus two `[:file_mutation]` annotations on `storage/local_test` and `package_metadata_test`). So scenario (ii) "escape-hatch annotation" is **ruled out**.

The reason the guard does not catch this is **scenario (i), and it is fundamental to the guard's design**:

- The guard (`test/async_safety_guard_test.exs`) only inspects modules whose **own** header declares `async: true` (`async_true?/1`, line 147). It flags `Application.put_env` **only inside an `async: true` module**.
- The `with_counting_repo` mutators are `async: false`, so the guard **correctly ignores them** — by its contract, an `async: false` module is "allowed" to mutate global state.
- **But `Application.put_env` is process-global regardless of the mutator's async flag.** An `async: false` module still executes *concurrently with the entire `async: true` pool* (see §2). So a put_env from an `async: false` module pollutes `async: true` readers — a cross-pool race the guard's per-module model **cannot see**, because it never reasons about a global mutation in module A being observed by module B.

**GAP, stated precisely:** the guard enforces "an `async: true` module must not mutate shared global state." It does **not** enforce "no module (of any async flag) may mutate a *globally-read library config key* like `:rindle, :repo` while async readers exist." The dangerous primitive here is not "put_env in an async module" — it is "put_env of a key that `lib/` reads on every request, from *any* module, ever." Closing that gap is part (iv) of this area's scope.

## 2. ExUnit / async-pool semantics ruling (load-bearing)

This determines the fix, so it is stated definitively with the canonical model.

> **`async: false` does NOT make a test run exclusively of the `async: true` pool.**

ExUnit's concurrency model:

- `async: true` modules are distributed across a worker pool of size `System.schedulers_online()` (configurable via `--max-cases`) and **run concurrently with each other**.
- `async: false` modules run **serially with respect to each other** — ExUnit guarantees no two `async: false` modules run at the same time. **It does NOT guarantee an `async: false` module runs alone in the BEAM.** The standard documented behavior (and ExUnit's actual scheduler) interleaves `async: false` modules *with* the running `async: true` pool. Concretely, the runner keeps the async pool saturated and slots sync modules into the schedule; while a sync module runs, async modules from the pool are still executing on other scheduler cores.

**Therefore:** marking a `put_env` mutator `async: false` only guarantees it does not collide with *other sync mutators*. It provides **zero protection** against an `async: true` test reading the polluted global concurrently. This is exactly why demoting `StreamingDispatchTest` to `async: false` "fixed" it — both mutator and victim are now sync, so they are serialized **with each other** — but it does nothing for the *next* async test to read `Config.repo()`. The only way to make the global-put_env approach safe is to demote **every reader of that global** to `async: false`, i.e. effectively serialize the suite around this one key. That is the tax we are paying and the reason this milestone exists.

**Corollary (this drives the recommendation):** a global `Application.put_env` of a key that production code reads on every call is **unsafe in an async suite no matter what async flag the mutator carries.** The fix must eliminate the *global* mutation, not relabel modules. Per-process scoping is the only mechanism that makes the override invisible to concurrent readers.

## 3. Options

### Option (i) — Per-process repo override in the resolver (`$callers`-aware)

**Mechanism.** Make `Rindle.Config.repo/0` consult a process-dictionary override (walking `$callers` so spawned Tasks / inline-Oban jobs inherit it) *before* falling back to the application env. The test helper sets the override only in the calling process's dictionary; concurrent readers in other process trees never see it.

```elixir
# lib/rindle/config.ex
@repo_override_key {__MODULE__, :repo_override}

@spec repo() :: module()
def repo do
  with nil <- repo_override(self()) do
    Application.get_env(:rindle, :repo, Rindle.Repo)
  end
end

# Walk self() then the $callers chain (Task/async, Oban inline) so a repo override
# set in the test process is visible to processes it spawned.
defp repo_override(pid) do
  case process_get(pid, @repo_override_key) do
    nil -> caller_repo_override(pid)
    mod -> mod
  end
end

defp caller_repo_override(pid) do
  pid
  |> process_get(:"$callers")
  |> List.wrap()
  |> Enum.find_value(fn caller -> caller != pid && repo_override(caller) end)
end

defp process_get(pid, key) when pid == self(), do: Process.get(key)
defp process_get(pid, key) do
  case Process.info(pid, :dictionary) do
    {:dictionary, dict} -> Keyword.get(dict, key)
    _ -> nil
  end
end
```

Test helper sets it **process-locally**, no global mutation:

```elixir
# in CountingFailingTxnRepo / a test-only setter
def put_repo_override(mod), do: Process.put({Rindle.Config, :repo_override}, mod)
def delete_repo_override, do: Process.delete({Rindle.Config, :repo_override})
```

**Pros.** Eliminates the global mutation entirely → the race is gone by construction; tests stay `async: true`; idiomatic (this is exactly the Mox/Ecto-Sandbox `$callers` pattern). One small resolver change covers all ~40 call sites because they all funnel through `Config.repo/0`.

**Cons / tradeoffs.** (a) **`lib/` touch** — modifies the adopter-facing resolver. (b) Reads another process's dictionary via `Process.info/2` for the `$callers` walk (a few extra reductions on the **fast path of every repo call**); the common case is a single `Process.get` of `nil` which is cheap, but the `$callers` walk runs only when an override is present, so production cost is effectively one `Process.get` returning `nil`. (c) Adds a tiny ambient capability (a process-dict override) to the production resolver.

**lib/-touch verdict.** Yes, but **adopter-invisible**: the default branch is unchanged (`Application.get_env(:rindle, :repo, Rindle.Repo)`), the override key is `Rindle`-namespaced and only ever set by test support, and no public function signature changes. Blast radius is one private function in `Config`. This is the seam the library already exposes for repo configuration; making it process-aware is consistent with how every mature Elixir lib resolves a swappable dependency.

### Option (ii) — Mark all `with_counting_repo` consumers `async: false`

**Mechanism.** Status quo + ensure both mutator modules stay `async: false`.

**Cons.** **Insufficient by §2.** Both mutators are *already* `async: false` and the pollution still occurred (it bit `StreamingDispatchTest`). The only way to make this sound is to *also* demote **every** `async: true` module that transitively reads `Config.repo()` — which is most of the suite — re-serializing it and defeating the v1.20 async program. It is also unprovable: nothing stops a future async test from reading the repo and silently re-introducing the flake. **Reject as the primary fix.** (Keeping the two mutators `async: false` is fine as defense-in-depth, but it is not the cure.)

### Option (iii) — Ecto SQL Sandbox-level failure injection (no repo swap)

**Mechanism.** Implement "fail the Nth transaction" *without* swapping the global repo, by injecting the failure at a process-scoped layer. The cleanest is to keep `:rindle, :repo` = `Rindle.Repo` always, and have the double be selected via the **already-process-scoped** `tags[:sandbox_repo]` seam that `Rindle.DataCase.setup_sandbox/1` supports:

```elixir
# test/support/data_case.ex:22-25 — this seam ALREADY exists
def setup_sandbox(tags) do
  repo = tags[:sandbox_repo] || Rindle.Repo
  pid = Sandbox.start_owner!(repo, shared: not tags[:async])
  ...
end
```

But `sandbox_repo` only scopes the **sandbox owner**, not what `Config.repo()` returns — so on its own it does not redirect `lib/` to the counting double. To make the double process-scoped you still need the resolver to read a per-process value → this collapses into Option (i). The "pure-Sandbox" variant (fail a transaction via a Sandbox callback/connection wrapper) is **not** a first-class Ecto feature: Ecto Sandbox isolates connections per process but offers no public "fail the Nth transaction" hook, so you'd be wrapping the adapter — more invasive than (i) and still needs per-process selection.

**Verdict.** The *isolation* lesson from Sandbox is right (per-process, not global) and we should emulate it — but the *implementation* still requires a process-scoped resolver. So (iii) reduces to (i) plus extra adapter plumbing we don't need. **Reject as standalone; adopt its principle inside (i).**

### Option (iv) — Strengthen the v1.20 async-safety guard to close the cross-pool gap

**Mechanism.** Add a rule that does **not** depend on the mutating module's async flag: forbid `Application.put_env`/`delete_env` of a **globally-read library config key** (at minimum `:rindle, :repo`) **anywhere in the test tree**, directing authors to the process-scoped override instead. Two concrete shapes:

```elixir
# New detection in async_safety_guard_test.exs, applied to ALL modules (not just async:true):
# flag any Application.put_env(:rindle, :repo, _) / delete_env(:rindle, :repo)
defp classify_global_repo_swap(
  {{:., meta, [{:__aliases__, _, [:Application]}, m]}, _, [:rindle, :repo | _]}, _)
  when m in [:put_env, :delete_env],
  do: {:global_repo_swap, line(meta)}
```

Because today's guard only walks `async: true` module bodies, this new rule must be hoisted to scan **every** module (the mutators are `async: false`). The allow-list stays available for the (now nonexistent) legitimate case.

**Pros.** Makes the cross-pool footgun *un-reintroducible*: a future contributor who reaches for `Application.put_env(:rindle, :repo, ...)` gets a red, merge-blocking gate pointing them at `Config.put_repo_override/1`. Directly closes the structural gap from §1.4.

**Cons.** Pure meta-test; doesn't fix anything by itself — it is the *enforcement* half that must ship **together with** (i) (otherwise it just flags code with no sanctioned alternative).

**Verdict.** **Adopt as the enforcement complement to (i).** This is the coherent continuation of HARD-01: HARD-01 said "async modules may not mutate shared state"; this says "no module may swap the globally-read repo key." Same philosophy, closes the hole HARD-01 structurally couldn't.

## 4. Idiomatic-for-Elixir analysis

The `$callers`-aware process-dictionary override is **the** canonical Elixir pattern for "swap a global dependency for the current test without polluting concurrent async tests." It appears verbatim in the ecosystem:

- **`$callers`** is a BEAM convention: `Task.async`, `Task.Supervisor`, and Oban (inline testing) all propagate `:"$callers"` so a spawned process can find the dictionary/allowance state of the test that started it. Walking `self()` then `$callers` is exactly how Mox and Ecto Sandbox make per-process state visible to spawned workers. Our resolver must walk it so inline-Oban jobs and `Task`-spawned work (which `lib/rindle/workers/*` and streaming paths use) inherit the override.
- **Process-dictionary as the override store** is idiomatic *for test seams* precisely because it is process-local: it cannot leak to a concurrently-running test in another process tree. (It would be a smell in *business* logic; as a test-override seam consulted before the global default, it is standard.)
- **Sandbox's `allow/3` and `{:shared, pid}`** are the gold standard: per-process ownership by default, explicit `allow(repo, owner_pid, allowed_pid)` to extend a checkout to a spawned process, and `{:shared, pid}` *only* when `async: false`. Note `Rindle.DataCase` already does the right thing — `shared: not tags[:async]` (data_case.ex:24) — which is why the guard explicitly does **not** flag the Sandbox (guard docstring lines 16-23). Our repo override should mirror this: process-scoped by default, `$callers`-aware for spawned work.

## 5. Peer-library lessons (right-to-emulate + footguns)

| Lib | What it got RIGHT (emulate) | Footgun it avoids (our current bug) |
|---|---|---|
| **Mox** | `set_mox_private` (default) + `allow/3` + `$callers` propagation. Expectations live in the **test process**, not a global; concurrent async tests each have their own. | `set_mox_global` makes mocks process-global — the exact analogue of our `put_env(:rindle, :repo)`. Mox *documents this as unsafe for async* and the v1.20 guard already flags `set_mox_global`. Our repo swap is the un-flagged sibling. |
| **Ecto.Adapters.SQL.Sandbox** | Per-process connection ownership; `allow/3` for spawned procs; `{:shared, pid}` **only** with `async: false`. The whole point of Sandbox is "global DB, per-process isolation." | Shared mode under async → cross-test data bleed. We already gate this correctly via `shared: not tags[:async]`; the repo swap is the one place we *didn't* apply the same per-process discipline. |
| **Oban (`testing: :inline`)** | Inline jobs run in the **enqueuing process** (config/test.exs:56 sets `testing: :inline`), so `$callers` carries test state into the job. | A `$callers`-blind resolver would lose the override when work hops into an inline Oban job — hence the resolver MUST walk `$callers`. |

**The cross-cutting lesson:** every mature Elixir lib that lets tests swap a global dependency does it **process-scoped with `$callers` propagation**, never via a `try/after` global `put_env`. Global config swap inside an async suite is a known footgun; we have an instance of it and the demote-the-victim "fix" is precisely the anti-pattern these libraries were designed to make unnecessary.

## 6. DX / reliability

- **Stays maximally async (fast).** Option (i) lets us **revert `StreamingDispatchTest` (and the two mutators, if otherwise clean) back to `async: true`**, recovering the serialization tax that the workaround imposed. No reader of `Config.repo()` ever needs `async: false` *for this reason* again.
- **A contributor knows isolation holds** because (a) the override is set with a named helper (`with_counting_repo` / `Config.put_repo_override/1`) that is self-evidently process-local, and (b) the strengthened guard (iv) makes the **wrong** way (`Application.put_env(:rindle, :repo, …)`) a red merge gate with a message pointing at the right way. Enforcement + a sanctioned alternative = the contributor literally cannot reintroduce the flake.
- **The guard should enforce it** — yes. This is the coherent extension of HARD-01 and the only thing that prevents regression. The proof test (§8) plus the guard rule together make the property *executable* and *un-droppable*.

## 8. LOCKED Recommendation

**Make `Rindle.Config.repo/0` consult a `$callers`-aware process-dictionary override before the application env, set per-process by the counting double (eliminating the global `Application.put_env(:rindle, :repo, …)` entirely), and strengthen the v1.20 async-safety guard to forbid any global swap of `:rindle, :repo` anywhere in the test tree.** This is a **deliberate, adopter-invisible `lib/` touch** (one private function in `Config`; default behavior unchanged, no public-API/semver impact) — it does **not** require escalation under the research-driven posture, but because v1.20 carried a *zero-lib-change* invariant, this milestone must **explicitly relax that invariant for this single seam** in its charter (the touch is the adopter-facing repo resolver, justified because the bug is *in* that resolver's global nature and cannot be fixed in test-only code). **Guard-gap verdict: real and structural** — HARD-01's per-module model cannot see a global mutation in an `async: false` module observed by the `async: true` pool; the new guard rule (applied to *all* modules) closes it.

### Coherence with prior decisions

- **HARD-01 continuation, not contradiction.** HARD-01 forbids unsafe primitives in `async: true` modules; this adds "no module may swap the globally-read `:rindle, :repo` key," the structural hole HARD-01 couldn't reach.
- **Zero-lib-change posture.** v1.20 was zero-lib-change by design; v1.21 must charter this one resolver change as an explicit, scoped exception (adopter-invisible, no semver bump). This is the **one escalation-worthy call** in the area and should be surfaced in the milestone charter for a one-line confirmation.
- **Adopter-first Repo ownership** (PROJECT.md:445, 468-469) is preserved — adopters still own `:rindle, :repo`; the process override is a test-only seam they never touch.

### Requirement-ready bullets (atomic, testable)

- **ISO-01** — `Rindle.Config.repo/0` resolves a per-process override (`Process.get({Rindle.Config, :repo_override})`) before falling back to `Application.get_env(:rindle, :repo, Rindle.Repo)`; the resolver walks the `:"$callers"` chain so `Task`-spawned and inline-Oban work inherit the override. Production default branch is byte-for-byte unchanged when no override is set.
- **ISO-02** — Expose test-only setters `Rindle.Config.put_repo_override/1` and `delete_repo_override/0` (process-dictionary writes only).
- **ISO-03** — `Rindle.Test.CountingFailingTxnRepo.with_counting_repo/2` sets the repo via `put_repo_override/1` in the calling process and clears it in `after`; it performs **no** `Application.put_env(:rindle, :repo, …)`. (The `:counting_failing_txn_repo` fail-after config likewise moves to the process dictionary.)
- **ISO-04** — Revert the defensive `async: false` demotions made for this reason: restore `Rindle.Delivery.StreamingDispatchTest` to `async: true` (and any other module demoted solely due to repo-swap pollution), retaining the comment trail.
- **ISO-05** — Strengthen `test/async_safety_guard_test.exs` with a rule (`:global_repo_swap`) that flags `Application.put_env`/`delete_env` of `:rindle, :repo` in **any** test module regardless of its async flag, with a failure message pointing at `Config.put_repo_override/1`. (Hoist the relevant scan to all modules, not just `async: true`.)
- **ISO-06 (PROOF)** — Add a regression test that runs the counting double **concurrently** with an async repo reader and proves no cross-pollution: in process A, `with_counting_repo(1, fn -> ... end)` force-failing a transaction; in a *separate* spawned process B (no override in its dictionary, no `$callers` link to A), assert `Rindle.Config.repo() == Rindle.Repo` and that B's `transaction/1` succeeds while A's first transaction is force-failed. The test must fail against the old global-`put_env` implementation and pass against the process-scoped one.
- **ISO-07 (charter gate)** — Milestone charter explicitly authorizes the single adopter-invisible `lib/rindle/config.ex` change as a scoped exception to the v1.20 zero-lib-change invariant (no public-API change, no semver bump).

### How to PROVE it

ISO-06 is the executable proof. Sketch:

```elixir
test "process-scoped override does not pollute a concurrent reader" do
  parent = self()
  # Reader in an unrelated process tree (no override, not a $callers descendant of the mutator).
  reader = Task.async(fn ->
    receive do :go -> :ok end
    {Rindle.Config.repo(), Rindle.Config.repo().transaction(fn -> :ok end)}
  end)

  CountingFailingTxnRepo.with_counting_repo(1, fn ->
    # In THIS process, the override is active and the 1st txn force-fails.
    assert Rindle.Config.repo() == Rindle.Test.CountingFailingTxnRepo
    assert {:error, :plan, _, _} = Rindle.Config.repo().transaction(fn -> :ok end)
    send(reader.pid, :go)
    # The reader runs concurrently inside the window:
    assert {Rindle.Repo, {:ok, :ok}} = Task.await(reader)   # ← unpolluted
  end)
end
```

Against today's global `put_env`, the reader sees `Rindle.Test.CountingFailingTxnRepo` and its transaction force-fails → red. Against the process-scoped resolver, the reader sees `Rindle.Repo` and succeeds → green. That delta is the proof the milestone delivers.
