**Date:** 2026-05-06
**Confidence:** HIGH on protocol mechanics, library mappings, peer-library lessons, and Rindle seam fit. MEDIUM on adopter-pull justification (no concrete GCS adopter has filed a request; this remains a deferred carry-over from v1.1). ## 1. Locked scope (additive contract)

v1.6 ships **one new storage adapter** (`Rindle.Storage.GCS`) and **one new upload-flow family** (resumable). Everything is additive:

**In:**
- `Rindle.Storage.GCS` adapter implementing the existing `Rindle.Storage` behaviour, plus four new resumable callbacks added to that behaviour as `@optional_callbacks`.
- Two new capability atoms become **shipped** (no longer reserved): `:resumable_upload` and `:resumable_upload_session`. (Vocabulary already reserved in `lib/rindle/storage/capabilities.ex`.)
- New broker entrypoints: `initiate_resumable_session/2`, `resumable_session_status/2`, `cancel_resumable_session/2`. Existing `verify_completion/2` is the convergence point — no new completion verb.
- Additive `MediaUploadSession` columns: `session_uri` (string, encrypted at rest via `Cloak.Ecto.Binary` recommendation, but adopter-controlled), `session_uri_expires_at` (utc_datetime_usec), `last_known_offset` (bigint, default 0), `region_hint` (string, nullable). New `upload_strategy` value `"resumable"`.
- Oban worker updates to `Rindle.Workers.AbortIncompleteUploads` / `Rindle.Ops.UploadMaintenance` to call `cancel_resumable_session/2` before deleting GCS-strategy session rows.
- A standalone GCS proof lane in CI (Wire-format integration test against the real `storage.googleapis.com` endpoint behind a service-account-credentialed lane; no in-process emulator required for the first ship — fakegcs / Bypass for unit-level fixtures only).
- Docs: new `guides/storage_gcs.md`, capability matrix update in `guides/storage_capabilities.md`, updated `guides/troubleshooting.md` for resumable failures, `mix rindle.doctor` GCS check.

**Out:**
- tus protocol — separate candidate.
- Provider/streaming adapters (Mux, CF Stream, Transloadit) — separate candidate.
- Auto-fallback resumable→PUT or PUT→resumable.
- Generic "unified resumable" abstraction across S3 multipart and GCS resumable.
- Per-request authorizer hook for resume calls beyond what the adopter already does at `initiate_resumable_session/2`.
- A bespoke `Rindle.Storage.R2` or any new S3 sibling.
- HTML form-POST upload paths.
- Customer-Managed Encryption Keys (CMEK), Object Versioning, Object Lifecycle policies — adopter-bucket concerns, not adapter concerns.
- IAM SignBlob mode in v1.6 (deferred — see §13).

**Hard rule:** the existing presigned PUT and S3 multipart contracts ship unchanged. No adapter is forced to advertise resumable atoms. `Rindle.Storage.S3.capabilities/0` does **not** add resumable atoms — S3 multipart is not resumable in this vocabulary.

## 2. Phase breakdown (concrete, with names + REQ-IDs + success criteria)

Five phases. Each ends on a runnable success criterion that doubles as the verification gate.

**Phase 33 — GCS Adapter Foundation (`GCS-01..GCS-04`)**
- `GCS-01`: `Rindle.Storage.GCS` implements `store/3`, `download/3`, `delete/2`, `head/2`, `url/2` against the real GCS bucket using `Goth` for auth and `Finch` (already a Goth transitive dep) for HTTP. No resumable behavior yet.
- `GCS-02`: `Rindle.Storage.GCS.capabilities/0` returns `[:signed_url, :head]` only at end of phase (resumable atoms added in Phase 35).
- `GCS-03`: V4 signed URL generation via `gcs_signed_url ~> 0.4.6` with private-key auth mode; signed-URL TTL respects `Rindle.Config.signed_url_ttl_seconds/0`. Document that GCS V4 signed URLs do **not** safely enforce `response-content-disposition` / `response-content-type` (the Active Storage lesson — §10), so disposition/type goes into object metadata at `store/3`.
- `GCS-04`: GCS proof lane in CI gated behind `GOOGLE_APPLICATION_CREDENTIALS_JSON` secret; runs on PR only when secret is present, runs on release always (mirrors current MinIO lane discipline).
- **Success:** `mix test --only gcs` passes against a real bucket; `mix rindle.doctor` reports GCS adapter health when configured; image-only profile can `Rindle.upload/3` to GCS using local-file ingest path; signed delivery URL works.

**Phase 34 — Resumable Session Persistence + FSM (`RESUMABLE-01..RESUMABLE-03`)**
- `RESUMABLE-01`: Migration adds `session_uri`, `session_uri_expires_at`, `last_known_offset`, `region_hint` to `media_upload_sessions`. All columns nullable; `upload_strategy` allowed values widen to `["presigned_put", "multipart", "resumable"]`. Migration is reversible and ships in `priv/repo/migrations`.
- `RESUMABLE-02`: `Rindle.Domain.MediaUploadSession.changeset/2` casts the new fields. `Rindle.Domain.UploadSessionFSM` gains state `"resuming"` between `"signed"` and `"uploading"`, with allowed transitions: `"signed" → "resuming" → "uploading"` (existing `"uploading" → "uploaded" → "verifying" → "completed"` lane unchanged). Cancel path: any non-terminal state → `"aborted"`.
- `RESUMABLE-03`: Telemetry — `[:rindle, :upload, :resumable, :status]`, `[:rindle, :upload, :resumable, :cancel]` events. `session_uri` is **never** in telemetry metadata. Logger metadata `:session_uri` is filtered globally via a recommended `Logger.add_translator` snippet documented in `guides/storage_gcs.md`; the adapter itself never logs the URI.
- **Success:** new state column reachable; FSM tests pass; existing presigned-PUT and multipart proofs unchanged; doctor checks new columns exist.

**Phase 35 — Resumable Adapter Behaviour + Broker Wiring (`RESUMABLE-04..RESUMABLE-08`)**
- `RESUMABLE-04`: `Rindle.Storage` behaviour adds four `@optional_callbacks`: `initiate_resumable_upload/3`, `resumable_upload_status/3`, `cancel_resumable_upload/3`, `verify_resumable_completion/3`. (See §4 for arities and shapes.)
- `RESUMABLE-05`: `Rindle.Storage.GCS` implements all four. `capabilities/0` becomes `[:signed_url, :head, :resumable_upload, :resumable_upload_session]`.
- `RESUMABLE-06`: `Rindle.Upload.Broker.initiate_resumable_session/2` → `resumable_session_status/2` → `cancel_resumable_session/2` → existing `verify_completion/2`. The `verify_completion/2` flow is unchanged from S3's: it calls `head/2` on the storage key. GCS resumable completion writes the final object at the storage key, so `head/2` is the same trust gate.
- `RESUMABLE-07`: `Rindle.Storage.S3.capabilities/0` and `Rindle.Storage.Local.capabilities/0` confirmed not to advertise resumable atoms. Calling resumable broker entrypoints against a non-resumable adapter returns `{:error, {:upload_unsupported, :resumable_upload_session}}`.
- `RESUMABLE-08`: All locked error atoms in §4 are returnable from real adapter paths; tests exercise each.
- **Success:** end-to-end resumable upload to a real GCS bucket from a test fixture (chunked PATCH from a Finch streamed body), verified through `verify_completion/2` into a promoted asset.

**Phase 36 — Maintenance + Cancel Contract (`RESUMABLE-09..RESUMABLE-11`)**
- `RESUMABLE-09`: `Rindle.Ops.UploadMaintenance.abort_incomplete_uploads/1` calls `cancel_resumable_upload/3` on the adapter when `upload_strategy = "resumable"`, treating `{:error, :not_found}` and `{:error, :session_expired}` as idempotent success. Counters in the report distinguish `:resumable_aborts` from `:multipart_aborts`.
- `RESUMABLE-10`: Local row deletion **only** after remote cancel returns success-or-idempotent. Remote-failure path leaves the row in `"aborted"` (terminal in FSM) with `failure_reason` populated, available to a retry via existing Oban backoff.
- `RESUMABLE-11`: `mix rindle.runtime_status` surfaces `resumable_sessions_pending`, `resumable_sessions_expired`, and (count, not URI) of stale session URIs past `session_uri_expires_at`.
- **Success:** real GCS proof lane covers initiate→cancel→cleanup and initiate→expire→cleanup; orphaned sessions visible in `runtime_status`.

**Phase 37 — Onboarding, Docs, Doctor (`RESUMABLE-12..RESUMABLE-14`)**
- `RESUMABLE-12`: `guides/storage_gcs.md` written maintainer-to-maintainer: bucket setup, service-account JSON wiring, CORS rules for direct-from-browser PATCH (the GCS-specific gotcha — see §10 Uppy lesson), one-week expiry, region-pin cost, "session URI is a bearer token" callout.
- `RESUMABLE-13`: `mix rindle.doctor` adds GCS-aware checks (Goth running, bucket reachable, signed-URL key configured, CORS suspected if profile uses GCS+resumable). Profile-aware so image-only S3 adopters see no new noise.
- `RESUMABLE-14`: Package-consumer proof lane — fresh `mix phx.new` adopter installs Rindle, configures a GCS profile, runs the existing canonical adopter lifecycle test against a real bucket (gated by secret).
- **Success:** package-consumer GCS lane green; capability matrix updated; doctor returns clean on a happy-path GCS install.

## 3. Idiomatic Elixir/Phoenix/Ecto/Plug fit (with pinned hex versions)

**Auth — `goth ~> 1.4`** (pin `~> 1.4.5`). Uses `Finch` internally. Started in adopter app's supervision tree as `{Goth, name: MyApp.Goth, source: {:service_account, creds}}`. Rindle does **not** start Goth itself; the adapter looks up a configurable Goth name (`config :rindle, Rindle.Storage.GCS, goth: MyApp.Goth`). This matches the existing "adopter owns the runtime" posture (Repo, Oban, Goth all adopter-supervised).

**HTTP — `finch ~> 0.21`** (already transitive via Goth). Direct adapter HTTP via `Finch.build(:put, url, headers, stream_or_iodata) |> Finch.request(MyApp.Finch)`. Adopter supplies a Finch instance name; default is `Rindle.Finch` if started by the adopter. **Not Tesla, not Req for the adapter itself**: Finch streams request bodies, supports `Content-Range` headers natively, and is the lowest common runtime denominator — Goth already pulls it in. Req is an excellent test harness but is opinionated and adds ~2 transitive deps; the adapter's hot path stays on Finch.

Verified live: Finch 0.21.0 (2026-01-22), Req 0.5.17 (2026-01-05), Tesla 1.17.0 (2026-04-18), Goth 1.4.5 (2024-12-20).

**Signing — `gcs_signed_url ~> 0.4.6`** (pin `~> 0.4.6`). V4 only. Two transitive deps (`jose`, `jason` — already in tree via Goth/Jason). V2 is legacy per Google docs; v1.6 ships V4-only. Private-key auth in v1.6; IAM SignBlob mode is deferred (see §13).

**Client SDK — explicitly NOT `google_api_storage`.** v0.46.1 (2025-04-15) is auto-generated, Tesla-coupled, and exposes the resumable session as `storage_objects_insert_resumable/5` returning `{:ok, nil}` — it doesn't even cleanly surface the session URI (per the live HexDocs check). For an adapter that needs raw control of session URI, status query semantics, and PATCH-style uploads with `Content-Range`, hand-rolled `Finch` over the JSON API endpoint (`https://storage.googleapis.com/upload/storage/v1/b/$BUCKET/o?uploadType=resumable`) is *less* code, *more* control, and avoids a 200+ module Tesla-dependent transitive install for an adapter we own. **Locked: hand-rolled JSON API client over Finch.**

**Ecto — additive migration only.** Reuses adopter Repo (the v1.1 invariant). New columns are nullable; existing rows unaffected.

**Oban — same patterns as the multipart cleanup lane.** No new worker needed; `AbortIncompleteUploads` learns to dispatch on `upload_strategy`.

**Plug/Phoenix — zero new HTTP routes in core.** GCS resumable PATCHes go directly from the browser/client to `storage.googleapis.com`, not through Phoenix. This is the same posture as presigned PUT and S3 multipart. CORS is a bucket configuration concern, documented in `guides/storage_gcs.md`. **No mountable router.** Adopter routes for `initiate/status/cancel` are adopter-owned (consistent with current presigned PUT controllers in adopter apps).

**Telemetry — additive.** New events `[:rindle, :upload, :resumable, :start | :status | :cancel | :stop]` modelled on the existing `[:rindle, :upload, :start | :stop]` shape. **Hard rule: `session_uri` is never in telemetry metadata, never in logs, never in `inspect/2` output.** A custom `Inspect` impl on `MediaUploadSession` redacts `session_uri` to `"[REDACTED]"` when present.

## 4. Public API shape (locked function arities, return shapes, error atoms)

**Behaviour additions to `Rindle.Storage`** (all `@optional_callbacks`):

```elixir
@type resumable_init_result :: %{
        required(:session_uri) => String.t(),
        required(:upload_id) => String.t(),
        required(:expires_at) => DateTime.t(),
        optional(:region_hint) => String.t() | nil,
        optional(atom()) => term()
      }

@type resumable_status_result :: %{
        required(:committed_bytes) => non_neg_integer(),
        required(:state) => :in_progress | :complete | :expired,
        optional(atom()) => term()
      }

@callback initiate_resumable_upload(key :: String.t(), expected_size :: pos_integer() | nil, opts :: keyword()) ::
            {:ok, resumable_init_result()} | {:error, term()}

@callback resumable_upload_status(key :: String.t(), session_uri :: String.t(), opts :: keyword()) ::
            {:ok, resumable_status_result()} | {:error, term()}

@callback cancel_resumable_upload(key :: String.t(), session_uri :: String.t(), opts :: keyword()) ::
            {:ok, %{cancelled: boolean()}} | {:error, term()}

@callback verify_resumable_completion(key :: String.t(), session_uri :: String.t(), opts :: keyword()) ::
            {:ok, head_result()} | {:error, term()}
```

**Broker public API:**

```elixir
@spec initiate_resumable_session(profile_module :: module(), opts :: keyword()) ::
        {:ok, %{session: MediaUploadSession.t(),
                resumable: %{session_uri: String.t(),
                             upload_id: String.t(),
                             expires_at: DateTime.t()}}}
        | {:error, term()}

@spec resumable_session_status(session_id :: binary(), opts :: keyword()) ::
        {:ok, %{session: MediaUploadSession.t(),
                committed_bytes: non_neg_integer(),
                state: :in_progress | :complete | :expired}}
        | {:error, term()}

@spec cancel_resumable_session(session_id :: binary(), opts :: keyword()) ::
        {:ok, %{session: MediaUploadSession.t()}}
        | {:error, term()}
```

`Rindle.Upload.Broker.verify_completion/2` is unchanged. After client finishes streaming PATCHes to the session URI, adopter calls `verify_completion/2` like any other upload; it calls the adapter's `head/2`, finds the object, and promotes.

**Locked error atoms:**
- `{:error, {:upload_unsupported, :resumable_upload}}` — adapter doesn't advertise capability.
- `{:error, {:upload_unsupported, :resumable_upload_session}}` — caller invoked status/cancel against a non-resumable adapter or a non-resumable session row.
- `{:error, :session_uri_expired}` — GCS returned 410 Gone or session_uri_expires_at elapsed.
- `{:error, :session_uri_unknown}` — GCS returned 404 (URI never existed or > 1 week old).
- `{:error, {:offset_mismatch, %{server: integer(), client: integer()}}}` — status query / completion offset disagrees.
- `{:error, :region_pinned_initiation}` — guarded log/telemetry warning, not a failure (region pin is informational; initiation succeeds, but the broker emits a telemetry warning and stamps `region_hint` for adopter visibility).
- `{:error, {:gcs_http_error, %{status: integer(), body: binary() | nil}}}` — generic transport fallthrough.
- `{:error, :goth_unconfigured}` — adapter cannot resolve a Goth instance.
- `{:error, :missing_bucket}` — same shape as the existing S3 adapter.
- `{:error, :storage_object_missing}` — reused from existing broker contract for `verify_completion/2`.

These atoms are public; they ship in the docs and `Rindle.Error.t()` mapping.

## 5. Capability vocabulary changes

`Rindle.Storage.Capabilities.@known` already lists `:resumable_upload` and `:resumable_upload_session`. v1.6 promotes them from "reserved" to "shipped":

| Capability | Meaning (v1.6) |
| ---------- | -------------- |
| `:resumable_upload` | Adapter implements `initiate_resumable_upload/3` plus `verify_resumable_completion/3`. |
| `:resumable_upload_session` | Adapter additionally implements `resumable_upload_status/3` and `cancel_resumable_upload/3` (the operational surface). |

Adapters MAY advertise `:resumable_upload` alone (initiate + finalize, no observability), but the broker's `resumable_session_status/2` and `cancel_resumable_session/2` require `:resumable_upload_session`. Rindle's GCS adapter advertises **both**.

`Rindle.Storage.S3` and `Rindle.Storage.Local` advertise **neither**. There is no silent reuse of S3 multipart as "resumable."

## 6. Migration / Ecto shape (additive)

```elixir
defmodule Rindle.Repo.Migrations.AddResumableUploadColumns do
  use Ecto.Migration

  def change do
    alter table(:media_upload_sessions) do
      add :session_uri, :text
      add :session_uri_expires_at, :utc_datetime_usec
      add :last_known_offset, :bigint, default: 0, null: false
      add :region_hint, :string, size: 64
    end

    create index(:media_upload_sessions, [:session_uri_expires_at],
             where: "upload_strategy = 'resumable'",
             name: :media_upload_sessions_resumable_expiry_idx)
  end
end
```

- All new columns nullable; default `last_known_offset = 0`.
- Partial index supports the maintenance worker's expiry sweep without bloating the existing presigned-PUT/multipart hot paths.
- `session_uri` is `text` (not `string`); a GCS resumable session URI can exceed 255 chars.
- The adopter migration is generated by an existing migration generator pattern; v1.6 just ships a new templated migration file. Reversible.
- **Optional encryption-at-rest:** documented but not enforced. Adopters using `cloak_ecto` can wrap `session_uri` as `Cloak.Ecto.Binary`. Rindle does not pull in `cloak` as a dep.

## 7. Oban / maintenance / cleanup contract

Reuses the existing two-step lane (`AbortIncompleteUploads` then `CleanupOrphans`).

- `AbortIncompleteUploads` queries sessions where `state IN ("signed", "uploading", "resuming")` AND `expires_at < now()`. For `upload_strategy = "resumable"`, it dispatches `cancel_resumable_upload/3` on the adapter. Idempotency: `{:error, :session_uri_unknown}` and `{:error, :session_uri_expired}` count as success (the remote is gone or never existed).
- `CleanupOrphans` deletes rows in `state = "expired"`. For resumable rows, the row is **not** deleted unless the adapter returned `:ok` or an idempotent error from cancel. Failures leave the row in `"aborted"` with `failure_reason` populated; Oban backoff retries.
- Telemetry: `[:rindle, :cleanup, :run]` measurement gains `:resumable_aborts` count.
- `mix rindle.abort_incomplete_uploads` and `mix rindle.cleanup_orphans` learn the new path automatically (they delegate to the Ops module).

## 8. DX / UX / principle-of-least-surprise rules

1. **No silent fallback.** Asking for `initiate_resumable_session/2` against `Rindle.Storage.S3` returns `{:error, {:upload_unsupported, :resumable_upload}}`. Asking for `sign_url/2` (presigned PUT) against `Rindle.Storage.GCS` succeeds — GCS supports V4-signed PUT. Adopters choose the family explicitly.
2. **One trust gate.** Completion goes through `verify_completion/2` exactly as for presigned PUT and multipart. No new "resumable_complete" verb.
3. **Session URI is a bearer credential.** Documented loudly in `guides/storage_gcs.md`. Never logged. Never in `inspect/2`. Never in telemetry. Custom `Inspect` impl on `MediaUploadSession` redacts.
4. **Region pin is honest, not silent.** Initiation that fires from a region different from the configured bucket region emits a telemetry `:region_pinned_initiation` event with `region_hint`. Adopters can dashboard this. No automatic re-routing.
5. **One-week expiry is honest.** `session_uri_expires_at` is set to `initiation_time + 7 days` by the adapter and surfaced in the broker's `:initiate_resumable_session` return. Adopters can build their own UX around this.
6. **Capability matrix in the doctor output.** `mix rindle.doctor` shows per-profile capability advertisement so adopters can confirm their GCS profile actually has `:resumable_upload_session` and not just `:signed_url`.
7. **Disposition/type lives in object metadata at `store/3`.** Active Storage's lesson: GCS V4 signed URLs do not safely enforce `response-content-disposition` and `response-content-type`. The adapter writes `Content-Disposition` and `Content-Type` as `x-goog-meta-*` (the bucket-side fields, not the URL params) on `store/3` and on resumable initiation.
8. **Configuration shape is keyword-by-default with module-name keying** (matches existing S3 pattern):

   ```elixir
   config :rindle, Rindle.Storage.GCS,
     bucket: "my-bucket",
     goth: MyApp.Goth,                      # Goth instance name (required)
     finch: MyApp.Finch,                    # Finch instance name (required)
     signing_key: %{...},                   # service-account JSON or PEM (required for V4 signing)
     signed_url_ttl: 3600,                  # seconds; falls back to Rindle.Config.signed_url_ttl_seconds/0
     resumable_chunk_size: 8 * 1024 * 1024, # client-side hint, multiple of 256 KiB
     region_hint: "us-central1"             # informational, used in telemetry warnings
   ```

9. **CORS is the adopter's bucket concern.** `guides/storage_gcs.md` ships a copy-pasteable `gsutil cors set` recipe with the `PATCH`/`PUT` methods, `Content-Range` and `x-goog-resumable` allowed headers, and the appropriate origin allowlist.

## 9. Security invariants check (against PROJECT.md numbered list)

1. **Never trust client MIME/filename.** Held — `verify_completion/2` still does magic-byte sniffing post-`head/2` like the existing flow.
2. **Don't attach/process direct uploads until completion is verified.** Held — `head/2` after resumable finalize is the same gate.
3. **No unbounded variant explosion.** Untouched — variant logic is downstream of upload.
4. **Storage side effects not in DB transactions.** Held — `initiate_resumable_session/2` mirrors `initiate_multipart_session/2`: GCS HTTP call before DB transaction; compensating cancel on persist failure (mirrors `compensate_failed_multipart_persist/4` exactly).
5. **Purge async, idempotent, auditable.** Held — `cancel_resumable_upload/3` returns idempotent on 404/410.
6. **Concurrent replacement races resolve safely.** Held — storage-key generation is unchanged (`Rindle.Security.StorageKey.generate/3` produces a unique key per asset).
7. **Stale/failed states visible.** Held — `runtime_status` surfaces resumable-specific counters.
8–13. **AV invariants (FFmpeg argv, protocol whitelist, container metadata, HLS exclusion, FFmpeg minimum, temp roots).** Untouched — GCS adapter doesn't shell out, doesn't ingest manifests, doesn't transcode.

**New invariant added by v1.6 (must hold):** "Session URIs returned by resumable initiation are bearer credentials. They must never be logged, telemetry-tagged, inspected, or persisted unencrypted in shared logs. The adapter and broker enforce this; adopters are responsible for not exposing them in their own logs/UI."

## 10. Lessons from peer libs — DO / AVOID per peer

**Rails Active Storage `GCSService`** ([source](https://github.com/rails/rails/blob/main/activestorage/lib/active_storage/service/gcs_service.rb))
- DO: store `content_type` and `content_disposition` as object metadata, not URL params — GCS V4 signed URLs do not safely enforce `response-*` query params (per ActiveStorage commit history). Rindle does this in `store/3` and on resumable init.
- AVOID: ActiveStorage's `direct_upload` posture for GCS does not implement resumable session lifecycle observability — the client is on its own. Rindle ships `status/cancel` because adopters running into network failures need them.

**Shrine + `shrine-google_cloud_storage`** ([source](https://github.com/janko/shrine-google_cloud_storage))
- DO: separate plugins for S3-multipart vs GCS-resumable. Shrine's S3 multipart is a different plugin from its tus plugin. Rindle mirrors: distinct family, distinct API, distinct capability atoms.
- AVOID: Shrine's GCS adapter uses `uploadType=multipart` (the JSON multipart, not resumable) for normal flows and does **not** expose a resumable-session API to adopters — adopters wanting resume have to drop to the Ruby SDK. Rindle exposes resumable as the *primary* GCS direct-upload story because that's what GCS uniquely offers; v4 signed PUT is the secondary.

**CarrierWave Google Cloud Storage** (`carrierwave-google-storage`)
- DO: explicit `gcloud` credential discovery (env, file, ADC) rather than reinventing a config schema. Rindle mirrors via Goth.
- AVOID: CarrierWave's GCS plugin lacks any session lifecycle visibility; abandoned uploads silently consume bucket budget. Rindle's `runtime_status` resumable counters exist because of this exact pain point.

**django-storages `gcloud` backend** ([source](https://github.com/jschneier/django-storages/blob/master/storages/backends/gcloud.py))
- DO: validate `LOCATION` paths against directory traversal at the adapter layer (django-storages uses `safe_join`). Rindle's `Rindle.Security.StorageKey` already does this; the GCS adapter inherits it.
- AVOID: django-storages defaults to `mimetypes.guess_type()` for content type — guessing-based detection is wrong for security-sensitive files. Rindle never trusts client-reported MIME and re-sniffs at verify time.

**Laravel Filesystem + Spatie Media Library GCS adapter**
- DO: Spatie surfaces "media collections" as first-class operator-queryable rows — Rindle's `media_assets` already does this and the GCS path doesn't change it.
- AVOID: Laravel's Flysystem GCS adapter has no first-class resumable surface — adopters who hit large uploads end up building their own retry harness. Rindle ships the contract.

**Node `@google-cloud/storage` `resumable-upload.ts`** ([source](https://github.com/googleapis/nodejs-storage/blob/main/src/resumable-upload.ts))
- DO: default chunk size ≥ 8 MiB, multiple of 256 KiB, supports CRC32C/MD5 client-supplied checksums, treats 308 Resume Incomplete with `Range: bytes=0-N` as the canonical "how many bytes have you got?" semantics. Rindle's `resumable_chunk_size: 8 * 1024 * 1024` default mirrors this; status_result returns `committed_bytes` parsed from the `Range` response header.
- AVOID: nodejs-storage caches session URIs in client-side files for retry — that's a desktop-CLI pattern and a security hazard for multi-tenant servers. Rindle persists the URI in the DB row tied to the asset's lifecycle and treats it as a bearer credential (encrypted-at-rest recommended).

**Uppy `@uppy/google-drive` and Companion**
- DO: Uppy's GCS direct-upload posture relies on browser CORS being configured for `PATCH`, `PUT`, `Content-Range`, and `x-goog-resumable`. Document this loudly. Rindle ships the gsutil snippet.
- AVOID: Uppy's older Companion relayed bytes through a server proxy; modern flows go direct browser→GCS. Rindle's broker never proxies bytes — same posture as presigned PUT and S3 multipart.

**tus + S3 multipart (contrast only)**
- DO (tus): explicit offset/status/termination semantics. Rindle mirrors with `committed_bytes` / `state` / `cancel`.
- AVOID (S3 multipart): conflating "multipart parts assembled at completion" (S3) with "single object grown by appending offsets" (GCS). They are different protocols with different failure modes. Keeping them as separate capability families is the lesson.

## 11. Pros / cons / risks specific to Rindle

**Pros**
- Closes the v1.1 deferred gap with a bounded, well-shaped scope. Fits the existing capability seam exactly.
- Reuses every Rindle invariant: adopter-owned Repo, Oban-required cleanup, capability honesty, `verify_completion/2` as the trust gate, no storage I/O inside DB transactions, async idempotent cleanup.
- Adds genuine resumable semantics — not just another presigned-PUT clone with a different header — which is the main reason adopters would adopt GCS over R2 (R2 is already supported via S3 path).
- Public API surface delta is small: 3 broker functions, 4 adapter callbacks, 2 promoted capability atoms, 4 new DB columns. Each is independently testable.
- Aligns with the v1.4 lesson that "narrow public surface ships cleanly": the existing milestone retros (v1.2/v1.3/v1.4) confirm Rindle ships best when the milestone is one tight expansion.

**Cons / risks**
- The session URI is a bearer credential. If logged, leaked to clients beyond the uploader, or persisted unencrypted in operator dashboards, it becomes an "anyone can write to your bucket" vector. Mitigation: Inspect-redaction, doc-loud warnings, recommend `cloak_ecto` for at-rest encryption. Risk remains MEDIUM because adopters can still misuse it.
- One-week session expiry is foreign to S3 mental models. Adopters used to multipart's "abort-when-you-want" semantics may be surprised when 8-day-old sessions return 404 instead of 410. Mitigation: doctor + `runtime_status` surface this; docs explain.
- Region-pin cost is silent in GCS itself. We make it visible via telemetry; adopters still need to understand multi-region implications.
- New CI proof lane needs a real GCS bucket with a service account. Adds a secret to the release lane. Mitigation: gate behind `GOOGLE_APPLICATION_CREDENTIALS_JSON` secret; PRs without secret skip the lane (mirrors existing MinIO discipline).
- Hand-rolled JSON API client over Finch is ~250 lines we own. Acceptable: it's narrow, the protocol is stable (the JSON resumable API has not changed in years), and it avoids the Tesla transitive surface of `google_api_storage`.
- Adopter pull is unproven. v1.5 retro and current PROJECT.md note "no concrete adopter has filed for this." This is the only meaningful blocker to "go now."

## 13. Open decisions deliberately deferred (with reason)

- **IAM SignBlob auth mode** — `gcs_signed_url` supports it; service-account-impersonation is a real pattern in GKE/Cloud Run. v1.6 ships private-key-only because it covers the dominant adopter case (downloaded service-account JSON) and keeps the surface narrow. v1.7+ can add SignBlob behind a config flag without breaking 1.6 adopters.
- **Customer-supplied session URI** — adopters wanting to pre-sign a session URI on their own and hand it to clients instead of going through `initiate_resumable_session/2`. Possible but enlarges the auth model. Defer until requested.
- **Object Versioning, Object Lifecycle, CMEK** — bucket-level concerns, not adapter concerns. Adopter configures these in GCP console; Rindle's `head/2` will see whichever generation GCS returns.
- **Companion-style server-side proxy uploads** — explicitly out. Browser→GCS direct only, mirroring presigned PUT and S3 multipart posture.
- **Encryption-at-rest enforcement of `session_uri`** — documented (`cloak_ecto` recipe in `guides/storage_gcs.md`) but not enforced. Forcing a `cloak` dep would violate the lifecycle-library boundary.
- **`Rindle.Storage.GCSResumable` vs `Rindle.Storage.GCS`** — locked to `Rindle.Storage.GCS`. One adapter, multiple capabilities. Splitting "resumable" into its own adapter would imply non-resumable GCS is a separate runtime install, which it is not.
- **tus and provider/streaming work** — explicitly separate candidates. The synthesizer compares them; this memo does not.
