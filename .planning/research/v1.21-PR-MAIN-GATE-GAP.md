# v1.21 Research Area 3 — PR↔MAIN Gate-Coverage Gap

**Researched:** 2026-06-26
**Mode:** Feasibility + ecosystem comparison → ONE locked topology change
**Confidence:** HIGH (repo-grounded: every lane, wall-clock, and coupling read from source)

> **Problem in one line:** v1.20 (Phase 106) deliberately moved four lanes to `push:main`-only
> for the 15→≤7-min PR cut. The cost is a *gate-coverage gap*: a PR is fully green, then `main`
> goes RED on first run of those lanes after merge. This is exactly how the 2026-06-26 flake
> cluster reached `main`. This area closes the **regression-class** half of that gap without
> giving back the wall-clock win — and is explicitly **ordering-dependent** on the flake-kill
> areas (1/2/4) landing first.

### 1b. What stays ON the PR gate today (`CI Summary.needs`, 106 D-02)

`quality` (1.15/26 + 1.17/27), `optional-dependencies`, `integration` (60s/96s, MinIO-local),
`contract` (38s/61s), `proof` (36s/58s), `adopter` (58s/105s), **`adoption-demo-unit`**

**Measured PR p95 ≈ 7.2 min / avg ≈ 5 min** (106 D-02). The long pole is the image-smoke chain.

### 1c. The gap, precisely

`CI Summary` (the SOLE required check, Phase 105) lists in `needs:` only PR-side lanes. The four
above are **omitted from `needs:`** (106 D-09) — by design, so the skip-as-pass gate makes *no
claim* about them. Release readiness for the full matrix is proven separately by the **push:main
`ci.yml` run conclusion** that `release.yml gate-ci-green` polls (106 D-11). So:

- A PR can be green while a **render regression** (E2E class) or **cold-start break** (cohort class)
  sits latent. MTTD = 1 merge; it blocks the *next* author, not the introducer.
- The 2026-06-26 cluster (per MEMORY `reference_ci_lanes_only_on_main`) was *pre-existing flakes*
  in these main-only lanes that PR CI never exercised — i.e. the gap let a **flake**, not a
  regression, surface on main. **This is the crucial distinction for §7's ordering dependency.**

### 1d. Decisive de-risking finding (changes the calculus)

`examples/adoption_demo` Playwright config has **one project: `chromium`** (`playwright.config.js:29`),
and **the E2E suite is NOT a pixel-diff oracle** — `admin-screenshots.spec.js:3` states *"Screenshot
PNGs are audit/reference artifacts. The blocking signal is the computed polish/backstop assertions"*
(computed-style `assertAdminPolish` etc.). Only **1 of ~22 specs** even calls `toHaveScreenshot`, and
its assertions are computed-style, not pixel comparison. **This means the E2E lane's flakiness is
font/timing/boot variance, not the notoriously-flaky pixel-diff class** — materially safer to gate a
*lean subset* on PR than a generic "Playwright lane" reputation suggests.

## 2. Options — Per-Lane Shift-Left Decision

For each main-only lane: the mechanism choices are **(a) add lean variant to PR gate**,
**(b) `merge_group` (merge-queue)**, **(c) `paths:`-filter onto PR**, **(d) keep main-only**.

### Lane A — `adoption-demo-e2e` (the prime candidate; MEMORY explicitly recommends gating it)

| Option | Pros | Cons / tradeoff | PR wall-clock impact |
|---|---|---|---|
| **(a) Add a LEAN smoke variant to PR** ✅ **RECOMMENDED** | Catches the render-regression class pre-merge (PR-introducible); reuses pinned container; smoke + admin-console specs are deterministic (no pixel-diff) | Imports the lane's boot/font flake *unless areas 1/2/4 land first*; adds one chain to PR | **+~90–150s as a NEW PARALLEL chain** — does NOT extend the long pole if it finishes under the image-smoke chain (~414s p95). Net p95 ≈ unchanged. See §7 proof. |
| (b) `merge_group` | Full E2E at merge time, zero per-PR cost | New branch-protection surface right after the 105 single-required-check flip; solo-maintainer overhead; deferred in 106 | 0 on PR (runs at queue) |
| (c) `paths:`-filter | Only run when admin/template files change | Brittle path lists; misses cross-cutting changes; representative-drift risk | conditional |
| (d) keep main-only | No PR cost | The status-quo gap; the documented failure | 0 |

**Decision: (a) — a lean `adoption-demo-e2e-smoke` on PR.** The full 22-spec + screenshot matrix
**stays on push:main** (Lane A-full keeps its `!= pull_request` guard). The PR variant runs ONLY the
deterministic subset: `smoke.spec.js` (cold-boot homepage, 18 lines) + `admin-console.spec.js`
(mount/shell/no-raw-secrets, 109 lines), **excluding `admin-screenshots.spec.js`** (the
artifact/heavy spec). This is the "demo-unit vs full-e2e split that already half-exists" extended one
notch — `adoption-demo-unit` proves browser-free render; this proves *browser* render of the same
class, leanly.

### Lane B — `cohort-demo-smoke` (Docker cold-start)

**Decision: (d) KEEP main-only.** Argued: (1) its historical breaks (prod-auth-guard compile failure,
stale Elixir pin — see `ci.yml:847-850`) are **already caught on PR** by `quality` +
`optional-dependencies` (106 D-05); (2) the *uniquely* PR-introducible surface is Dockerfile/compose
edits — a narrow, rarely-touched path; (3) a Docker image build + full-stack boot is ~176s p95 of
**pure cold-start with its own flake surface** (image pulls, compose timing) that would import flake
into the PR gate for marginal regression coverage. *Optional polish:* a `paths:`-filter (option c)
that adds it to PR only when `docker/**` or `scripts/demo/**` change — but lean toward deferring
(Phase-107-style, matches 106 deferred `paths:` idea). **Not gated on PR in the locked rec.**

### Lane C — `package-consumer-full` (5-profile matrix + preflight + dry-run)

**Decision: (d) KEEP main-only — DO NOT touch.** This is the **release full-verification gate**
(hard invariant). The `image` profile + version-alignment **already gate PR** via the lean
`package-consumer` (106 D-08/D-10). The remaining breadth (video/tus/mux/gcs profiles +
`release_preflight` + `repo_hygiene` + `hex.publish --dry-run`) is *release-readiness*, proven by the
push:main run conclusion → `gate-ci-green` (106 D-11). Adding it to PR would re-import the **887s
long pole** that was THE wall-clock win. The only PR-introducible regression not covered by the lean
image profile is an adapter-specific install break (tus/gcs/mux) — addressable far more cheaply by
the **deferred `paths:`-filter** (add the matching profile to PR only when that adapter's files
change), not by gating the whole matrix. Recommend documenting that path-filter as a Phase-107-class
follow-up, **not** in this milestone's PR gate.

### Lane D — `mux-soak` (real API)

**Decision: (d) KEEP off the PR gate (label-gated stays as-is).** Argued: real `api.mux.com` traffic
→ **cost + quota**; fork-PR secret safety (secrets resolve empty on forks via the safe `pull_request`
trigger → fails closed, no leak — `ci.yml:1045-1056`). This is an **environment/quota-only** class, NOT
PR-introducible by a typical contributor. The **cassette-mode** `mux` profile already runs in
`package-consumer-full` (off-PR) and exercises the Mux *code path* without live API. Gating live Mux on
every PR is the textbook anti-pattern (third-party outage blocks merges). **No change.**

## 3. Idiomatic-for-Elixir / GitHub-Actions Analysis

**CI-tiering norm (Elixir/BEAM OSS):** The dominant idiom is a **representative gate on PR + breadth
post-merge/nightly**, NOT "everything on every PR." Rindle already adopted the **Tokio model**
(106 D-01): a `basics`-style representative PR gate, heavy/flaky lanes off the critical path. The
*pure-lib* projects (Ecto, Oban, Phoenix LiveView) run fuller matrices on PR **because their lane cost
is compile + ExUnit** (cheap); rindle's lanes (install-smoke matrix, Playwright, Docker cold-start) are
qualitatively heavier, so the deviation is correct and well-precedented.

**The idiomatic GHA pattern for "expensive integration lane that still gates":** the
**smoke-on-PR + full-on-merge split** — which rindle *already half-does* (`adoption-demo-unit` on PR,
`adoption-demo-e2e` on main). The locked rec simply completes the pattern by adding a *browser* smoke
on PR. This is more idiomatic than `merge_group` for a solo-maintainer 0.x lib (see §3a).

**Aggregate-gate idiom:** rindle's `CI Summary` (`if: always()`, bash loop over `needs.*.result`,
skip==pass; `eval_ci_summary.sh`) is the house pattern and the sibling-repo norm (sigra `ci-gate`,
lattice_stripe — ported from rindle). **Adding a lane to the PR gate = add it to `CI Summary.needs`
ONLY** (never a second required context) — this preserves the Phase-105 single-required-check
invariant. The skip-as-pass logic stays **byte-unchanged**: a lean lane that *always runs on PR* is
`success`, and on push:main it also runs — it never needs the omit-from-needs treatment that the
event-gated full lanes require (106 D-09).

### 3a. `merge_group` / GitHub merge queue — explicit consideration

**Verdict: DEFER (do not adopt in v1.21).** Pros: runs heavy lanes only at merge-queue time → same
MTTD as push:main but *blocks the merge itself* rather than the next PR; zero per-PR cost. Cons for a
**solo-maintainer 0.x lib**: (1) merge queue's value is *serializing concurrent merges from many
contributors* — rindle has ~1 active committer, so the queue is near-empty and the benefit is
marginal; (2) it adds a **new branch-protection surface** (`merge_group` trigger + required-on-merge
config) immediately after the hard-won Phase-105 single-required-check flip — exactly the complexity
106 deferred it to avoid; (3) `gate-ci-green`/release coupling keys off the **push:main** run
conclusion — a merge-queue topology would require re-validating that the release gate still sees a
full-matrix run on the merged SHA. **106 already deferred this explicitly** ("adds branch-protection
complexity right after the 105 flip; revisit in a later milestone"). Keep deferred; the
smoke-on-PR split delivers the regression-class coverage without the topology churn.

## 4. Peer-Lib Lessons

| Project | PR gate | Merge/nightly | Lesson for rindle |
|---|---|---|---|
| **Phoenix** | Full test matrix + formatter on PR; no Dialyzer in CI | — | Cheap lanes → everything on PR. Not transferable to rindle's heavy lanes. |
| **Ecto** | Curated-diagonal OTP×Elixir matrix on PR; lint-on-newest | — | Curated diagonal (rindle adopted nightly, 106 D-13). PR matrix stays 2-cell. |
| **Oban** | Cancel-on-PR / serialize-on-push concurrency; Dialyzer gated on PR | — | rindle mirrors the concurrency pattern (106 D-06); rindle **rejects** PR-Dialyzer (≤7-min budget). |
| **Bandit / Tokio** | Representative gate on PR; heavy/flaky lanes off PR | breadth post-merge | **The model rindle follows.** Right-to-emulate: smoke-on-PR + full-elsewhere. |
| **sigra (sibling)** | `install_smoke` on PR + event-gated `install_matrix` absent from gate `needs` | full on push | **Exact shape rindle shipped** for package-consumer (106 D-08). Footgun avoided: never put the event-gated lane in `needs`. |

**Cross-ecosystem (Rails/Django):** Both tier CI — Rails runs the heavy Active Storage/integration
suites in dedicated jobs, not gating every PR variant; Django uses a representative gate + a broad
nightly/buildbot matrix. Transferable lesson: **the "trustworthy main" guarantee comes from
post-merge breadth + fast MTTD + alerting, not from making the PR gate exhaustive.** rindle already
has the push:main run conclusion + (this milestone should add) main-failure alerting.

**Footgun to avoid (from the gap itself):** main-only lanes with **no failure alerting** mean a red
main can sit unnoticed. Nightly already has `nightly-failure-issue` (`nightly.yml:435`). **The
`push:main` `ci.yml` run has NO equivalent issue-on-failure** — a flake/regression on a main-only lane
relies on the maintainer noticing. This is a real sub-gap (couple to Area 1/2/4 or flag here).

## 5. DX / Contributor Experience

**Least-surprise principle: a green PR should mean a green main.** Today it doesn't, for the
render-regression and cold-start classes. The locked rec restores this for the **PR-introducible**
render class (the highest-value, contributor-facing one — a contributor editing the admin console or a
template sees the break on *their* PR, not on the next victim's).

**"0 human verification / gate the path that broke" (MEMORY `feedback_automate_uat_shift_left`):**
The 2026-06-26 cluster broke on a path PR CI never exercised. Gating the lean browser smoke converts
the previously-main-only render check into an automated, merge-blocking PR signal — directly executing
the maintainer's shift-left principle.

**Fast-green-PR vs trustworthy-main balance:** The rec adds ONE *parallel* chain (~90–150s) that does
**not** extend the long pole, so the ≤7-min budget holds (§7). Contributors keep their fast PR; main
gets the render-regression guarantee back. The two *expensive/flaky/environment* lanes
(cohort-Docker, full-matrix, mux-soak) correctly stay off-PR — gating them WOULD trade away the
wall-clock win for low-value or environment-only coverage.

**byte-unchanged** (only `CI Summary` required). **CRITICAL ORDERING: this lane lands only AFTER
Areas 1/2/4 kill the flakes** — gating a still-flaky lane imports the flake into the PR gate.

### 7d. Wall-clock proof (≤7-min budget holds)

- PR long pole today ≈ **image-smoke chain ~414s p95** (106 D-02), PR p95 ≈ **7.2 min**.
- New lane chain = `quality`/`optional-dependencies` fan-out (~184s p95) **+** lean E2E. Full E2E is
  318s p95 for **22 specs incl. screenshots**; the lean subset (smoke 18-line + admin-console 109-line,

  boot. Lean E2E chain ≈ **~340–400s p95** — **at or under the existing image-smoke long pole**, so it
  runs *in parallel* and does **not** extend PR p95. **Net PR p95 ≈ unchanged (~7.2 min), within ≤7-min
  target band** (106 already accepts ~7.2 p95 / 5 avg).
- **Guardrail (testable):** the lean lane's p95 MUST stay ≤ the image-smoke long pole. If
  measurement shows it becoming the new long pole (pushing p95 materially > 7.2 min), narrow the spec
  set to `smoke.spec.js` only, or apply the `paths:`-filter. **This is a hard acceptance criterion.**

### 7f. ORDERING DEPENDENCY (load-bearing — state explicitly)

> **Areas 1/2/4 (kill the flakes) MUST land BEFORE this lane is added to `CI Summary.needs`.**
> The 2026-06-26 incident was a *flake* reaching main, not a regression. Gating a still-flaky lane on
> PRs **imports the flake into the merge-blocking gate** — turning a main-only annoyance into a
> PR-blocking one, eroding trust (the exact failure the "false reds erode trust more than they catch
> bugs" DNA warns of). **Sequence: (1) de-flake the E2E lane via Areas 1/2/4 → (2) prove N consecutive
> green main runs of `adoption-demo-e2e` → (3) THEN add the lean PR variant.** The roadmap MUST order
> the flake-kill phases before this gate-flip phase.

### 7g. Blast radius / escalation

- **Blast radius:** MEDIUM. Touches `ci.yml` (new job + 1 `needs:` line in two aggregators) and one
  `e2e_local.sh` line (back-compatible default). No `lib/` change. No branch-protection change. The
  full E2E lane is unchanged.
- **Escalation flags (VERY-impactful per MEMORY):** (1) **the ordering dependency** — flipping this
  before flakes die is a self-inflicted PR-gate flake; escalate if the roadmap can't guarantee
  ordering. (2) **the push:main issue-on-failure sub-gap** (§4/§6) — adding alerting for a red
  *main* `ci.yml` run is arguably in-scope for "close the gate gap" and worth a maintainer call on
  whether it belongs here or in an alerting area.
