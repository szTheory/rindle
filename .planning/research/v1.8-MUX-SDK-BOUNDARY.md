# Mux SDK Boundary Research

**Date:** 2026-05-08
**Purpose:** Preserve the current recommendation about how Rindle should
depend on Mux without re-litigating it in the next milestone turn.

## Question

Should Rindle use the official Elixir Mux SDK, or own a custom client, so the
core library stays lean?

## Findings

- The official `mux` package is the right low-level dependency for Mux REST
  calls and token/webhook primitives.
- Rindle already uses a thin boundary instead of inlining Mux behavior:
  `Rindle.Streaming.Provider.Mux.HTTP` delegates to the SDK, while
  `Rindle.Streaming.Provider.Mux` normalizes behavior into Rindle's contract.
- The custom logic that remains in Rindle is the valuable part: state
  normalization, redaction, telemetry, retries, webhook wiring, and
  persistence.
- The SDK itself appears stable but not especially fast-moving, so Rindle
  should keep the adapter boundary even if the SDK remains the backend.

## Recommendation

Use the official SDK and keep Rindle-owned wrapper code thin and intentional.
Do not build a bespoke general-purpose Mux client unless the SDK becomes a
hard blocker.

If core size becomes a stronger goal later, the better extraction is a separate
`rindle_mux` package, not a rewrite of the upstream SDK.
