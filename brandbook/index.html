01

Essence

Positioning

Rindle is a Phoenix-native media lifecycle library for turning uploads into durable, secure, observable media assets. It manages everything after upload: staged objects, validation, analysis, attachments, variants, derivatives, previews, background processing, signed delivery, cleanup, regeneration, and operational visibility.

Brand promise

Rindle makes media boringly reliable in production. Not flashy. Not magical. Not a SaaS wrapper. Rindle is the dependable layer that helps Phoenix teams ship media features without losing control of state, storage, security, or background jobs.

The brand idea: a protected current

A rindle is a small runnel or watercourse. Media enters as raw bytes. Rindle carries it through validation, processing, storage, delivery, and maintenance. The system is calm, strict, visible, and hard to derail. Use the metaphor subtly — Rindle guides media through a safe path from upload to ready asset. Never make it look like a water company.

Naming

Use Rindle in prose. Use rindle only for the package, CLI commands, repo names, config keys, and code. Never RINDLE, never "Rindle Media Manager Pro", never "the Rindle Cloud". Use media-lifecycle vocabulary (asset, attachment, variant, recipe, pipeline, session, delivery, cleanup); avoid identity/auth-protocol vocabulary.

02

Logo

The mark is a confluence: three tributaries — uploads, formats, sources — merge into one durable current that runs on beneath the name. The mark and the logotype interlock; the current passes below the baseline of ri and stops under the i. The top tributary carries Rindle Green; everything else is Deep Current. Letters are outlined from Space Grotesk Medium — no font dependency in any committed SVG.

Construction rules

  • Clearspace: keep a margin of half the mark's height on all sides. Exception by design: the current's under-run already interlocks with the type — never add other elements inside the clearspace.
  • Minimum sizes: full lockup 96px wide; icon-only mark 24px; below that, use the favicon cut (two tributaries, heavier strokes — favicon.svg).
  • Tagline: only rindle-logo-subtitle.svg carries "Media, made durable." The primary lockup never does.
  • Color: green tributary + deep current on light; porcelain + green on dark; one-color mono anywhere those fail.

Misuse

Do

  • Use the committed SVGs as-is; regenerate via brandbook/src/logo.mjs
  • Let the current break the baseline — that boundary-break is the identity
  • Use the mono variant when color can't be guaranteed

Don't

  • Put the mark in a box, badge, circle, or container of any kind
  • Separate the mark from the type with extra gap, or recolor strokes
  • Add taglines, gradients, shadows, or outlines
  • Use cloud-upload arrows, camera shutters, padlocks, database cylinders, AI sparkles, or purple potion motifs anywhere near the brand

Rules that are enforced, not suggested

  • Rindle Green and Rind Lime are accent-only on light surfaces (measured 1.81:1 / 1.18:1). They are text-capable only on Ink and dark surfaces.
  • Focus rings: Deep Current on light (11.34:1), Rindle Green on dark (9.38:1). Green-on-light fails the 3:1 non-text requirement — the seed's original rule was corrected by measurement.

  • Slides/projectors: rely on Ink-on-Shell or Shell-on-Deep-Current; never on the Shell-vs-Porcelain distinction.

05

Voice & writing

Voice formula: clear technical explanation + production empathy + calm confidence. Rindle writes like an experienced Phoenix maintainer explaining hard-won production lessons to another developer.

Rindle stores variants as records, not hidden filenames, so failed and stale outputs are visible.
Rindle magically handles all your media!
This variant is stale because its recipe changed. Regenerate it to update the stored output.
Never worry about uploads again.

Writing principles

  1. Explain state. Users should always know what happened, what is happening, and what to do next.
  2. Name the footgun. Don't hide risks; explain them calmly.
  3. Verbs over abstractions. "Regenerate stale variants", not "perform maintenance".
  4. Concrete examples. Avatars, product photos, private documents, galleries.
  5. Never overpromise.

Docs tone

Practical, copy-pasteable, production-aware, honest about tradeoffs, structured around real Phoenix workflows. Guides follow: what you'll build → when to use this → install/configure → define the profile → attach or upload → render or deliver → process in the background → failure modes → production notes → full example.

Slogans (use sparingly)

Upload is the beginning, not the lifecycle. · Variants are records. · Recipes are versioned. · Processing is idempotent. · Storage I/O is not a database transaction. · Missing media should be visible. · Strict defaults, explicit escape hatches.

08

Imagery, iconography & motion

Acceptable imagery

Mostly abstract, technical: layer diagrams, branching variant maps, state machines, processing timelines, storage topology, cropped UI screenshots, code + output compositions, subtle channel / current lines. Warm neutral backgrounds, thin lines, rounded containers, small accents of Rindle Green or Rind Lime, clear labels.

Never: stock photos of photographers, people staring at laptops, upload-cloud illustrations, AI-generated glossy 3D blobs, literal rivers and waterfalls, hacker imagery, corporate blue dashboards, camera-brand identity.

Diagram rules

  • Every diagram teaches one lifecycle idea; every state is labeled.
  • Solid lines for required/eager work; dashed lines for async/lazy work.
  • Muted colors for inactive states; semantic colors only with labels.
  • Arrows sparingly and consistently.

Iconography

Rounded line icons on a 24px grid: 2px stroke, rounded caps and joins, minimal fill, optical balance over strict geometry, legible at 16/20/24px. Icon needs: asset, attachment, variant, pipeline, processor, storage, signed URL, quarantine, retry, regenerate, cleanup, stale, missing, ready. Status icons always pair with labels. (Drawing the set is deliberately out of b1.0 scope — this grid spec governs it when it happens.)

Motion

Materialization, not entertainment. Durations are tokens: press 120ms · popover 160ms · toast 200ms · lifecycle transition 300ms · diagram 600ms, easing cubic-bezier(0.2, 0, 0, 1). Respect prefers-reduced-motion; no bouncing; status changes are never playful; async processing shows real state, not fake progress.

Proof points (product proof, not hype)

  • Variants are queryable records; recipes are versioned so stale outputs are visible.

Do / Don't

Do

  • Use Rindle as a calm infrastructure brand
  • Use lifecycle diagrams and real state names
  • Use strict, practical security language
  • Use warm technical colors and readable type
  • Show code examples early; show admin/operations value
  • Explain failure modes

Don't

  • Make it look like a photo app or a cloud-storage SaaS
  • Overuse water imagery, locks, or shields
  • Use AI-sparkle imagery as the brand
  • Say "magic" or "never worry again"; hide tradeoffs
  • Use color as the only status indicator
  • Use generic purple Elixir branding
